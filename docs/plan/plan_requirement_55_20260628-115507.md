# 需求 #55 开发计划：MCP 服务手动启停与请求头密钥可配置

## 背景

AutoPlan 内置 MCP（Model Context Protocol）服务端，供外部 agent（Claude/Codex 等）通过 MCP 工具调用 AutoPlan 能力。当前实现存在两处使用限制：

1. **只能随应用启动自动拉起，无法手动控制**。`src/main.js` 在 app ready 时调用 `scheduleMcpServerStart()` 自动启动 MCP（`startMcpServer()` → `createMcpServer(...).start()`），全应用生命周期内只有一个模块级 `mcpServer` 单例；既没有「停止」「再次启动」的 IPC 通道，前端也没有入口。想停掉只能设 `AUTOPLAN_MCP_ENABLED=0` 后重启整个应用。
2. **请求头密钥（authToken）及其它配置的持久化能力不完整**。`src/mcpConfig.js` 的 `saveMcpSettings()` 只识别并保存 `authToken`，其它项（`transport/host/port/path/enabled`）被忽略；且只有 `authToken` 变化才会触发 `scheduleMcpServerRestart()`。更严重的是 `normalizeMcpAuthToken()`（`src/mcpConfig.js:29`）在传入空值时抛 `MCP 密钥不能为空`，导致用户**无法清空密钥**回到无鉴权模式。

现状补充：`AutoPlanMcpServer`（`src/mcpServer.js`）已实现 `start()/stop()/status()`（start 内含 `validateSafeHost` 远程绑定校验与 `timingSafeEqual` 密钥比对，未带 `Authorization: Bearer <key>` 返回 401），本需求的主要工作在 **IPC 暴露、配置持久化逻辑修正、快照状态准确性 + 密钥脱敏、前端管理面板**。`src/loop/snapshots.js` 的 `mcpStatusSnapshot(db)` 目前从 settings/env 推导 `transport/host/port/path/authToken`，并**以明文返回 `authToken` 与 `authHeader`**（安全隐患）；其 `running` 字段是从「最近一条 `mcp.started`/`mcp.start.failed` 事件」推导而来，并非进程真实状态，手动停止后无法准确反映。前端 `WorkspaceSettingsView.tsx` 的 `mcp` 面板只展示状态信息 + 单个 authToken 输入框，无 transport/host/port/path 配置、无启停按钮、无密钥显示/隐藏/生成/清空。

## 设计约束

- **复用 `AutoPlanMcpServer` 既有能力，不重写服务端**：`start()/stop()/status()`、`validateSafeHost`（默认仅 localhost、远程需 `AUTOPLAN_MCP_ALLOW_REMOTE=1`）、`timingSafeEqual` 比对 token、401 鉴权、stdio/http 双传输均已实现，本需求只在其外层补 IPC、配置持久化与 UI。`start()` 内部已处理端口占用/stdio 失败：失败时 `await this.stop()` 后将错误写入 `state.lastError` 并抛出。
- **手动启停基于现有 `mcpServer` 单例 + `createMcpServer` 重建**：「停止」调用 `mcpServer.stop()`；「启动」需重新读取最新配置（`mcpServerConfig(db)`）并经 `createMcpServer(...)` 重建实例后 `start()`，以应用保存后的 transport/host/port/path/authToken 变更——**不能复用旧实例的冻结配置**。启动需做幂等守卫：若 `mcpServer.status().running` 已为 true，直接返回当前状态，避免重建第二个实例导致 `EADDRINUSE`。启停全程**不依赖应用整体重启**。
- **状态准确性优先用「实时状态注入」而非事件推导**：`mcpStatusSnapshot` 当前仅以 `db` 入参、靠最近事件猜 `running`，手动停止后无法及时反映。改为向 `LoopService`/`snapshot` 注入一个实时 `mcpStatusProvider`（main.js 持有 `mcpServer` 句柄，提供 `() => mcpServer?.status?.()`），快照将「实时运行态（running/lastError/transport/host/port/url/startedAt）」叠加到「db 推导的配置」之上。同时事件流仍记录 `mcp.started`/`mcp.start.failed`/新增 `mcp.stopped`，`mcpStatusSnapshot` 的「最近事件」查询需纳入 `mcp.stopped`，作为 `lastEvent`/`lastError` 来源。
- **密钥不在快照中明文下发到渲染层**：当前 `mcpStatusSnapshot` 明文返回 `authToken` 与 `authHeader: 'Authorization: Bearer <token>'`，违反「默认脱敏展示」。改为移除明文 `authToken`，仅下发 `hasAuthToken: boolean` 与 `authTokenMasked`（如 `····1234` 仅显示末 4 位，无密钥时为空），`authHeader` 改为不含密钥的模板提示（`Authorization: Bearer <token>`）。渲染层表单的密钥输入框**默认留空**（占位提示「已设置密钥（····1234）」），用户输入新值即覆盖、点「清空」即置空——不再从快照回填明文。
- **允许清空 authToken**：`src/mcpConfig.js` 的 `normalizeMcpAuthToken()` 改为允许空串（trim 后为空也返回空串、不抛错）；保存逻辑区分「设置密钥（非空）」与「清除密钥（显式空串）」，空串以空串落库（表示关闭 Bearer 鉴权，`isAuthorizedRequest` 在 `authToken` 为空时直接放行）。前端「保存」密钥变更应明确确认（按钮文案/二次确认），避免误操作。
- **完整保存 MCP 配置**：`saveMcpSettings` 扩展为按 `mcp.` 前缀保存 `enabled/transport/host/port/path/authToken`，任一项变化都返回「发生变化」并由调用方触发 `scheduleMcpServerRestart()`，使服务按新配置重启生效。`enabled` 主要控制「应用启动时是否自动拉起」（`scheduleMcpServerStart`/`mcpServerConfig` 读取）；手动「启动」可在当前会话显式拉起，不受 `enabled=false` 阻断（保留手动入口的可用性）。
- **远程绑定安全约束不放松**：保存非 localhost 的 host 后，restart 会触发 `validateSafeHost`；若未设 `AUTOPLAN_MCP_ALLOW_REMOTE=1`，`start()` 抛错并记 `mcp.start.failed`，UI 经 `lastError` 展示该中文提示。`allowRemote` 仍仅由环境变量决定，不通过 UI/settings 放开。
- **随机密钥生成在渲染层完成**：「生成随机密钥」按钮在渲染层用 `window.crypto.getRandomValues` 生成高熵随机串（如 32 字节 base64url/hex）填入输入框，无需经主进程往返。
- **IPC 命名与校验沿用既有风格**：新增通道遵循 `namespace:action`（`mcp:start`/`mcp:stop`/`mcp:status`/`mcp:saveConfig`），失败抛中文错误；启停/保存后返回 `loop.snapshot(projectId)` 以刷新 UI。preload 在 `window.autoplan` 上以一行式 `ipcRenderer.invoke` 暴露。
- **文件长度约束**：`npm run check`（含 `scripts/check-file-lengths.js`）对单文件行数有限（`src/main.js` 800、`src/renderer/types.ts` 980，默认 600）。`WorkspaceSettingsView.tsx` 现约 423 行，新增启停按钮 + 多字段配置表单 + 密钥控件后体积会增长——表单状态/校验逻辑下沉到 `workspaceForms.ts`，必要时将 MCP 面板抽成子组件，避免单文件超限。
- **不回退既有能力**：现有项目/需求/反馈/计划/任务/验收/事件流/CLI 后端/MCP 工具链路与工作区导航均不得回退；既有 `loop:configure`/`saveMcpSettings` 对 authToken 的处理改为兼容扩展（仍能接收原 `mcpAuthToken` 入参并正确归一）。
- **不要把运行测试/回归/构建拆成普通开发任务**；最终检查统一放入最后的「完整验收」节点。
