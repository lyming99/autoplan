# Plan #26: 工作区对话模块（Chat）

## 目标

在工作区新增「对话」模块：用户自配 LLM 接口（OpenAI / Anthropic / DeepSeek 等 OpenAI 兼容端点），在 AutoPlan 内与 AI 对话。AI 通过**工具调用（function/tool calling）**能力操作当前项目，严格约束为只读 + 安全创建，**不允许写入文件、不允许执行命令**。

## 安全边界（核心）

| 类别 | 内容 |
|------|------|
| **允许的工具** | `read_file`、`search_files`、`create_requirement`（autoRun=true）、`create_feedback`（autoRun=true）、`create_script`（仅落库不执行） |
| **禁止的工具** | `write_file`、`edit_file`、`delete_*`、`run_command`、`run_script` 等任何文件写入/删除/命令执行 |
| **文件访问校验** | 所有文件操作必须经 `resolveWorkspaceFilePath` + `isInsidePath` 校验，拒绝越界/目录访问/不存在的路径 |
| **read_file 上限** | 单文件 256KB，超出截断并提示 |
| **search_files 上限** | 结果数 ≤50 条 + 5s 超时保护 |

## 架构

```
渲染进程 ChatView ──chat:* IPC──▶ 主进程 chatController
                                    ├─ llmClient（fetch+SSE，可注入 stub 测试）
                                    └─ chatTools（白名单工具，复用 IntakeService/main 既有校验）
```

核心原则：**所有 LLM 请求与文件/系统访问只在主进程发生**，渲染进程只做展示与 IPC。LLM 增量 token 与工具事件经 `chat:chunk` 事件推给渲染进程实时拼装（沿用现有 `onLoopUpdate` / `onUpdateStatus` 的事件推送范式）。

## 总体验收标准

### 验收命令

```bash
node --test src/chat/llmClient.test.js
node --test src/chat/chatTools.test.js
node --test src/chat/chatController.test.js
```

### 验收范围

1. **LLM Client**：OpenAI 兼容协议 + Anthropic Messages 协议的请求构造、SSE 增量解析、tool_call 解析、AbortController 中止
2. **Chat Tools**：白名单工具执行（read_file 越界拒绝/截断、search_files 上限/超时、create_requirement/create_feedback 走 IntakeService、create_script 仅落库）、白名单不含写/执行工具
3. **Chat Controller**：agent loop 多轮编排（用户消息 → LLM → 工具执行 → 回灌 → LLM）、最大轮次限制（8）、中止/超时/错误处理
4. **数据库**：chat_messages 表 CRUD、chat.* 默认配置种子、按 project_id 隔离
5. **IPC**：chat:send/stop/clear/history/saveConfig/getConfig 通道 + chat:chunk/chat:done 事件推送
6. **UI**：ChatView 消息渲染（markdown + 工具调用卡片）、输入框发送/停止切换、清空会话、设置页 AI pane 配置表单

### 通过标准

- 全部单元测试通过（零失败）
- 手工人机验证：打开 AutoPlan → 进入项目 → 侧边栏可见「对话」导航 → 在设置 AI pane 配置 LLM 接口 → 切换到对话 tab → 发送消息 → 收到流式 AI 回复 → 工具调用卡片显示工具名/参数/结果 → 停止按钮可中断生成 → 清空会话后消息清空 → 切换项目后对话隔离

- [x] P006: 实现 Agent Loop 编排（chatController.js）<!-- scope: src/chat/chatController.js -->
  - createChatController({ db, llmClient, chatTools, projectId, workspacePath, onEvent, onDone })
  - onEvent({type, data})：推送给主进程 IPC 层的回调，type='chunk'|'tool_start'|'tool_result'|'error'|'status'
  - onDone({status, error?})：本轮结束回调，status='done'|'aborted'|'error'|'max_rounds'
  - send(message)：主入口 → 存储 user 消息到 chat_messages → 构建 LLM messages 数组（从 chat_messages 取最近 N 条，N 按 token 预算粗略估计约 2000 token ≈ 20 条） → 进入 agent loop
  - agent loop：LLM(带 tools) → 纯文本回复则 break 结束 → tool_calls 则逐个执行工具 → 每条工具结果存储为 role=tool 消息并 onEvent 推送 → 全部工具执行完后回灌 LLM → 下一轮；最大 8 轮
  - 每轮 LLM 回复存储为 role=assistant 消息到 chat_messages（tool_calls 序列化为 JSON）
  - stop()：调用 AbortController.abort()，当前轮次标记 aborted
  - 超时：单次 LLM 请求超时（如 120s），超时后 onDone({status:'error'})
  - 错误：LLM 调用异常 → 追加 role=system 错误消息到 chat_messages → onDone({status:'error'})
  - getHistory(projectId)：从 chat_messages 查询，按 created_at 升序
  - clearHistory(projectId)：DELETE FROM chat_messages WHERE project_id = ?
