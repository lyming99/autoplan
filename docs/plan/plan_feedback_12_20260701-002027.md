# 反馈 #12 开发计划：删除需求/反馈时级联删除计划与任务

## 背景与目标

反馈要求：删除需求或反馈后，关联计划要删除，关联任务要删除，正在执行的关联任务要停止并删除。

基于当前代码梳理，需求与反馈分别存储在 `requirements`、`feedback` 表，单条记录生成计划后会通过 `linked_plan_id` 绑定 `plans.id`；任务存储在 `plan_tasks` 表。当前 `src/main.js` 的 `requirements:delete` / `feedback:delete` 会先调用 `loop.interruptPlan()`，再删除 intake 记录与附件，但 `interruptPlan()` 只是将计划标记为 `interrupted`、将未完成任务标记为 `blocked`，不会删除 `plans` 与 `plan_tasks`。这导致用户删除需求/反馈后，关联计划和任务仍可能在计划与任务视图、验收视图、搜索结果或快照中可见。

目标是把删除需求/反馈改成服务层的原子级联流程：先停止关联 plan 的所有运行中任务/验收/修复操作，再删除关联任务、计划、计划文件和 intake 记录，并保证快照、前端状态和回归测试都反映“已删除”。

## 影响范围

- 后端 IPC：`src/main.js`
- 循环与运行时：`src/loopService.js`、`src/loop/runtime.js`、`src/loop/taskExecution.js`、`src/loop/validation.js`
- 文件与扫描缓存：`src/loop/workspaceFiles.js`
- 前端删除交互：`src/renderer/components/IntakePanel.tsx`、`src/renderer/hooks/useWorkspaceController.ts`、`src/renderer/pages/WorkspacePage.tsx`
- 类型与测试：`src/renderer/types.ts`、`src/loop/intakeDeletion.test.js`、`src/renderer/pages/WorkspacePage.test.tsx`、`scripts/smoke-test.js`

## 任务拆解

- [x] P001: 停止并隔离关联计划的运行中操作 <!-- scope: src/loopService.js,src/loop/runtime.js,src/loop/taskExecution.js,src/loop/validation.js -->
  - 验收要点：删除需求/反馈时，能够按 `linked_plan_id` 找到同项目关联 plan，并停止该 plan 下正在运行的单任务、并发批次任务、验收命令、修复操作或计划生成后续操作。
  - 验收要点：所有匹配 `planId` 的 `runtime.activeOperations` / `activeChildren` 都会被终止并归档或移除，快照中不再显示已删除 plan 的 active operation。
  - 验收要点：子进程被终止后，即使异步执行链继续返回，也必须重新检查 plan/task 是否仍存在；已删除目标不得再写入 `completed`、`failed`、`validation_failed`、`plan.completed` 等状态或事件。
  - 验收要点：并发执行场景中，同一 plan 的多个运行中任务会全部停止，不只停止当前 `runtime.activeOperation`。

- [x] P002: 级联删除 intake、计划、任务与计划文件 <!-- scope: src/main.js,src/loopService.js,src/loop/workspaceFiles.js,src/renderer/types.ts -->
  - 验收要点：`requirements:delete` 与 `feedback:delete` 改为调用统一服务层方法，删除逻辑不再分散在 `src/main.js` 的局部 helper 中。
  - 验收要点：有关联 plan 时，按同项目校验后删除 `plan_tasks` 中该 plan 的全部任务，再删除 `plans` 中该 plan 记录，最后删除对应需求/反馈、附件；无关联 plan 时仍能只删除 intake 与附件。
  - 验收要点：删除需求时，仍保留现有行为：同项目反馈的 `requirement_id` 会置空，避免悬挂引用。
  - 验收要点：计划 Markdown 文件只在安全路径校验通过时删除：解析后的路径必须位于当前项目 workspace 的 `docs/plan` 下，并且文件名匹配 AutoPlan 为该 intake 生成的 `plan_requirement_<id>_*.md` 或 `plan_feedback_<id>_*.md` 模式；不满足条件时只删除数据库记录并记录可追踪事件。
  - 验收要点：删除计划文件后，同步清理 `scan_files` 中对应 `scan_type='plan'` 的记录，后续快照和扫描缓存不会把已删除计划重新展示出来。

- [x] P003: 更新前端删除提示与删除后的阅读状态 <!-- scope: src/renderer/components/IntakePanel.tsx,src/renderer/hooks/useWorkspaceController.ts,src/renderer/pages/WorkspacePage.tsx -->
  - 验收要点：需求/反馈删除确认文案明确说明关联计划、任务和运行中执行会被停止并删除，避免用户误以为只会中断任务。
  - 验收要点：删除成功后的 snapshot 会使需求/反馈列表、计划列表、任务列表、验收列表和搜索结果中不再出现已删除 intake 及其关联 plan/task。
  - 验收要点：如果用户正在阅读即将被删除的关联 plan，删除成功后关闭或重置 plan reader 状态，不保留已删除 plan 的旧内容。
  - 验收要点：删除失败时保留当前 UI 状态并展示后端错误，不做前端乐观删除。

  - 验收要点：覆盖删除已绑定 plan 的反馈后，`feedback`、关联 `plans`、关联 `plan_tasks`、反馈附件均被删除，需求记录不被误删。

## 总体验收标准

最终验收命令：

```powershell
npm run check
npm test
npm run smoke
npm run build
```

验收范围：

- 数据层：`requirements`、`feedback`、`attachments`、`plans`、`plan_tasks`、`scan_files` 的级联删除和同项目边界校验。
- 运行时：单任务、并发任务、验收命令、修复命令等关联 active operation 的停止与归档。
- 文件层：AutoPlan 自生成计划文件的安全删除，以及非 AutoPlan 生成或越界路径的保护。
- 展示层：需求/反馈、计划、任务、验收、搜索和 plan reader 状态与最新 snapshot 一致。

通过标准：

- 删除需求/反馈后，关联计划和任务不再出现在数据库查询、`loop.snapshot(projectId)`、前端列表与搜索结果中。
- 删除时正在执行的关联任务会被终止，且不会在后续异步完成回调中重新写入任务状态或计划状态。
- 安全路径内的 AutoPlan 自生成计划文件被删除，扫描缓存同步清理；不安全路径不会被误删。
- 所有最终验收命令通过，且没有新增 TypeScript、Node 语法、构建、smoke 或文件长度检查错误。
