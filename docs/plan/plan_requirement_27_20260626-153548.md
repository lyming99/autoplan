# 需求 #27 开发计划：使用 scripts/release.py 发布一个版本

## 需求概述

使用项目内已有的 `scripts/release.py` 脚本完成一次版本发布。该需求重点是执行发布前准备、调用发布脚本创建并推送 release tag，并确认 GitHub Actions / Release 流程被正确触发和完成。

## 任务列表

- [x] P001: 确认发布版本号与发布说明 <!-- scope: release,scripts/release.py -->
  - 验收要点：
    - 明确本次发布使用的 tag，且符合 `vX.Y.Z` 或项目约定的 `v` 前缀格式。
    - 明确本次发布说明来源，可使用命令行 `--notes` 或 `--notes-file` 传入。
    - 确认待发布版本对应的提交范围和主要变更点已整理完成。
    - 确认不需要修改业务代码即可完成本需求。

- [x] P002: 完成发布前环境检查 <!-- scope: scripts/release.py,git,github-actions -->
  - 验收要点：
    - 确认当前目录是 Git 工作区且远端仓库配置可用。
    - 确认工作区干净，不存在未提交或未跟踪的发布相关变更。
    - 确认本地和远端均不存在目标 release tag。
    - 确认具备向远端推送 tag 的权限；如使用 GitHub CLI，确认登录状态可用。

- [x] P003: 执行 release 脚本发布版本 <!-- scope: scripts/release.py,release -->
  - 验收要点：
    - 使用 `python scripts/release.py <tag>` 及必要参数执行发布，不手动绕过脚本创建 tag。
    - 发布过程中按脚本提示确认创建并推送 tag。
    - 本地生成的 tag 与远端推送的 tag 名称一致。
    - 脚本执行失败时保留错误信息，并根据脚本提示处理后重新执行。

- [x] P004: 确认发布流水线与产物 <!-- scope: .github,dist,release -->
  - 验收要点：
    - 确认推送 tag 后已触发对应 GitHub Actions release workflow。
    - 确认 release workflow 成功完成，没有失败 job。
    - 确认 GitHub Release 已创建或更新，版本号与目标 tag 一致。
    - 确认发布产物存在且名称、平台、版本信息符合项目发布约定。

- [x] P005: 完整验收 <!-- scope: validation -->
  - 验收要点：
    - 最终验收命令：`git status --porcelain`、`git ls-remote --tags origin <tag>`，以及通过 GitHub Actions 页面或 `gh run list --workflow <release-workflow> --limit 5` 查看发布流水线状态。
    - 验收范围：目标 tag、远端 tag 推送结果、GitHub Actions release workflow、GitHub Release 页面和发布产物。
    - 通过标准：工作区保持干净；远端存在且仅存在预期目标 tag；发布流水线成功完成；GitHub Release 与目标 tag 匹配；发布产物可见且无明显缺失。

## 总体验收标准

- 必须通过 `scripts/release.py` 完成版本发布，不接受手动 `git tag` / `git push tag` 替代主流程。
- 发布 tag 必须符合脚本校验规则，并与 GitHub Release 中显示的版本一致。
- 发布前工作区必须干净，避免把未确认变更混入发布流程。
- 发布后必须能在远端仓库查询到目标 tag，并能在 GitHub Actions 中看到对应 release workflow 成功完成。
- GitHub Release 页面必须存在本次版本，且发布说明和产物满足本次发布要求。
