# 反馈 #16 开发计划：任务分组折叠与搜索结果弹窗定位

## 背景

反馈 #16 指出：“任务列表，分组要支持折叠；搜索结果要通过popup显示搜索结果，支持点击打开定位到指定项。” 当前需要优化两个交互链路：一是任务列表已经存在按 Plan 或状态等维度组织的分组展示，但缺少可折叠状态，长任务列表下浏览成本高；二是工作区搜索结果目前嵌入页面流中展示，点击后主要切换到对应栏目，缺少弹窗式结果面板和对具体记录/任务/计划的精确定位与高亮。

## 设计约束

- 本计划只处理任务列表分组折叠、搜索结果 popup 展示、搜索结果点击定位和必要的状态类型扩展，不改动任务执行、计划生成、CLI 调用和数据库结构。
- 任务分组折叠应优先在前端状态内完成，默认展开策略需兼顾当前正在执行的任务和用户已有浏览上下文。
- 搜索结果 popup 需保持现有搜索数据来源、排序和分组统计能力，不重新实现搜索索引，不引入后端全文检索。
- 点击搜索结果必须能打开对应工作区 tab，并尽量滚动/聚焦/高亮到指定项；无法定位到 DOM 节点时也要有可见兜底提示。
- 不要把运行测试、回归、构建拆成普通开发任务；最终检查统一放入最后的“完整验收”节点。

## 总体验收标准

- 分组可控：任务列表分组支持展开/收起，折叠状态清晰可见，运行中任务不会因为默认折叠而被隐藏。
- 搜索浮层化：搜索结果通过 popup 展示，不挤占主内容布局，并支持清空、关闭、空结果和按来源查看。
- 点击可定位：点击任一可定位搜索结果后能打开对应 tab，定位到具体记录或任务项，并给出可感知的高亮反馈。
- 兜底明确：目标项不存在、被过滤或暂不可见时有非阻塞提示，不出现点击无反应或静默失败。
- 兼容现有能力：搜索命中来源、排序、分组统计、任务状态过滤、计划卡片和事件列表等现有功能不因本次改动退化。
- 最终验收命令：执行 `npm run check && npm run smoke`；验收范围为任务分组折叠、搜索结果 popup、搜索结果点击定位、高亮反馈和现有 smoke 主流程；通过标准为命令退出码为 0，且手动验证任务、计划、反馈、事件搜索结果均能按预期展示或定位。
