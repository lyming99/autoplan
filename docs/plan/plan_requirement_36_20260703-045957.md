# 需求 #36：对话中创建的需求/反馈可直接打开

> 生成时间：2026-07-03
> 需求：对话中通过 `create_requirement`/`create_feedback` 工具创建的需求/反馈，目前只返回 ID、标题、状态。希望它们能在对话中直接打开（可点击入口/链接/按钮），并支持用户用“打开需求 #xx / 打开反馈 #xx”打开；打开后可查看标题、正文、状态、执行计划等详情；有 UI 时优先用详情页/弹窗/侧边栏，无 UI 时在对话中展示详情或提供可复制链接；条目不存在/无权限时返回明确错误。

---

## 需求分析与方案

### 当前现状（基于代码分析）

对话（Chat）模块的工具链与渲染：

| 环节 | 文件 | 现状 |
| --- | --- | --- |
| 工具注册表 | `src/chat/chatTools.js` `getChatToolDefinitions({ db, projectId, workspacePath, intakeService })` | 仅 5 个工具：`read_file`/`search_files`/`create_requirement`/`create_feedback`/`create_script` |
| 创建需求/反馈返回 | `chatTools.js` `handleCreateRequirement`/`handleCreateFeedback` | 仅返回 `{ id, title, status }`（`chatTools.js:285`、`chatTools.js:317`），**无类型、无 projectId、无可点击入口** |
| Agent 执行与事件 | `src/chat/chatController.js` `executeTool` → `emitEvent({ type:'tool_result', data:{ name, result } })`；`main.js:710` 转发为 `chat:chunk` IPC | 工具结果随 `name` 一并落库到 `chat_messages.tool_result`，并推送渲染层 |
| 渲染 | `src/renderer/hooks/useChat.ts` `onChatChunk` 处理 `tool_result`；`src/renderer/components/workspace/ChatView.tsx` `ChatToolCard` | 工具卡片**通用渲染**：名称 + 参数摘要 + 弹窗里展示原始 JSON（`ChatView.tsx:19-107`），**无任何按工具类型的富展示或动作按钮** |
| 详情已有数据 | `AppSnapshot.requirements`/`feedback`（`types.ts:199-239`） | 已含 `body`、`status`、`linked_plan_id`、`plan_title`、`plan_file_path`、`plan_status`、`plan_completed`、`plan_total`，`IntakePanel` 已据此渲染正文+绑定 Plan 卡片 |
| 视图导航 | `src/renderer/hooks/useWorkspaceController.ts` `selectTab(tab)`；`WorkspaceTab` 含 `requirement`/`feedback` | 切 tab 走 URL `?tab=`；intake 列表项 DOM 锚点 `workspace-requirement-${id}`/`workspace-feedback-${id}`（`IntakePanel.tsx:397`） |
| 定位+高亮 | `src/renderer/pages/WorkspacePage.tsx` `locateWorkspaceSearchResult` | 已实现「切 tab → 等 120ms → `getElementById(anchorId)` → `scrollIntoView` + `search-locate-highlight`」可复用模式（`WorkspacePage.tsx:200-210,574-596`） |

**关键结论**：详情数据（正文/状态/执行计划）已完整存在于快照与 IntakePanel；缺的是「对话侧的可点击入口 + 把工具结果/用户意图转化为打开动作」。因此本需求以**渲染层编排 + 工具结果富化**为主，不需要改 DB schema 或新增 IPC 通道（导航完全可在渲染层完成）。

`ChatView` 仅在 `chat` 标签页渲染（`WorkspacePage.tsx:559`），侧边栏（`WorkspaceSidebar`）只渲染对话列表与导航、选中对话会 `onTab('chat')` 打开 ChatView，**不渲染工具卡片**。因此可打开入口只需注入 `ChatView`。

### 设计方案

**1. 工具结果富化（创建后即可打开）**：`create_requirement`/`create_feedback` 返回值追加 `type`（`'requirement'`/`'feedback'`）、`projectId`、`openable:true`，作为渲染层识别「可打开卡片」的稳定契约（`id` 仍为定位主键）。

**2. 新增 `open_requirement`/`open_feedback` 对话工具**（LLM 原生路径 + 无 UI 详情降级）：
- 入参支持 `id`（精确）或 `title`（模糊包含）。
- 从注入的 `db` + `projectId` 查询：命中则返回结构化详情 `{ type, projectId, id, title, body, status, createdAt, updatedAt, linkedPlan:{ id, title, filePath, status, completed, total } | null, openable:true }`；未命中返回 `{ error, errorCode:'INTAKE_NOT_FOUND' }`。
- 该结果天然满足「无 UI / 无点击时在对话中展示详情（含执行计划）」，并在卡片上提供「打开」入口。

**3. 用户意图直达（确定性路径，零 LLM 往返）**：发送消息前用纯函数 `parseOpenIntakeIntent(text)` 识别「打开/查看 需求/反馈 #N 或 N」，命中即立即触发 `onOpenIntake({type,id})` 直达（仍可继续把消息发给 AI 补充说明）。

**4. 渲染层编排**：
- `ChatToolCard` 识别可打开工具结果（`name ∈ {create_requirement,create_feedback,open_requirement,open_feedback}` 且 `result.id` 为正整数），渲染富卡片：标题、状态徽标、（`open_*` 时）正文摘要 + 绑定 Plan 进度行，并提供动作「打开」（调 `onOpenIntake`）与「复制链接」（复制应用内定位链接）。
- `WorkspacePage` 新增 `handleOpenIntake({type,projectId,id})`：必要时 `navigate` 到目标项目、`selectTab(type)`，复用既有 120ms 锚点定位+高亮模式定位 `workspace-${type}-${id}`；定位失败给提示（条目不在当前项目快照中）。

**5. 错误与降级**：条目不存在/不属于当前项目 → `open_*` 返回 `INTAKE_NOT_FOUND`；意图直达定位不到锚点 → 复用 `searchLocateNotice` 提示。有 UI 走详情页（IntakePanel，含正文+绑定 Plan 卡片）或对话内富卡片弹窗；无 UI（如仅消费工具结果文本的外部链路）由 `open_*` 返回的详情文本兜底。

**6.（次要）MCP 工具一致性**：`mcpTools.js` 的 `create_requirement`/`create_feedback` 返回值追加应用内可分享链接（深链/路由）与详情字段，供外部 MCP 客户端（codex/claude CLI 等）也能拿到入口；不改变既有 `intakeSummary`/`intakeDetail` 契约。

## 总体验收标准

**最终验收命令**
- 单测：`npm test`（全量 `node --test` 套件，新增 `chatIntents.test.ts` 与扩展的 `chatTools.test.js`、`WorkspacePage` 结构断言必须通过）。
- 构建：`npm run build`（TypeScript + Vite + electron 构建无错误）。

**验收范围**
- 对话工具层：`chatTools.js` 的 `create_*` 富化与新增 `open_requirement`/`open_feedback`。
- 渲染层：`ChatView.tsx`（可打开富卡片 + 意图直达）、`WorkspacePage.tsx`（`handleOpenIntake` 切 tab+定位+下发）、`types.ts`（`ChatIntakeOpenRef`/`onOpenIntake`）、`chatIntents.ts`/`workspaceLocate.ts` 工具函数。
- MCP 层（次要）：`mcpTools.js` 创建结果附带可分享链接/详情。
- 相关测试：`chatTools.test.js`、`chatIntents.test.ts`、`WorkspacePage.test.tsx`、`WorkspacePage.projectId.test.js`。

**通过标准**
1. 对话中通过 `create_requirement`/`create_feedback` 创建后，回复（工具结果卡片）包含可直接打开的入口（按钮/链接），并带标题、状态；点击即可跳转到对应需求/反馈并定位高亮。
2. 用户输入“打开需求 #35”/“打开反馈 #xx”（含全角＃、空格等容忍形式）时，系统可确定性定位并打开对应条目（切 tab + 锚点定位+高亮）。
3. AI 也可通过 `open_requirement`/`open_feedback`（按 ID 或标题）打开；返回结果含标题、正文、状态、绑定执行计划（`linkedPlan` 标题+进度）等详情，并在卡片提供打开入口。
4. 有 UI 环境：优先以需求/反馈详情页（IntakePanel，含正文+绑定 Plan 卡片）或对话内富卡片/弹窗打开；无 UI（无 `onOpenIntake` 或仅消费工具文本的链路）：在对话中展示详情文本并/或提供可复制链接。
5. 条目不存在或不属于当前项目（无权限）时：`open_*` 返回 `INTAKE_NOT_FOUND` 明确错误；意图直达定位不到锚点时给出明确提示；不静默成功、不抛未处理异常。
6. 工具结果契约仅做增量（新增 `type`/`projectId`/`openable`/`linkedPlan`/链接），不破坏既有 `id/title/status` 与 `chatController` 落库/事件链路；MCP `intakeSummary`/`intakeDetail` 向后兼容。
7. `ChatView` props 变更同步更新相关结构断言测试，既有渲染（思考态、流式、composer、通用 JSON 弹窗、其它 tab）不回归。
8. `npm test` 全绿，`npm run build` 成功。
