# 需求 #70 开发计划：新增验收模块（对已完成计划/任务逐项验收）

## 背景

AutoPlan 工作区左侧导航分为「工作区」（概览 / 需求 / 反馈）与「执行」（任务与计划 / 脚本 / 事件流 / 设置）两组（`src/renderer/components/workspace/WorkspaceSidebar.tsx:14-30`）。品牌副标题早已写明「需求 · 计划 · 执行 · 验收」，但「验收」这一支柱一直只有**循环自动验收阶段**（`src/loop/validation.js` 在所有任务执行完毕后跑「完整验收」任务、把计划推进到 `completed`），**缺少人面对已完成产物做最终确认的模块**。

数据层面：`plans` 与 `plan_tasks` 表（`src/database.js:133-168`）只有执行态 `status`（计划 `pending/running/ready_for_validation/validation_failed/completed/interrupt/draft`；任务 `pending/running/completed/done/passed/...`），**没有任何「是否已通过人工验收」的标志位**。任务已完成判定在渲染层为 `status ∈ ['completed','done','passed','accepted']`（`src/renderer/utils/planTasks.ts:87`），计划已完成为 `status === 'completed'`（`src/loop/taskExecution.js:359-369` 在全部任务完成后置 `ready_for_validation`，`validation.js` 验收通过后置 `completed`）。

本需求要新增一个**验收模块**，位置排在「反馈」模块下面；打开后展示**「已完成但未验收」的计划与任务**，并支持**像 todo 一样逐项勾选验收**（可取消）。流程上要求**先用 HTML 设计一版设计稿存到 `ui/` 文件夹，再开始模块开发**。

现状补充：渲染层快照（`src/loop/snapshots.js`）的 `SELECT * FROM plans`（`:32`）与 `SELECT plan_tasks.*, plans.file_path`（`:36`）会自动带出新列，无需改查询；`src/loopService.js` 已有 `runTask`/`stopTask`/`runTaskBatches` 等动作的写法与 `addEvent` 事件流记录；`src/main.js` 已用 `ipcMain.handle('tasks:run'|'tasks:stop'|'feedback:create'...)` + `requiredProjectId`/`requiredRecordId` 风格注册通道，`src/preload.js` 已用一行式 `ipcRenderer.invoke('...', input)` 暴露 API；`WorkspacePage.tsx`（约 540 行，默认上限 600）已用「每个 tab 一个 `<section className="view">` + `tabTitle`/`tabSubtitle` 文案表」的模式渲染视图。`validation.js` 中既有的 `acceptanceTask` 指循环**自动**验收任务，与本次「人工逐项验收」概念正交、不冲突。

## 设计约束

- **先做 HTML 设计稿再做开发**：需求明确「先用 html 设计一个设计稿，存到 ui 文件夹；然后开始模块开发」。`ui/acceptance.html` 复用 `ui/tasks.html`/`ui/workspace.html` 的设计 token（`--brand-*`/`--bg-*`/`--text-*`/`--success` 等、`--r-*` 圆角、`--shadow-*` 阴影、icon-sprite 用法与 `.sidebar`/`.topbar`/`.view` 布局骨架），静态预览验收模块布局（顶部说明 + 待验收清单 + 已验收区 + 空态）；设计稿先行，后续视图实现以稿为准。
- **「验收」是独立于执行态的标志位，不挪用 status**：执行态（pending/running/completed/...）描述「机器是否跑完」，验收态描述「人是否确认」。二者正交——一个任务可能 `status='completed'` 但「未验收」。故新增**可空时间戳列** `accepted_at TEXT`（NULL=未验收、非空 ISO 时间=已验收、清空即「取消验收」），落在 `plans` 与 `plan_tasks` 两张表；**不新增 status 取值、不改 `matchesTaskStatusFilter` 的「已完成」集合**（`['completed','done','passed','accepted']` 保持不变）。`accepted_at` 同时承担「何时验收」的时间信息，供事件流/展示复用。
- **「已完成」判定沿用既有语义**：计划已完成 = `status === 'completed'`（已通过循环自动验收阶段）；任务已完成 = `status ∈ ['completed','done','passed']`（与渲染层 `matchesTaskStatusFilter(task,'completed')` 一致）。**待验收集合** = 已完成 `AND accepted_at IS NULL`；**已验收集合** = `accepted_at IS NOT NULL`（含可取消回退）。
- **逐项验收「像 todo」**：验收视图把「待验收计划」作为可勾选项，其下挂「该计划内待验收任务」作为子项（两级 checklist），勾选即「验收」、取消勾选即「取消验收」；提供「全部验收」便于对单个计划或当页批量处理。勾选/取消走 IPC → 落库 `accepted_at` → 记事件 → 回灌 `loop.snapshot(projectId)` 刷新 UI；不做脱离 snapshot 的本地乐观状态，避免与轮询快照不一致。
- **导航位置**：新 tab `acceptance` 加入 `WORKSPACE_NAV`（「工作区」分组），**排在 `feedback` 之后**（满足「排列到反馈模块下面」）；`WorkspaceTab` 联合类型、`tabTitle`/`tabSubtitle` 文案表、`WorkspacePage` 的 `<section>` 视图块同步补 `acceptance`。验收模块是既有计划/任务的**筛选视图**，不引入新的搜索数据源——全局搜索命中计划/任务仍跳转 `tasks` tab（`WorkspaceSearchSourceConfig.targetTab`/`WorkspaceTab` 仅做联合类型放宽，不新增搜索源类型）。
- **事件流一致**：验收/取消验收记录 `plan.accepted`/`plan.unaccepted`/`task.accepted`/`task.unaccepted` 事件，消息与 meta 风格与既有 `feedback.created`/`mcp.stopped` 一致（`loop.addEvent(projectId, type, message, meta)`），事件流 tab 可见，便于审计。
- **IPC/preload/类型沿用既有风格**：通道命名 `namespace:action`（`acceptance:accept`/`acceptance:unaccept`），入参 `{ projectId, targetType: 'plan'|'task', id }`，失败抛中文错误；通道返回 `loop.snapshot(projectId)` 刷新 UI。`preload.js` 在 `window.autoplan` 上以一行式 `ipcRenderer.invoke` 暴露 `acceptItem`/`unacceptItem`（命名与 `runTask`/`stopTask` 一致）。`types.ts` 给 `Plan`/`PlanTask` 补 `accepted_at: string | null`、`WorkspaceTab` 联合补 `'acceptance'`。
- **文件长度约束**：`npm run check`（含 `scripts/check-file-lengths.js`）默认单文件上限 600 行，`WorkspacePage.tsx` 现约 540 行——新增一个 `<section>` 视图块后可能逼近上限，故**验收视图必须抽成独立组件** `src/renderer/components/workspace/AcceptanceView.tsx`，`WorkspacePage` 只负责透传 controller 字段/动作；其余新增尽量复用既有 `workspaceForms.ts`/`planTasks.ts` 工具与设计 token，不重复声明 `--brand-*` 等。
- **schema 迁移沿用 `ensureColumn`**：`src/database.js` 的 `ensureColumn('plans','accepted_at','TEXT')` 与 `ensureColumn('plan_tasks','accepted_at','TEXT')`（与 `started_at`/`finished_at` 等列同款增量迁移，`:252-257`），对既有库幂等补列、默认 NULL（未验收）。
- **不回退既有能力**：循环自动验收阶段、计划/任务执行链路、需求/反馈 intake、事件流、CLI 后端、MCP 工具链路、工作区导航、全局搜索均不得回退；`matchesTaskStatusFilter` 的「已完成」集合与 `groupTasksByPlan` 等既有渲染逻辑保持原语义（仅在新工具中复用其判定，不改其实现）。
- **不要把运行测试/回归/构建拆成普通开发任务**；最终检查统一放入最后的「完整验收」节点。
