# 需求 #21 开发计划：需求/反馈输入框支持粘贴图片并确保附件可读

## 背景

当前需求、反馈界面已经支持通过文件选择和拖拽添加附件，附件会持久化到本地附件目录并在快照中返回。但输入框尚未明确支持从剪贴板直接粘贴图片；同时，需求/反馈进入计划生成流程时，AI 工具只看到正文内容，附件路径与可读取说明没有被稳定注入到 prompt 或计划上下文中。

本需求目标是：用户可以在需求、反馈输入框内直接粘贴截图或图片，图片按附件进入现有保存链路；当这些需求/反馈触发计划生成和后续开发执行时，工具能看到附件清单、读取到附件文件内容，并能在必要时基于图片内容整理计划。

## 设计约束

- 需求与反馈共用同一套输入组件能力，不能只支持其中一个页面。
- 粘贴纯文本、输入正文、Enter 发送、文件选择、拖拽上传等既有交互不能回归。
- 粘贴图片应进入现有待提交附件列表，提交前允许预览和移除；只粘贴图片、正文为空时仍允许提交。
- 剪贴板图片通常没有稳定文件路径，保存链路需要支持无路径的二进制图片来源，不能依赖 `File.path`。
- 附件保存后需要继续记录原始名称、MIME、大小、hash、持久化路径，并保持现有快照与搜索/展示行为兼容。
- 计划生成 prompt 必须包含需求/反馈附件清单和可读取路径；如果附件缺失或读取失败，要有可追踪的日志或事件提示。
- 不应把大体积图片直接内联进 prompt；prompt 中传路径、hash、MIME 和读取说明，由工具按需读取文件内容。

  - 验收要点：`PendingAttachment` 或等价类型能够区分路径附件与剪贴板图片附件，至少包含来源类型、名称、MIME、大小、预览所需信息和保存所需载荷。
  - 验收要点：`WorkspacePage` 的待提交附件状态可以同时容纳文件选择/拖拽附件和粘贴图片附件，去重、移除、提交后清空逻辑均不破坏现有路径附件。
  - 验收要点：需求与反馈两个 `Composer` 实例使用相同的数据模型和回调，不出现一端可粘贴、一端不可粘贴的分叉实现。
  - 验收要点：TypeScript 类型检查能覆盖 create/update 入参中的两类附件来源，避免主进程收到结构不明的附件对象。

- [ ] P002: 实现输入框粘贴图片交互 <!-- scope: src/renderer/components/Composer.tsx,src/renderer/pages/WorkspacePage.tsx,src/styles.css -->
  - 验收要点：在需求输入框和反馈输入框中粘贴截图、PNG、JPEG、WebP 等图片时，图片会立即出现在待提交附件列表中。
  - 验收要点：粘贴纯文本时仍按浏览器默认行为进入正文，不会被附件处理逻辑阻断；同时粘贴文本和图片时，文本与图片都能保留。
  - 验收要点：多个剪贴板图片项可以一次性加入；不支持的剪贴板项会被忽略且不弹出误导性错误。
  - 验收要点：待提交图片显示缩略图、文件名、大小，并可在提交前移除；附件列表过长时布局不遮挡发送按钮。
  - 验收要点：仅有粘贴图片、正文为空时允许发送；发送成功后正文与待提交附件都清空，发送失败时保留待提交附件。

- [x] P003: 支持剪贴板图片持久化保存 <!-- scope: src/attachments.js,src/main.js,src/renderer/types.ts -->
  - 验收要点：`saveAttachments` 或等价保存函数既支持现有 `path` 文件复制，也支持剪贴板图片的二进制/base64 载荷写入持久化附件目录。
  - 验收要点：剪贴板图片保存后写入 `attachments` 表，字段包含 `project_id`、`owner_type`、`owner_id`、`original_name`、`stored_path`、`mime_type`、`size`、`hash`、`created_at`。
  - 验收要点：保存时根据 MIME 推导安全扩展名和默认文件名，文件名经过清洗，不能产生路径穿越或覆盖其他附件的问题。
  - 验收要点：保存后的 `size` 与实际写入字节数一致，`hash` 基于真实文件内容计算；重复粘贴同一图片不会导致数据库记录指向错误文件。
  - 验收要点：现有文件选择、拖拽上传、需求更新、反馈更新的附件保存行为保持兼容。

- [x] P004: 注入附件可读清单到计划生成 <!-- scope: src/loopService.js -->

  - 验收要点：附件清单至少包含原始名称、MIME、大小、SHA256、持久化本地路径，并明确说明工具可以通过该路径读取附件内容。
  - 验收要点：有附件时，生成出的 plan 文件应要求保留必要的附件路径或引用，避免后续执行任务时失去图片上下文。

  - 验收要点：无附件的需求/反馈继续生成原有简洁 prompt，不额外引入空清单或无意义噪声。

## 总体验收标准

- 需求界面和反馈界面的输入框均支持直接粘贴图片，粘贴后图片进入待提交附件列表并可预览、移除和随正文一起提交。
- 粘贴文本、输入正文、文件选择、拖拽上传、仅正文提交、仅附件提交、发送失败保留草稿等核心交互均符合预期。
- 路径文件附件与剪贴板图片附件都能持久化到本地附件目录，并在数据库、项目快照和前端展示中保持正确 owner 关系。
- 需求/反馈触发计划生成时，AI CLI prompt 能获得附件清单和本地可读路径；工具按该路径读取到的文件内容与数据库记录的大小/hash 一致。
- 附件缺失、不可读、类型异常或保存失败时，用户界面或事件日志能看到可定位的错误信息，不会静默生成缺失上下文的计划。
- 安全性满足基本边界：文件名清洗、路径穿越防护、MIME/扩展名合理推导、大体积附件不内联到 prompt、图片预览不执行脚本。
- 验证通过：`npm run check` 与 `npm run smoke` 均通过；手动验收至少覆盖从截图工具复制图片后分别粘贴到需求和反馈输入框并生成计划。
