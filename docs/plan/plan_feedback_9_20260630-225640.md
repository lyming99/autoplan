# 反馈 #9 开发计划：草稿计划执行后退出草稿态

## 背景与目标

反馈要求：“草稿执行后，应该将 plan 变成非草稿状态”。基于当前代码，草稿计划由 `plans.status = 'draft'` 表示，快照层在 `src/loop/snapshots.js` 中根据 `status === 'draft'` 派生 `is_draft`，前端计划列表在 `src/renderer/components/PlanLists.tsx` 和 `src/renderer/components/plans/PlanList.tsx` 中用 `plan.is_draft || plan.status === 'draft'` 判断是否显示为草稿。

现有后端已有 `LoopService.activateDraftPlan()`，并在 `runTask()`、`runTaskBatches()` 与 `processPlan()` 前调用，但仍需要加固三处风险：后端激活应返回数据库最新非草稿计划对象；任务同步、进度刷新和并发执行不能继续使用陈旧 draft 状态；前端“执行草稿”按钮当前直接调用 `window.autoplan.runTask()`，没有消费 IPC 返回的最新 snapshot，可能导致列表在执行后仍停留在草稿视觉状态直到下一次更新事件。

目标是：用户显式执行草稿计划后，该计划立即从 `draft` 流转为非草稿执行态，并在数据库、快照、计划 Tab、计划卡片、关联需求/反馈状态中保持一致。

## 任务拆解

  - 验收要点：`activateDraftPlan()` 对 `status='draft'` 的计划执行原子化状态变更，成功后数据库 `plans.status` 必须不再是 `draft`，返回给后续执行链路的 plan 对象也必须是最新非草稿状态。
  - 验收要点：若状态已被其它入口提前改为非草稿，激活逻辑应读取并返回当前数据库记录，避免继续把陈旧 draft plan 传入 `executeTask()`、`completeTask()`、`validatePlan()` 或并发批处理。
  - 验收要点：手动单任务执行、草稿并发执行、自动 `processPlan()` 三条入口均复用同一激活语义；`plan.draft.started` 事件只在真正从 draft 转为非草稿时记录一次。

  - 验收要点：计划列表的“草稿”Tab 数量减少，“执行中/已完成”分类按真实状态接收该计划，计划卡片不再显示草稿样式或“执行草稿”按钮。
  - 验收要点：`PlanLists.tsx` 的草稿执行入口应消费 `tasks:run` / `tasks:runParallel` 返回的最新 snapshot，或通过父级回调统一走 `runLoopAction`，确保用户点击完成后 UI 立即刷新到非草稿状态。
  - 验收要点：关联需求/反馈的 `linked_plan_status` 与计划快照保持一致，不继续显示 `draft`。

  - 验收要点：smoke 场景覆盖草稿计划的并发执行入口，确认 `runTaskBatches()` 后计划同样退出 draft 状态，并记录既有并发完成事件。
  - 验收要点：渲染层源码测试覆盖草稿执行入口会写回返回 snapshot 或调用父级统一刷新回调，避免只依赖异步 update 事件刷新 UI。

## 总体验收标准

最终验收命令：

```powershell
npm run check
npm run smoke
npm run build
```

验收范围：
- 后端 `plans.status` 从 `draft` 到非草稿执行态的流转。
- `runTask()`、`runTaskBatches()`、`processPlan()` 三条执行入口。
- `syncPlanTasks()`、进度刷新、最终验收后的计划状态延续。
- `loop.snapshot()` 中 `status`、`is_draft`、关联需求/反馈 `linked_plan_status` 的一致性。
- 前端计划列表 Tab、计划卡片、草稿执行按钮和返回 snapshot 刷新链路。

通过标准：
- 草稿计划被用户显式执行后，数据库、快照和 UI 均不再显示为草稿。
- 单任务执行、并发执行、全部任务完成进入验收、验收通过完成计划的状态链路均正确。
- 普通计划自动执行、任务列表手动执行、计划并发执行、计划阅读器、需求/反馈关联计划展示无回归。
- 最终验收命令全部通过，且没有新增文件长度、TypeScript、Node 语法或构建错误。
