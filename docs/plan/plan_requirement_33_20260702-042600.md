# Plan: 根据设计稿重新开发对话界面

## 需求概述

需求 #33 要求按 `file:///D:/project/GitHub/autoplan/ui/chat.html` 重新开发 AutoPlan 工作区的对话界面。设计稿明确对应 `ChatView.tsx` 的全部交互态：会话列表下沉到左侧主导航的「对话」项下方，主区只保留消息流与输入 composer；输入区改为与需求/反馈界面一致的圆角悬浮 composer，并增加模型配置药丸下拉；消息气泡、工具调用卡片、思考折叠区、流式光标、系统消息、空状态、浅深色 token 均需对齐 `workspace.css`。

## 现状分析

- 当前项目是 Electron + React + Vite，工作区入口在 `src/renderer/pages/WorkspacePage.tsx`，对话页通过 `activeTab === 'chat'` 渲染 `ChatView`。
- 当前 `ChatView.tsx` 内部自带 `ConversationSidebar`，样式是独立的 `.chat-sidebar` 二级侧栏；这与设计稿要求的主侧栏嵌套对话列表不一致。
- `useChat.ts` 已负责会话列表、当前会话、AI 配置、流式消息、工具调用、思考阶段、发送/停止/清空等状态；重构时应优先复用这条数据流，避免重复加载或重复订阅流式事件。
- `WorkspaceSidebar.tsx` 当前只接收导航、项目、循环状态和脚本数量，不知道聊天会话状态；要实现设计稿，需要把聊天会话列表和操作回调暴露给主侧栏。
- `workspace.css` 已有聊天样式、composer 通用样式、设计 token 和深色主题规则，但当前对话输入仍是 `.chat-input-bar`，需要替换为设计稿里的 `.chat-composer-zone` / `.chat-composer` 结构。
- 现有 `WorkspacePage.test.tsx` 使用源码级断言保护工作区结构和聊天配置回归，可继续补充对话界面结构、主侧栏嵌套列表、composer 和会话排序的回归断言。

## 设计落地原则

- 不硬编码设计稿里的示例模型名和示例会话，所有会话、模型配置、API Key 可用状态都从 `useChat` / `ai_configs` / `conversations` 实际数据派生。
- 保留现有聊天能力：发送、停止、清空、切换会话、创建会话、重命名、删除、历史加载、工具调用展示、思考内容折叠、Markdown 渲染、缺少 API Key 时仍可看见会话入口。
- 侧栏、消息流、composer 都使用现有设计 token，避免新增割裂的颜色体系；深色主题只做必要覆盖。
