# 反馈 #19 开发计划：Plan 列表选择与任务列表联动过滤

## 背景

反馈 #19 指出：“plan列表要支持选择，联动任务列表过滤plan任务”。当前任务与计划页同时展示 PlanList 与 TaskList，但计划列表主要承担阅读计划、执行计划任务和展示进度的职责，缺少显式选择态；任务列表虽然可按搜索结果间接受影响，但没有通过选择某个计划来只查看该计划任务的直接交互。需要补充 Plan 选择模型，让用户在计划列表中选中一个计划后，任务列表自动过滤为该计划关联任务，并提供清晰的选择态、重置入口和空状态说明。

## 设计约束

- 本计划只处理前端 Plan 选择态、任务列表联动过滤、相关类型/样式和必要交互覆盖，不改动任务执行、计划生成、数据库结构、后端 IPC 协议和计划文件解析逻辑。
- Plan 选择应是任务与计划视图内的浏览过滤状态，不应影响全局搜索条件、任务运行状态、计划阅读器打开状态或任务执行队列。
- 任务与 Plan 的关联优先使用 `task.plan_id === plan.id`，必要时兼容已有 `file_path` 关联兜底，避免历史数据无法被过滤出来。
- 当选中 Plan 没有关联任务、任务被搜索条件排除或快照刷新后 Plan 不存在时，需要给出明确空状态或自动清理失效选择。
- 不要把运行测试、回归、构建拆成普通开发任务；最终检查统一放入最后的“完整验收”节点。

## 总体验收标准

- Plan 可选择：计划列表支持选中某个 Plan，选中态清晰可见，并提供恢复查看全部任务的入口。
- 任务联动过滤：选中 Plan 后任务列表仅展示该 Plan 关联任务；未选中 Plan 时保持原有任务列表展示逻辑。
- 过滤可解释：搜索过滤、任务状态过滤和 Plan 选择同时存在时，任务列表有明确文案说明当前过滤来源和结果数量。
- 边界可恢复：选中 Plan 无任务、任务被搜索排除、Plan 被删除或快照刷新后失效等场景都有空状态、提示或自动清理策略。
- 现有功能不退化：计划阅读、计划任务运行、任务单项运行/停止、任务分组展示、搜索结果展示等既有能力不因本次改动退化。
- 最终验收命令：执行 `npm run check && npm run smoke`；验收范围为 Plan 列表选择、任务列表联动过滤、清空选择、组合过滤提示、边界空状态和现有任务与计划主流程；通过标准为命令退出码为 0，且手动验证上述交互均符合反馈要求。
