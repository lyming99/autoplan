# 反馈 #22 开发计划：让新建脚本时的脚本名称输入框可被发现

## 背景

反馈 #22：在「脚本」模块点击「新建脚本」后，直接点「创建」会提示「请填写脚本名称」，但用户找不到在哪里填写脚本名称。

根因定位：脚本名称输入框**确实存在**，位于脚本详情弹窗的头部 `src/renderer/components/workspace/ScriptEditorModal.tsx`（`<input className="mh-title-input" value={draft.name} placeholder="未命名脚本" … />`）。但它被样式化成了一段**纯文本标题**：

- `src/renderer/styles/workspace.css:393` 中 `.mh-title-input { border: 0; outline: none; background: transparent; padding: 0; … }` —— 无边框、透明背景、无内边距、加粗，视觉上与静态标题文字完全一致。

因此用户打开新建弹窗时看不到一个「可输入的框」，留空点「创建」后 `validateScriptDraft`（`src/renderer/utils/workspaceForms.ts:343`）返回「请填写脚本名称」，但用户不知道该往哪里输入。

本次计划的目标：让脚本名称输入框在**视觉与交互上可被发现**——具备可识别的输入框外观、新建空态有明确引导文案、新建态自动聚焦，使用户在看到「请填写脚本名称」提示时能立刻定位填写位置。

范围界定：

- 仅修复「看不到在哪填脚本名称」的可发现性问题；不改动脚本校验规则（`validateScriptDraft` 文案与逻辑保持不变）、不改动保存/运行/删除/复制流程与后端持久化。
- 不调整弹窗右侧配置面板（描述、触发方式、执行设置、上下文注入）各字段，也不调整脚本列表卡片视图。
- 文件行数约束：`ScriptEditorModal.tsx` 当前 582 行（`scripts/check-file-lengths.js` 默认上限 600 行，仅余 18 行），新增聚焦逻辑须控制在限额内；如确实超出，再按既有先例在该脚本中为该文件单独放宽上限并注明原因，但首选通过精简控制行数。

## 总体验收标准

- 功能范围：仅修复新建脚本时脚本名称输入框「看不到在哪填」的可发现问题；不改动脚本校验规则（`validateScriptDraft` 文案与逻辑）、保存/运行/删除/复制流程与后端持久化行为。
- 可发现性范围：脚本名称输入框需同时满足——视觉上具备可识别的输入框外观（边框/背景/聚焦态）、新建空态占位符为明确引导文案、新建态自动聚焦。三者结合确保用户在出现「请填写脚本名称」提示时能立刻定位输入位置。
- 兼容范围：弹窗头部布局（标题行、状态 chip、操作按钮）、右侧配置面板各字段（描述、触发方式、执行设置、上下文注入）、代码编辑器与运行日志区、脚本列表卡片视图均不受影响。
- 代码约束：改动不得触发 `scripts/check-file-lengths.js` 报错（`ScriptEditorModal.tsx` 余量紧张，须重点核对）；超出时按既有先例为该文件单独放宽上限并注明原因。
- 最终验收命令：`npm run build` 与 `npm run check`。
- 通过标准：两个命令均返回 0；三项人工验证全部满足；脚本名称输入框在新建与编辑态均清晰可发现、可编辑、可保存。
