# 需求 #37 开发计划

## 需求理解

**需求原文：**「ai配置，属于全局配置」。

结合代码实际分析，当前 AI 配置是**项目级（project-scoped）**的，需要提升为**全局级（app-level / 跨项目共享）**：

1. **数据库层**：`ai_configs` 表带 `project_id INTEGER` 列与索引 `idx_ai_configs_project`（`src/database.js`）。`migrateChatToAiConfigs` 会为**每个项目**各创建一条「默认配置」，导致不同项目重复存同样的 Provider/Key。

2. **服务层**：`aiConfigService.js` 的 `listAiConfigs(db, projectId)` 按 `project_id` 过滤；`createAiConfig` 经 `normalizeProjectId` **强制要求 projectId**（为空抛错）；`resolveAiConfigForConversation` 的项目首条配置回退也限定在当前 `project_id` 内。

3. **IPC 层**：`ai-config:list` / `ai-config:create`（`src/main.js`）都先调 `requiredProjectId`，无项目上下文直接抛「项目不存在」。

4. **渲染层**：`useChat(projectId)` 调用 `aiConfigList({ projectId })`；`WorkspaceSettingsView` 的「AI 对话」面板用 `hasValidProjectContext` 把「新建配置」按钮禁用掉（无项目时无法配置 AI）。`workspaceForms.aiConfigInputFromForm(projectId, ...)` 同样强制 projectId。

5. **绑定关系**：`conversations.ai_config_id` 引用项目级配置；`chatController.resolveConversationAiConfigId` 还会校验配置归属同一项目（`WHERE id=? AND project_id=?`）。

**目标：** AI 配置（Provider / Base URL / API Key / Model / Temperature / 思考深度）一次配置、所有项目与所有对话共用——这正符合用户的心智模型（API Key 是用户的、不是某个项目的）。`conversations` 仍保持项目归属，但它引用的是**全局** AI 配置。

**最合理工程方案（不另建全局设置壳）：** 保留现有「AI 对话」设置面板，但使其读写**全局**配置集合；`ai_configs.project_id` 改为可空，全局配置存 `NULL`；旧的项目级数据迁移为全局（按 name/provider/base_url/model 去重，并重映射 `conversations.ai_config_id`）。这样改动集中、向后兼容、无需引入全新的全局设置路由。
