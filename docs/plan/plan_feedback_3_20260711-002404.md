# 修复 MCP 批量需求未按创建顺序执行

## 需求概要

反馈 #3 指出，通过 MCP 连续创建多个有先后关系的需求后再启动 AutoPlan，计划生成与任务执行没有稳定遵循需求创建顺序。当前 `IntakeService.createRequirement` 会按调用顺序写入需求，但 `LoopService.runOnce` 选择待处理需求时只使用 `created_at ASC`，同一毫秒内创建的记录缺少 `id` 次序保证；而每轮又只取查询结果第一条生成计划，因此入口顺序不稳定会继续传导到 Plan 的 `sort_order` 和执行队列。本次修复应建立明确、可重复的需求 FIFO 选择规则，并验证 MCP 批量创建、循环启动、计划生成及队首执行的完整链路，同时保留现有计划状态、生成失败退避和草稿计划语义。

## 任务拆解

- [x] P001: 为待处理需求建立稳定的 FIFO 选择与入队规则 <!-- scope: src/loopService.js -->
  - 验收要点：同一项目内未生成计划且仍可处理的需求按 `created_at ASC, id ASC` 稳定排序，`runOnce` 每轮只选择最早的一条生成计划。
  - 验收要点：多个需求具有相同 `created_at` 时，以递增 ID 保持 MCP 实际落库先后，重复启动或扫描不会改变队首。
  - 验收要点：先创建需求生成的 Plan 先取得较小的 `sort_order`，后创建需求生成的 Plan 只追加到队尾，不会覆盖或抢占已有前序 Plan。
  - 验收要点：已完成、已关闭、已有 Plan 或处于既有生成失败退避条件的需求仍按原规则过滤，草稿计划、分阶段计划和单次循环只生成一个需求计划的现有语义保持不变。
- [x] P002: 补充 MCP 批量需求到循环执行队列的顺序回归覆盖 <!-- scope: src/loopService.test.js,src/mcpTools.test.js -->

  - 验收要点：覆盖多个需求时间戳完全相同的场景，断言按需求 ID 递增生成计划，避免依赖 SQLite 未声明的返回顺序。
  - 验收要点：覆盖前序 Plan 尚有待执行任务时，后续需求即使已经生成 Plan 也只在队列中等待；前序 Plan 完成后才推进下一 Plan。
  - 验收要点：覆盖不同时间戳、已关联 Plan、终态需求和生成失败退避记录混合存在的场景，确认 FIFO 修复只作用于当前可处理队列且不会重复生成计划。
  - 验收要点：覆盖空队列首次生成的非草稿 Plan 仍可在同一轮执行，以及草稿 Plan 不会被自动执行的既有行为。
