# 需求 #26 开发计划：计划显示 CLI 后端与思考深度

## 背景

当前项目已支持项目级、需求级和反馈级配置 CLI 后端，并能把 Codex 思考深度传递给执行链路。新需求要求“每个计划”都能显示当前使用的 CLI 后端与思考深度，因此需要把生成该计划时实际采用的 CLI 配置固化为计划级元信息，并在计划列表、Plan 全文阅读与相关任务展示中使用同一套展示口径。

## 设计约束

- 计划显示的信息必须来源于该计划生成时实际解析出的 CLI 配置快照，而不是页面当前项目默认配置，避免后续切换项目级配置后误导历史计划。
- Codex 计划显示 CLI 后端与思考深度；Claude 计划显示 CLI 后端，并明确思考深度不适用或不展示为有效参数。
- 历史计划缺少计划级 CLI 字段时必须兼容：后端默认按 Codex、思考深度默认按 medium 或以任务/事件中已有字段回填展示，不得导致列表或阅读器崩溃。
- UI 展示文案需要与现有 `agentCliProviderLabel`、`codexReasoningEffortLabel` 保持一致，避免“后端”“CLI”“思考深度”多套命名混用。
- 计划元信息只用于展示和追踪，不改变任务执行时已有的单条记录优先级、项目级默认值和 Codex/Claude 调用逻辑。
- 最终验收节点只负责整体验收，不参与普通开发任务并发或功能实现。

## 总体验收标准

- 功能完整性：每个计划都能在计划列表和 Plan 全文阅读器中展示生成该计划时实际使用的 CLI 后端；Codex 计划同时展示规范化后的思考深度。
- 数据一致性：计划级 CLI 元信息随计划持久化，快照、类型定义和 UI 读取同源；历史计划、缺失字段和未知后端有稳定降级表现。
- 兼容性：现有需求/反馈单条 CLI 覆盖、项目级默认配置、任务执行、失败修复、验收重试和事件日志逻辑不被改变。
- 可读性：新增展示不挤压计划进度、验收状态、并发建议、任务操作和全文阅读体验，不出现空值、英文内部字段名或重复分隔符。
- 最终验收命令：执行 `npm run check` 和 `npm run smoke`；两条命令均退出码为 0，并完成手动 UI 验收覆盖 Codex、Claude、历史兼容和切换默认配置后历史计划不漂移四类场景。
