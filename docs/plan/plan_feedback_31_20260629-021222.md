# 反馈 #31 开发计划：计划与任务列表按时间倒序排列

## 背景

反馈 #31：「计划和任务，都应该按照时间倒序排列，最新的放前面」——用户希望工作区里「计划记录」与「任务记录」两列的展示顺序都严格按时间倒序，最新活动的排在最前。逐项定位当前排序实现如下：

1. **数据来源（快照层）**
   - `snapshot`（`src/loop/snapshots.js:31-41`）查询 plans 与 tasks 时分别用 `ORDER BY sort_order ASC, created_at ASC, id ASC`（`snapshots.js:32`）与 `ORDER BY plans.sort_order ASC, plans.created_at ASC, plans.id ASC, plan_tasks.sort_order ASC, plan_tasks.id ASC`（`snapshots.js:39`），即数据库层是**升序**（旧在前）。
   - 但这两份数据交给前端后会各自重新排序，故数据库顺序不直接决定最终展示；需求/反馈（`snapshots.js:78、89`，`updated_at DESC`）、事件（`snapshots.js:108`，`id DESC`）已是倒序，不在本次范围。

2. **计划列表（`PlanList`）——已满足时间倒序**
   - `PlanList`（`src/renderer/components/PlanLists.tsx:354`）以 `orderedPlans = [...plans].sort(comparePlanOrder)` 重排；`comparePlanOrder`（`PlanLists.tsx:469-475`）主键为 `created_at` 倒序（`String(right.created_at).localeCompare(String(left.created_at))`，最新创建在前），并以 `Number(right.id) - Number(left.id)` 倒序兜底保证稳定。
   - 各状态 tab（全部/执行中/已完成/草稿，`PlanLists.tsx:356`）均基于 `orderedPlans` 过滤，顺序一致；且已有源码断言测试（`src/renderer/pages/WorkspacePage.test.tsx:248-265`）明确「`comparePlanOrder` 不应以 `sort_order` 升序作为主键」——即计划侧刻意按创建时间倒序，**已满足反馈**，本次无需改动。

3. **任务列表（`TaskList`）——分组排序存在「运行中置顶」破坏时间倒序**
   - `TaskList`（`PlanLists.tsx:91`）调用 `groupTasksByPlan`（`src/renderer/utils/planTasks.ts:144-200`）分组渲染。
   - 分组**内**任务排序（`planTasks.ts:173-182`）以 `taskSortTime`（`planTasks.ts:96-98`，`finished_at ?? started_at ?? updated_at`）倒序为主键、原始 `index` 兜底——**已满足时间倒序**。
   - 分组**之间**排序（`planTasks.ts:191-199`）当前为：`hasRunningTask`（运行中分组）优先置顶 → `sortTime`（`latestTaskTime`，`planTasks.ts:100-108`，分组内最新活动的 finished_at/started_at/updated_at）倒序 → `firstIndex` 兜底。**第一优先键 `hasRunningTask` 会把「较早创建但当前有任务运行中」的 Plan 分组排到「最新创建但无运行任务」的 Plan 分组之前，违反反馈要求的纯时间倒序。**

4. **修复方向**
   - 移除 `groupTasksByPlan` 分组排序（`planTasks.ts:191-199`）中的 `hasRunningTask` 优先键，使分组顺序完全由 `sortTime`（最新活动时间）倒序决定，`firstIndex` 兜底保持稳定。
   - `hasRunningTask` 字段（`planTasks.ts:188`）与 `getDefaultExpandedTaskGroupKeys`（`PlanLists.tsx:314-318`，决定默认展开运行中分组）保持不变——它们控制「默认展开哪个分组」，与「排序」无关，运行中的任务仍默认展开、可见可操作。
   - 不改动计划列表排序（已 created_at 倒序）、分组内任务排序（已 taskSortTime 倒序）、数据层查询顺序、任务状态筛选/计数、计划卡片渲染、并发执行、会话恢复。

范围界定：

- 仅调整任务列表**分组之间**的排序为纯时间倒序（去掉 `hasRunningTask` 优先置顶）；不触碰计划列表、分组内任务排序、数据层 SQL、筛选/默认展开/并发/会话恢复等无关逻辑。
- 计划列表（`comparePlanOrder`，created_at 倒序）、需求/反馈列表（updated_at 倒序）、事件列表（id 倒序）顺序保持不变，且既有源码断言测试（`WorkspacePage.test.tsx:248-265`）继续通过。
- 文件行数约束（`scripts/check-file-lengths.js`，默认上限 600）：`planTasks.ts` 当前 245 行，改动极小，远在限额内。
