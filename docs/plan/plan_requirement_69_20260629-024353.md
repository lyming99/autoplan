# 需求 #69 开发计划：修复 macOS 下 MCP 因网络/沙箱权限无法启动的问题

## 背景

需求原话：「macos 系统，没有启动网络服务的权限，导致 mcp 服务无法启动；另外，要注意使用非 sendbox 机制，避免无法访问本地文件夹」。需求包含两个相关但独立的 macOS 症状：**(1) MCP 服务因无网络监听权限无法启动**；**(2) 必须以非沙箱机制运行，否则无法访问本地文件夹**。两者在 macOS 上同源——一旦 App Sandbox（OS 级沙箱）被启用，便会同时阻断入站网络监听（需 `network.server` entitlement）与限制任意本地文件访问（仅允许用户选择项与 App 容器），故本需求的根因治理落在**确保 macOS 构建为非沙箱**上。

现状梳理（结合代码）：

1. **MCP 默认以 HTTP 传输启动，绑定本机端口；该绑定失败时已有 catch，但缺少可读/可处置的错误指引**。`src/mcpConfig.js:1` 的 `MCP_DEFAULT_CONFIG` 固定 `transport:'http'`、`host:'127.0.0.1'`、`port:43847`、`path:'/mcp'`；`normalizeMcpTransport`（`mcpConfig.js:67`）只区分 `'stdio'` 与兜底 `'http'`，故入库默认即 HTTP。`src/mcpServer.js:77` 的 `startHttp()` 经 `http.createServer(...).listen(this.config.port, this.config.host, ...)`（`mcpServer.js:88`）绑定端口，`listen` 失败由 `this.httpServer.once('error', onError)`（`mcpServer.js:87`）reject，向上经 `start()` 的 catch（`mcpServer.js:48`）写入 `stoppedState(...).lastError`（`mcpServer.js:50`）并原样 rethrow（`mcpServer.js:51`）。**错误目前只携带底层 `error.message`（如 `EADDRINUSE`/`EACCES` 裸码），既不分类也不告诉用户该如何处置（切换 stdio / 放行防火墙 / 改端口）**。

2. **MCP 在应用启动时自动拉起，失败已被捕获并事件化，链路只需更好的文案**。`src/main.js:53` 的 `createApp()` 末尾调用 `scheduleMcpServerStart()`，由 `app.whenReady()`（`main.js:56`）触发；`scheduleMcpServerStart`（`main.js:427`）以 `startMcpServer().catch(recordMcpStartupError)` 兜底，`startMcpServer`（`main.js:433`）用 `mcpServerConfig(db)`（`main.js:436`）建实例并 `start()`，失败走 `recordMcpStartupError`（`main.js:452`）→ `loop.addEvent(projectId, 'mcp.start.failed', ...)`（`main.js:455`）+ `console.error`。`mcp.status().lastError` 经 IPC 透传到渲染层，`src/renderer/components/workspace/McpControlPanel.tsx:110` 与 `McpControlPanel.tsx:171` 已把 `mcp.lastError` 显示为「最近错误」。**即错误信息只要在 P002 里生成好，现有链路即可让 macOS 用户在界面看到处置指引，无需新增 IPC/事件通道**。

3. **stdio 传输已完整实现，但 GUI 进程内的 stdio 对外部 MCP 客户端无意义**。`mcpServer.js:110` 的 `startStdio()` 用 `@modelcontextprotocol/sdk` 的 `StdioServerTransport` 读/写**本进程的 stdin/stdout**。对独立 CLI（`package.json:20` 的 `npm run mcp:stdio` → `node src/mcpServer.js --stdio`，`mcpServer.js:286` 的 `startStandalone`）这是正确的——由外部 MCP 宿主（Claude Desktop 等）spawn 该进程并通过 stdio 通信；但对 Electron GUI 进程，其 stdin/stdout 不连接任何客户端，故 UI 里「传输方式=stdio」对 in-app 拉起而言**无法让外部客户端接入**。因此 macOS 网络受限场景的正确降级路径是「运行独立 stdio 进程」或「放行/非沙箱以恢复 HTTP」，而**不是**在 GUI 内切 stdio——这一点需在文档与错误指引里讲清，避免用户误用。

4. **macOS entitlements 已为非沙箱，但缺网络 entitlements，且需显式锁定非沙箱**。`build/entitlements.mac.plist`（及其 inherit 版）当前仅含 `com.apple.security.cs.allow-jit`、`com.apple.security.cs.allow-unsigned-executable-memory`、`com.apple.security.cs.disable-library-validation` 三项（硬运行时 JIT/动态库相关），**不含 `com.apple.security.app-sandbox`**（即当前确为非沙箱，能访问本地文件夹的根因前提已满足），也**不含任何 `network.*` entitlement**。`package.json:56` 的 mac 构建配置 `hardenedRuntime:true`、`entitlements:build/entitlements.mac.plist`、`entitlementsInherit:build/entitlements.mac.inherit.plist`、target 仅 `dmg`/`zip`（**非 `mas`，避免 Mac App Store 强制沙箱**），`afterSign:scripts/notarize-mac.js`（`package.json:29`）。本需求要把「非沙箱」从「恰好没开」升级为「显式锁定 + 网络兜底 + 非 MAS 校验」，并补一行注释固化该决策，防止后续误开沙箱。

5. **「非 sandbox 机制」指向 macOS App Sandbox（OS 级），而非 Electron 渲染进程沙箱**。`src/main.js:39` 的 `BrowserWindow` 用 `contextIsolation:true`、`nodeIntegration:false`（`main.js:48-49`）且无 `sandbox:true` 显式开关；Electron 33 在 `nodeIntegration:false` 时默认开启渲染层沙箱，但**文件访问发生在主进程**（IPC handler、`registerFileProtocol`（`main.js:460`）等），渲染层仅经 preload 暴露的 `window.autoplan.*` 调用——故渲染层沙箱不影响文件夹访问。需求中的「无法访问本地文件夹」与「无网络监听权限」成对出现，是 macOS App Sandbox 的典型表现，故治理对象是 OS 级沙箱/entitlements，**不动** Electron 渲染层沙箱（保持既有安全姿态）。

6. **既有链路成熟可复用、且必须不回退**。`normalizeMcpConfig`（`mcpServer.js:183`）/`mcpServerConfig`（`mcpConfig.js:34`）的多别名入参与 env 覆盖（`AUTOPLAN_MCP_TRANSPORT`/`AUTOPLAN_MCP_HOST`/`AUTOPLAN_MCP_PORT`/`AUTOPLAN_MCP_PATH`/`AUTOPLAN_MCP_ENABLED`）、`validateSafeHost`（`mcpServer.js:199`）的本机绑定守卫、stdio/HTTP 两条启动路径、`isAuthorizedRequest`（`mcpServer.js:240`）Bearer 鉴权、`stop()` 的幂等关闭均保持不变；Windows/Linux 平台行为零改动。

综上，本需求的核心工作为：**(P001)** 显式锁定 macOS 构建为非沙箱并补网络 entitlements 兜底 + 非 MAS 校验（一举覆盖「无法访问本地文件夹」与「无网络监听权限」的沙箱根因）；**(P002)** 在 `mcpServer.js` 对 HTTP `listen` 失败做错误分类并生成 macOS 平台感知的可处置指引（端口占用 / 网络监听权限或防火墙 → 提示放行或改用独立 stdio 进程）；**(P003)** 在 README 增补 macOS 专节（非沙箱/非 MAS、防火墙放行、in-app HTTP 与独立 stdio 进程的区别、相关环境变量）；**(P004)** 补充 MCP 监听失败错误分类的单测；**(P005)** 完整验收。

## 设计约束

- **macOS 必须保持非沙箱（OS 级 App Sandbox），这是两个症状的共同根因治理**：entitlements（`build/entitlements.mac.plist` + `.inherit.plist`）**不得**出现 `com.apple.security.app-sandbox`，否则网络监听需 `network.server`、文件访问被收窄到容器/用户选择项——正是需求所述两个症状。以注释固化「AutoPlan 以非沙箱直接分发（非 MAS），保证任意本地工作区文件夹可读写」。
- **补充网络 entitlements 仅作兜底/显式声明，不开启沙箱**：追加 `com.apple.security.network.server`（入站，供 in-app MCP HTTP 监听）与 `com.apple.security.network.client`（出站，供回写/上传等）。这两项是 App Sandbox 专属 entitlement，**非沙箱时无效也无害**，写入是为「万一沙箱被误开」与代码审查可追溯，不改变当前非沙箱运行语义。
- **构建目标保持 `dmg`/`zip`，绝不引入 `mas`**：Mac App Store（`mas` target）强制沙箱，与本需求「非 sandbox」直接冲突。`package.json:56` 的 mac target 仅 `dmg`/`zip`（x64+arm64）保持不变；`afterSign`/`notarize-mac.js`（`scripts/notarize-mac.js:94` 的 `notarizeMac`，非 darwin 自动 skip）行为不变。
- **错误指引在 `mcpServer.js` 内生成、平台感知，复用现有事件链路**：HTTP `listen` 失败按错误码（`EADDRINUSE`→端口占用；`EACCES`/`EPERM`→无网络监听权限或被防火墙拦截；其它→通用）映射为中文可处置文案，并在 `process.platform==='darwin'` 时追加「可放行防火墙/确认非沙箱，或改用独立 stdio 进程（`npm run mcp:stdio`）」的提示；该文案经既有 `lastError`→`mcp.start.failed` 事件→IPC→`McpControlPanel` 的「最近错误」直接展示，**不新增** IPC/事件/preload 通道。
- **不改默认传输、不改鉴权、不改 stdio 语义**：`MCP_DEFAULT_CONFIG.transport` 仍为 `'http'`（非 darwin 行为零变化）；macOS 不在代码层强制改默认传输为 stdio（避免误伤既有 HTTP 客户端与既有 in-app 行为），改由「非沙箱 entitlements + 防火墙放行」恢复 HTTP 可用性，stdio 作为文档化的独立进程降级路径；`isAuthorizedRequest`/`timingSafeEqualString`/`validateSafeHost` 不动；`startStdio`（`mcpServer.js:110`）逻辑不动。
- **macOS 平台判定用 `process.platform`，与既有风格一致**：`main.js:59` 的 `window-all-closed` 已用 `process.platform !== 'darwin'`；`mcpServer.js` 的平台感知文案沿用同一判据，不引入 `os.platform()` 等多余抽象。
- **不改动 Electron 渲染层沙箱配置**：需求「非 sandbox」针对 OS 级 App Sandbox，文件访问在主进程完成（IPC/`registerFileProtocol`），渲染层 `contextIsolation:true`+`nodeIntegration:false` 的安全姿态保持不变；不动 `webPreferences`、不动 preload。
- **既有链路不得回退**：HTTP/stdio 双传输、env/设置多别名覆盖、本机绑定守卫、Bearer 鉴权、`stop()` 幂等关闭、事件流、手动启停、Windows/Linux 行为均与现状逐字一致。
