# 需求 #23 开发计划：GitHub Action 自动三端发布与一键推送 Release

## 背景

当前项目是 Electron 桌面应用，`package.json` 已有 `package:win` Windows 打包命令，`electron-builder` 配置也已有 Windows NSIS 目标，但还缺少 macOS、Linux 打包配置、跨平台 CI 发布流水线，以及本地一键创建/推送 Release 的脚本。

本需求目标是：实现一个 GitHub Action，在发布 Release 版本时自动编译并打包 Windows、macOS、Linux 三端程序；同时新增一个 Python 脚本，支持开发者在本地一键指定 tag 和版本说明，推送 tag 并触发 GitHub Release 发布流程。

## 设计约束

- 发布流程应优先复用 `electron-builder`，避免引入与现有 Electron 构建体系冲突的新打包工具。
- 三端产物应在对应系统 runner 上构建，Windows、macOS、Linux 分别使用 `windows-latest`、`macos-latest`、`ubuntu-latest`。
- GitHub Action 应支持手动触发和 tag 触发，便于本地脚本推送 tag 后自动发布。
- Release 产物命名应稳定、可区分平台、版本和架构，便于用户下载和脚本校验。
- Python 脚本应只负责本地发布编排：参数校验、tag 创建/推送、版本说明传递或生成；不应在脚本中重复执行三端构建。
- 脚本应对常见失败场景给出明确提示，包括未安装 Git、非 Git 仓库、工作区未清理、tag 已存在、远端推送失败、缺少 GitHub CLI 或 token 等。
- Release 说明应支持命令行直接传入文本，也支持从文件读取，避免长说明在 shell 中转义困难。
- 不在本计划中实现代码签名、公证、自动更新服务器；如需后续支持，应预留配置位置但不阻塞基础发布能力。

## 任务列表

- [x] P001: 补齐三端打包配置 <!-- scope: package.json -->
  - 验收要点：`package.json` 中新增或调整 Windows、macOS、Linux 打包脚本，能够分别调用 `electron-builder` 的对应平台目标。
  - 验收要点：`build` 配置包含 macOS 与 Linux 的基础 target、artifactName、输出目录等信息，且不破坏现有 Windows NSIS 配置。
  - 验收要点：产物命名包含产品名、版本、平台或架构信息，三端产物不会互相覆盖。
  - 验收要点：新增脚本命名清晰，例如 `package:mac`、`package:linux`、`package:all` 或等价命令，便于 GitHub Action 调用。

- [x] P002: 新增 Release 工作流 <!-- scope: .github/workflows/release.yml -->
  - 验收要点：新增 GitHub Action workflow，支持 `push` tag 触发，tag 规则覆盖发布版本格式，例如 `v*`。
  - 验收要点：workflow 支持 `workflow_dispatch` 手动触发，并允许输入 tag、版本说明或说明文件来源的必要信息。
  - 验收要点：workflow 使用 matrix 或等价 jobs 在 Windows、macOS、Linux runner 上分别安装依赖、执行构建和打包。
  - 验收要点：workflow 上传每个平台生成的安装包或压缩包为 artifact，并在发布 job 中统一附加到 GitHub Release。
  - 验收要点：workflow 使用仓库内置 `GITHUB_TOKEN` 完成 Release 创建或更新，不要求用户手动配置额外 token 才能完成基础发布。

- [x] P003: 实现本地发布脚本 <!-- scope: scripts/release.py -->
  - 验收要点：新增 Python 脚本支持指定 `--tag`，并校验 tag 格式、是否已在本地或远端存在。
  - 验收要点：脚本支持指定版本说明，至少包含 `--notes` 直接传入和 `--notes-file` 从文件读取两种方式。
  - 验收要点：脚本在推送前检查当前目录是 Git 仓库、远端存在、工作区状态符合发布要求，并输出可操作的错误信息。
  - 验收要点：脚本能够创建本地 tag 并推送到远端，成功后明确提示 GitHub Action 将由 tag 推送触发。
  - 验收要点：脚本提供 dry-run 或确认机制，避免误发布不可撤销的 tag。

- [x] P004: 串联版本说明传递 <!-- scope: scripts/release.py,.github/workflows/release.yml -->
  - 验收要点：本地脚本生成或提交的版本说明能够被 Release workflow 获取，作为 GitHub Release 正文使用。
  - 验收要点：当没有显式版本说明时，流程应提供合理兜底，例如使用最近提交摘要或默认模板。
  - 验收要点：版本说明中的换行、中文、Markdown 列表、链接等内容在脚本、Git、workflow 和 Release 页面中保持可读。
  - 验收要点：重复运行同一 tag 的发布流程时，Release 正文与附件处理策略明确，例如更新已有 Release 或失败提示。

- [x] P005: 补充发布使用文档 <!-- scope: README.md,docs/release.md -->
  - 验收要点：文档说明 GitHub Action 的触发方式、tag 命名规范、三端产物位置和发布权限要求。
  - 验收要点：文档给出 Python 脚本的典型用法，包括指定 tag、直接传入版本说明、从文件读取版本说明、dry-run 示例。
  - 验收要点：文档列出发布前检查项，包括依赖安装、版本号更新、工作区干净、远端权限、GitHub Actions 权限设置。
  - 验收要点：文档说明当前不包含代码签名、公证和自动更新能力，避免使用者误解发布产物能力边界。

- [x] P006: 完整验收 <!-- scope: validation -->
  - 验收要点：最终验收命令包含 `npm run check`，确认现有静态检查、主进程脚本检查和 smoke 脚本语法检查通过。
  - 验收要点：最终验收命令包含三端打包命令的可执行性检查；在当前本地平台至少实际运行本平台打包命令，其他平台通过 GitHub Action 配置和脚本命令审查确认。
  - 验收要点：最终验收命令包含 `python scripts/release.py --help` 和一次 dry-run 示例，确认参数解析、帮助信息、错误提示和非破坏性流程可用。
  - 验收要点：在 GitHub Actions 页面手动触发或推送测试 tag 后，Windows、macOS、Linux 三个平台 job 均成功，Release 页面生成并附带三端产物。
  - 验收要点：总体验收通过标准为检查命令无失败、dry-run 不产生远端 tag、实际发布流程可生成目标 tag 的 GitHub Release，且 Release 正文与附件符合预期。

## 总体验收标准

- GitHub Action 能在推送符合规则的 tag 后自动执行发布流程，并构建 Windows、macOS、Linux 三端安装包或压缩包。
- Release 页面能自动创建或更新，正文使用用户指定的版本说明，附件包含三端产物且命名清晰不冲突。
- `scripts/release.py` 能在本地一键发起发布，支持 `--tag`、`--notes`、`--notes-file` 和 dry-run 或确认机制，并对异常状态给出明确错误提示。
- 文档覆盖发布前准备、脚本用法、Action 触发方式、产物说明、权限要求和能力边界。
- 最终验收命令：`npm run check`、本平台打包命令、`python scripts/release.py --help`、`python scripts/release.py --tag v0.0.0-test --notes "test release" --dry-run`，以及 GitHub Actions 中一次测试 tag 发布验证。
- 通过标准：所有本地命令退出码为 0；dry-run 不创建本地或远端 tag；GitHub Actions 三端 job 全部成功；GitHub Release 正文、tag、附件与输入参数一致。
