# 反馈 #1 开发计划：验收模块批量确认（人工批量验收 / 批量取消验收）

## 背景

AutoPlan 工作区「验收模块」(`activeTab === 'acceptance'`，`WorkspacePage.tsx:347`) 把**已完成且未验收**的计划/任务按计划分组渲染为两级 checklist（`AcceptanceView.tsx`），逐项勾选即验收、取消勾选即回退——交互走 `onAccept/onUnaccept` → `useWorkspaceController` 的 `acceptItem/unacceptItem`(`useWorkspaceController.ts:395/397`) → `window.autoplan.acceptItem/unacceptItem`(`preload.js:68/69`) → `acceptance:accept/unaccept` IPC(`main.js:197/203`) → `loopService.acceptItem/unacceptItem`(`loopService.js:650/683`)，落库 `plans.accepted_at`/`plan_tasks.accepted_at`（`acceptanceTargetRow`，`loopService.js:716`），记 `plan.accepted`/`task.accepted`/`plan.unaccepted`/`task.unaccepted` 事件并 `emitUpdate` 刷新快照。**验收态与执行态 `status` 正交**：`accepted_at` 仅是「人工是否确认」的时间戳，不改变 `status`（`acceptance.test.js` 已覆盖）。

反馈 #1 要求：**验收模块的功能是人工确认，应该要支持批量确认；不需要执行验收脚本、其它任务。** 对照现状，差距有两点：

1. **批量确认不完整、非原子**。当前「批量」全部在渲染层用 `forEach` 逐项调 `onAccept`：`acceptAllPending`(`AcceptanceView.tsx:82`) 遍历当页全部 group 的 plan+tasks、`acceptPlanGroup`(`:89`) 遍历单个计划的 tasks——每条都是一次独立 IPC（N 次往返、N 次 `emitUpdate`、N 条事件），中途可见半验收态、且任一项失败会留下不一致结果。后端**没有**批量入口：`acceptItem/unacceptItem` 都是单目标。同时**不支持跨项多选**（只能「全部」或「整计划」，无法勾选任意子集一并验收），「已验收（最近）」区(`AcceptedSection`，`:296`)**没有批量取消验收**（只有逐项 `取消验收` 按钮，`:359`）。
2. **「不执行验收脚本」是约束而非新功能，但需固化**。当前 `acceptItem/unacceptItem` 确实只写 `accepted_at`，**不**调用 `validatePlan`/`runShell`/不执行 `validation_command`——执行验收脚本的是另一条链路（计划最后一个 `完整验收` 任务 → `runTask` 命中 `isFinalAcceptanceTask`(`loopService.js:399`) → `validatePlan`(`loopService.js:967` → `loop/validation.js:66`) 才跑 `validation_command`，`validation.js:99/145`）。本需求要把「验收模块的人工确认绝不触发脚本/任务执行」固化为代码结构与回归断言，避免后续误耦合。

本需求在不回退「逐项验收」「验收态/执行态正交」「事件流」「snapshot 回灌」等既有能力的前提下，新增**人工批量验收 / 批量取消验收**：

- **后端批量入口（原子）**：`loopService` 新增 `acceptItems/unacceptItems`，接收 `{ targetType, id }[]`，**先全量校验再一次性写入、单次 `emitUpdate`**，复用既有 `acceptanceTargetRow` 与 UPDATE/事件语句；只置/清 `accepted_at`，绝不调用 `validatePlan`/`runShell`/执行 `validation_command`。
- **IPC/preload/类型**：新增 `acceptance:acceptBatch`/`acceptance:unacceptBatch` 通道与 `acceptItems`/`unacceptItems` preload 方法，类型补 `AcceptBatchInput`/`AutoplanApi.acceptItems`/`unacceptItems`（沿用 `AcceptanceItemInput`）。
- **渲染层多选批量**：`AcceptanceView` 给每个待验收 plan/task 行与已验收行加**选择框**，选中后顶部出现**批量操作条**（批量验收 / 批量取消验收 / 全选 / 取消选择），既有「全部验收」「全部验收本计划」改为走批量入口（单次原子调用）；控制器新增 `acceptItems`/`unacceptItems` 动作。
- **设计与测试**：`ui/acceptance.html` 同步多选与批量条；`acceptance.test.js` 补批量用例（原子性、混合 plan+task、幂等、非法目标拒绝、事件、**不执行脚本**断言），`smoke-test.js` 补源码级断言。

## 设计约束

- **批量入口是「先校验后写」的原子操作，只置/清 `accepted_at`，绝不执行脚本或任务**：`acceptItems(projectId, targets)`/`unacceptItems(projectId, targets)` 中 `targets` 为 `{ targetType: 'plan'|'task', id: number }[]`——去重、规范化后**先逐条 `acceptanceTargetRow` 校验**（验收用 `requireCompleted:true`、取消验收用 `requireCompleted:false`，沿用 `:716` 既有语义与中文报错），**任一目标非法即整体抛中文错误、不写任何行**（全有或全无，避免批量中出现半验收/半取消的不一致）；全部通过后用同一个 `acceptedAt`/`updatedAt` 时间戳逐条 `UPDATE` + `addEvent`，**最后一次 `emitUpdate(projectId)`**。返回 `{ accepted|unaccepted: N, items: [{ targetType, id, accepted_at }] }`。**严禁**在批量方法内（或经其触发的调用链）调用 `validatePlan`/`runShell`/`runCodex`/执行 `validation_command` 或启动任何任务/脚本——验收模块是纯人工确认，与 `完整验收` 任务的脚本执行链路（`runTask`→`validatePlan`）完全解耦，后者行为不变。
- **复用既有单目标写法，最小化重复**：把单目标 `acceptItem`/`unacceptItem`(`:650/683`) 内「UPDATE accepted_at + addEvent」抽为同文件私有 helper（如 `writeAcceptance(targetType, row, acceptedAt, projectId)`，`acceptedAt=null` 即取消），单目标与批量方法共同调用；`acceptItem`/`unacceptItem` 的对外签名、返回值（单对象）、中文报错与事件 meta 结构**保持不变**（既有 `acceptance.test.js` 不得回归），仅内部实现改为走 helper。`acceptanceTargetRow` 不改逻辑，批量复用。
- **入参校验与单目标一致**：批量目标列表为空或非数组时抛中文错误（如「批量验收目标列表为空」/「验收目标列表无效」）；`targetType` 非 `plan`/`task` 或 `id<=0` 复用 `acceptanceTargetRow` 的「验收目标类型无效」/「验收目标 ID 无效」；目标不存在或不可验收复用「计划不存在」/「任务不存在」/「仅可验收已完成的计划/任务」。批量幂等：列表内含已验收（验收）/未验收（取消验收）项不报错，按单目标既有幂等语义处理（验收刷新时间、取消保持 NULL）。
- **IPC/preload 沿用既有通道风格，类型最小增量**：`acceptance:acceptBatch`/`acceptance:unacceptBatch`(`main.js` 在 `:197/203` 单目标 handler 旁) 用 `requiredProjectId(input)` 取项目、自行校验 `Array.isArray(input.targets)` 后透传给 `loop.acceptItems/unacceptItems`，返回 `loop.snapshot(projectId)`（与单目标 handler 返回一致）；`preload.js` 在 `:68/69` 旁加 `acceptItems`/`unacceptItems`。`types.ts` 新增 `AcceptBatchInput extends ProjectIdInput { targets: AcceptanceItemInput[] }`（复用 `:915` 的 `AcceptanceItemInput`，即 `{ projectId, targetType, id }`），`AutoplanApi`(`:946`) 加 `acceptItems: (input: AcceptBatchInput) => Promise<AppSnapshot>`、`unacceptItems` 同；既有 `AcceptanceItemInput`/`acceptItem`/`unacceptItem` 不破坏。preload 无需新增白名单片段（沿用 `acceptItem` 写法）。
- **渲染层多选为增量、不回退逐项验收语义**：`AcceptanceView`(`AcceptanceView.tsx`) 新增**选择态** `selection: Set<string>`（key 形如 `plan:${id}`/`task:${id}`，与 `recordId`/事件 meta 的 targetType+id 对齐），每个 `PendingPlanCard` 计划头、`PendingTaskRow` 任务行、`AcceptedRow` 已验收行左侧加一个**选择 checkbox**（`type="button" role="checkbox"`，复用 `AcceptanceCheck`(`:174`) 的样式与 `check` 图标）；选中项数 ≥1 时顶部 `.accept-bar` 渲染**批量操作条**（「已选 N 项」+「批量验收」/「批量取消验收」（按当前所在区是待验收还是已验收决定按钮）+「全选当页」+「取消选择」）。既有逐项 `AcceptanceCheck`（点击立即验收单项）保留、不受影响；既有「全部验收」(`:122`)、「全部验收本计划」(`:234`)、逐项「取消验收」(`:359`) **改为经批量入口**（`全部验收` = `acceptItems(当页全部 plan+task)`、`全部验收本计划` = `acceptItems(本计划 plan+tasks)`、单条与多选的取消 = `unacceptItems([...])`），单次原子调用替代 `forEach` 逐项。`props` 新增 `onAcceptItems`/`onUnacceptItems`（接收 `{ targetType, id }[]`），`onAccept`/`onUnaccept` 单目标保留（逐项立即验收仍走单目标通道，行为不变）。
- **控制器与页面接线**：`useWorkspaceController.ts`(`:395/397` 旁) 新增 `acceptItems(targets: { targetType; id }[])`/`unacceptItems(targets)` 动作，`runLoopAction(() => window.autoplan.acceptItems({ projectId, targets: targets.map(t => ({ projectId, targetType: t.targetType, id: t.id })) }))`（payload 满足 `AcceptBatchInput`）；`WorkspacePage.tsx`(`:349` `<AcceptanceView>`) 透传 `onAcceptItems`/`onUnacceptItems`。批量操作后经既有 snapshot 回灌刷新（`onAcceptItems` 返回新 snapshot → `runLoopAction` 已有回灌链），选择态在批量成功后清空。
- **行数与样式约束**：`npm run check`（含 `scripts/check-file-lengths.js`）默认单文件上限 600 行。`AcceptanceView.tsx` 现约 388 行——新增选择 checkbox + 批量操作条若逼近上限，须把「批量操作条」抽为同文件内嵌子组件（参考既有 `PendingPlanCard`/`AcceptedSection` 的内嵌写法），不得外溢超限。`src/renderer/styles/workspace.css` 追加选择 checkbox 与批量操作条样式，复用既有 token（`--brand-*`/`--bg-*`/`--text-*`/`--r-*`），不重复声明既有变量。`planTasks.ts` 若需选择 key 工具（`acceptanceSelectionKey(targetType, id)`），新增小函数（现约 338 行，注意不超限）。
- **设计稿同步**：`ui/acceptance.html`(690 行) 已镜像视图，本需求对多选 checkbox、批量操作条、批量验收/取消验收按钮的 UI 改动须同步回该设计稿，使其可在浏览器直接预览且与实现视觉一致（沿用既有 `--brand-*`/`--bg-*`/`--text-*`/`--r-*`/`--shadow-*` token 与 `icon-sprite`，复用 `:363` 的 `icon-check-double`）。
- **不回退既有能力**：循环轮询/`runOnce`/任务执行（run/stop/并发批次）、`完整验收` 任务经 `validatePlan` 执行 `validation_command` 的链路（`runTask:399`→`validatePlan`）、逐项 `acceptItem`/`unacceptItem` 及其事件流、验收态/执行态 `status` 正交、需求/反馈 intake、事件流、CLI 后端（codex/claude/opencode/oh-my-pi）、MCP 工具链路、脚本模块、工作区导航与全局搜索、设计稿均不得回退。
- **不要把运行测试/回归/构建拆成普通开发任务**；最终检查统一放入最后的「完整验收」节点。
