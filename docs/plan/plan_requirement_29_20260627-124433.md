# 需求 #29 开发计划：提供外部 MCP 接入能力

## 背景

当前 AutoPlan 的项目创建、需求提交和反馈提交主要通过 Electron 渲染进程调用主进程 IPC 完成。新需求要求“增加 MCP 功能，给外部提供 MCP 接入，方便其它客户端创建项目、提需求、提反馈”，因此需要在现有桌面应用能力之外，新增一个可被 MCP 客户端调用的本地服务入口，并复用现有数据层、附件处理、CLI 配置规范化和计划生成循环能力。

## 设计约束

- MCP 接入必须复用现有项目、需求、反馈的核心写入逻辑，避免 Electron IPC 与 MCP 两套实现出现字段、默认值和事件行为不一致。
- MCP 服务默认只面向本机客户端使用，不应默认暴露公网；如支持配置监听地址，必须默认绑定 `127.0.0.1` 或 stdio 模式。
- 外部客户端创建项目、提交需求、提交反馈后，应返回可用于后续调用的稳定标识，例如 `projectId`、`requirementId`、`feedbackId` 和最新快照摘要。
- MCP 工具入参需要做严格校验，缺少项目、空内容、无效反馈关联需求、非法 CLI 后端或思考深度时要给出明确错误或按现有规则规范化。
- 附件能力如在首版无法完整支持文件上传，也必须在接口契约中明确支持范围与错误提示，避免外部客户端误以为静默成功。
- MCP 服务生命周期应与应用主进程清晰解耦：可启动、可停止、启动失败可见，不影响 GUI 正常启动与既有 IPC 行为。
- 最终验收节点只负责整体验收，不参与普通开发任务并发或功能实现。

## 总体验收标准

- 功能完整性：外部 MCP 客户端可连接 AutoPlan，并通过 MCP 工具完成创建项目、提交需求、提交反馈三类核心操作。
- 数据一致性：MCP 路径与 GUI IPC 路径复用同一套业务服务，创建后的数据库记录、快照、事件日志、CLI 配置和 loop 行为保持一致。
- 安全性：MCP 服务默认仅本机可访问；无效入参、跨项目关联和不可支持的附件场景有明确错误，不出现静默失败或误写数据。
- 可观测性：用户能在应用内或文档中看到 MCP 服务状态、连接方式、工具列表和示例请求，启动失败或禁用状态可诊断。
- 兼容性：现有项目管理、需求管理、反馈管理、计划生成、任务执行和 GUI 交互不因新增 MCP 服务退化。
- 最终验收命令：执行 `npm run check` 和 `npm run smoke`；两条命令均退出码为 0，并完成手动 MCP 客户端连接、工具调用、GUI 同步显示和重启持久化验证。
