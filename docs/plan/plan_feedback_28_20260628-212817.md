# 反馈 #28 开发计划：Claude 模式执行日志实时展示

## 背景

反馈 #28：「claude模式，没有显示执行日志」——选择 Claude 作为 Agent CLI 时，任务执行过程中概览页的日志面板看不到执行过程，长时间停留在「等待 Claude 输出…」。逐项定位如下：

1. **现象与数据流**
   - 日志面板由 `CodexLog` 渲染（`src/renderer/components/workspace/WorkspaceOverviewView.tsx:114`，props：`log=operation.logTail`、`activity=operation.activity`、`provider=operationProvider`）。
   - `CodexLog`（`src/renderer/components/CodexLog.tsx:43-47`）的渲染策略是「`activity` 非空→显示活动时间线；否则回退到 `log` 尾部；两者皆空→显示 `等待 ${providerLabel} 输出…`」。
   - `operation` 来自快照 `activeOperation`，其 `logTail` 与 `activity` 由 `operationSnapshotRow`（`src/loop/runtime.js:283-308`）从运行态 `operation.logBuffer` 与 `operation.activity` 暴露；而 `logBuffer`/`activity` 由 `runAgentCliAttempt` 的 `handleChunk`（`src/agentCli.js:287-296`）在每个 stdout/stderr chunk 上累积并 `offer` 给活动打印机。

2. **根因一：Claude 没有活动打印机**
   - `src/loopService.js:958` 构建 `activeOperation` 时：`activity: isCodexProvider ? new CodexActivityPrinter(200) : null`——只有 Codex 创建活动打印机，Claude/OpenCode 的 `activity` 恒为 `null`，因此 Claude 永远拿不到结构化活动行。

3. **根因二：Claude 用 `text` 输出格式，执行期间 stdout 静默**
   - `claudeCliArgs`（`src/agentCli.js:66-74`）使用 `--print --output-format text`。Claude Code 的 `text` 输出只在执行结束后一次性把最终回复写到 stdout，执行期间 stdout/stderr 基本不产生数据——`handleChunk` 收不到 chunk，`activeOperation.logBuffer` 一直为空，`operationSnapshotRow.logTail`（`runtime.js:303`）为空，面板只能显示「等待 Claude 输出…」。
   - 对比：Codex 走 `CodexActivityPrinter` 把 stdout 流实时解析成「思考/执行/完成」等活动行；OpenCode 走 stdout 也至少有原始尾日志。Claude 既无活动行、又无实时原始日志，故表现为「没有显示执行日志」。

4. **次要问题：Claude 新会话拿不到 session_id，跨任务 resume 失效**
   - `runAgentCliAttempt` 返回的 Claude `agentCliSessionId` 来自 `spawnSpec.agentCliSessionId`（`agentCli.js:367`、`:384-386`），而该值仅由入参决定（`claudeSessionLaunchSpec`，`agentCli.js:102-162`）；新建会话时入参为空，故新会话执行后不记录 session_id，同 plan 后续任务无法 resume（`previousPlanAgentCliSessionId`，`src/loop/taskExecution.js:70-85` 查不到值）。

5. **修复方向**
   - 将 Claude CLI 由 `--output-format text` 切换为 `--output-format stream-json --verbose`：Claude Code 在 `-p` 下以 NDJSON（每行一个 JSON 对象）实时流式输出 `system`/`assistant`/`user(tool_result)`/`result` 等事件，可被增量解析。
   - 新增 `ClaudeStreamJsonPrinter`（API 仿 `CodexActivityPrinter` 的 `offer/flush/getLines`，外加 `getResultText()`/`getSessionId()`），把 NDJSON 事件解析成活动行（助手文本→活动行；`tool_use`→执行行；`tool_result`→结果行；`result`→完成行+最终文本+session_id），并在 `loopService.js:958` 为 Claude 接入该打印机。
   - Claude 的 `lastFile`（last message）改由解析得到的最终助手文本写入，替代原先「整段 stdout 当 last message」（`agentCli.js:347-365` 的 `lastFileSource==='stdout'` 分支）。
   - 顺带从 `result` 事件捕获 `session_id`，使 Claude 新会话也能记录 id、后续任务可 resume。

范围界定：

- 仅改 Claude 模式的执行日志流式展示、last message 解析与会话 id 捕获；不改动计划生成、任务执行/验收循环、并发批次、会话恢复判定逻辑、脚本钩子、MCP/循环配置与持久化行为。
- Codex/OpenCode 路径完全不动；UI 组件 `CodexLog` 已能按 `activity`/`logTail` 通用渲染，无需修改。
- 文件行数约束（`scripts/check-file-lengths.js`，默认上限 600）：`agentCli.js`（598，几乎已满）、`loopService.js`（1570，上限 1600）、`agentCli.test.js`（175）余量紧张，改动须控制在限额内；新增 `claudeActivity.js`、`claudeActivity.test.js`。`agentCli.js` 余量极小，超出时按既有先例（如 `loopService.js:1600`、`agentCliConfig.js:700`）在 `fileLimits` 中单独放宽上限并注明原因，首选把解析逻辑收进 `claudeActivity.js` 以控制行数。

- [x] P001: 新增 Claude stream-json 活动解析器 <!-- scope: src/claudeActivity.js -->
  - 验收要点：新增 `ClaudeStreamJsonPrinter` 类（导出于 `src/claudeActivity.js`），对外 API 与 `CodexActivityPrinter`（`src/codexActivity.js`）对齐：`offer(rawChunk)`、`flush()`、`getLines()` 返回 `{ role, text, at }[]`（上限 200 行、返回最近约 120 行），并额外提供 `getResultText()` 与 `getSessionId()`。
  - 验收要点：`offer` 按 `\n` 拆行、逐行 `JSON.parse`；非 JSON 行（stderr 噪声/分片/进度行）静默跳过不抛错；跨 chunk 的不完整行用 `pendingLine` 缓冲，`flush()` 处理尾部残行（仿 `codexActivity.js:43-59`）。
  - 验收要点：事件映射——`assistant` 事件遍历 `message.content[]`：`type:'text'` → role `codex` 的活动行（与 `CodexLog.activityRoleConfig` 一致，UI 显示为 provider 标签）；`type:'tool_use'` → role `exec` 行，文案含工具名与截断入参；`user` 事件的 `tool_result` → role `info` 行，截断内容并尽量体现成功/失败；`result` 事件 → 记录 `result` 文本与 `session_id`，并 push 一条完成活动行。
  - 验收要点：`getResultText()` 优先返回 `result` 事件的最终文本，回退为累计的助手文本，再回退空串；`getSessionId()` 返回 `result` 事件的 `session_id`（沿用 `normalizeAgentCliSessionId` 风格做长度/字符校验，非法或缺失时返回空串）。
  - 验收要点：活动行做截断（超长截断至约 200 字符并加省略号）与邻近去重，避免刷屏；`at` 用 `new Date().toISOString()`；不引入新的运行时依赖；`claudeActivity.js` 行数须在 600 以内。

## 总体验收标准

- 功能范围：仅修复反馈 #28 的「Claude 模式不显示执行日志」——将 Claude 切到 stream-json 流式输出、新增并接入 `ClaudeStreamJsonPrinter`、解析 last message 与 session_id；不改动计划生成/执行/验收循环、并发批次、会话恢复判定、脚本钩子、MCP/循环配置等无关逻辑，也不改动 Codex/OpenCode 路径与 `CodexLog` UI 组件。
- 兼容范围：Codex/OpenCode 的活动解析、原始尾日志、会话 resume、并发执行均不受影响；Claude 的 `--resume`/`--session-id`/`--continue` 会话参数与回退新建逻辑保持不变；任务执行结果（exitCode、错误文案、超时处理）不受影响。
- 代码约束：改动不得触发 `scripts/check-file-lengths.js` 报错；`agentCli.js`（598/600）余量极小，超出时按既有先例单独放宽上限并注明原因（首选把解析逻辑收进 `claudeActivity.js`），`loopService.js`（1570/1600）、`agentCli.test.js`（175）须重点核对。
- 依赖约束：不引入新的运行时依赖（stream-json 解析用纯 Node 实现，仿 `CodexActivityPrinter`）。
- 最终验收命令：`npm run build` 与 `npm run check`，并按仓库既有方式运行 `agentCli.test.js` 与 `claudeActivity.test.js`。
- 通过标准：两个构建/检查命令均返回 0；相关测试断言全部通过；四项人工验证（Claude 实时日志、last message 正文、会话 resume、Codex/OpenCode 回归）全部满足。
