# 反馈 #27 开发计划：计划倒序、预览 Plan、文件夹链接、脚本导航稳定性与事件流增量加载

## 背景

反馈 #27 共提出五项问题，分布在「计划列表」「需求界面」「左侧面板」「事件流」四个区域。逐项定位如下：

1. **计划列表没有按创建日期倒序排序**
   - 现状：`src/renderer/components/PlanLists.tsx:336` 的 `orderedPlans = [...plans].sort(comparePlanOrder)`，而 `comparePlanOrder`（`PlanLists.tsx:451-457`）的主排序键是 `sort_order` 升序、其次 `created_at` 升序、最后 `id` 升序——整体表现为「旧计划置顶」，与「最新在前」的预期相反。
   - 目标：以创建时间 `created_at` 倒序（最新在前）为主键，`id` 倒序兜底，保证新建/新生成的 Plan 出现在列表最上方。

2. **需求界面「预览 Plan」无法预览**
   - 根因：`src/renderer/pages/WorkspacePage.tsx:197-200` 的 `intakePlanPreviewProps` 传入的是 `onOpenPlan: openIntakePlanReader`，但 `IntakePanel` / `PlanBindingCard` 实际读取的 prop 名是 `onPreviewPlan`（`src/renderer/components/IntakePanel.tsx:31`、`:293` 的 `canPreview = hasSnapshot && Boolean(onPreviewPlan)`、`:330` 的 `onPreviewPlan?.(type, linkedPlanId)`）——prop 名不一致导致 `onPreviewPlan` 恒为 `undefined`，「预览 Plan」按钮被 `disabled={!canPreview}` 永久禁用。
   - 次要问题：即便对齐 prop 名，调用签名也对不上——`PlanBindingCard` 以 `(type, linkedPlanId)` 触发，而控制器 `openIntakePlanReader(item, fallbackPlan?)`（`src/renderer/hooks/useWorkspaceController.ts:470-495`）期望 intake item。修复须同时统一 prop 名与调用签名，使点击真正进入「读取 Plan 全文 → 打开 `PlanReaderModal`」流程。

3. **左侧面板文件夹路径要链接形式显示**
   - 现状：`src/renderer/components/workspace/WorkspaceSidebar.tsx:110-123` 把工作区路径渲染成 `<button>`，并通过内联 `pathLinkStyle`（`:34-41`）把按钮外观重置成无边框/透明/无内边距，复用 `.project-path`（`workspace.css:12`）的字体与省略号——打开系统文件夹的行为已实现，但视觉上与普通文本无异，不像「链接」。
   - 目标：仅调整外观为可识别的链接（主题色文字、指针光标、悬停下划线），点击仍走既有 `window.autoplan.openProjectFolder({ projectId })`（`:73`）打开系统文件夹，不改变打开逻辑。

4. **点击脚本面板有时候点不动、会跳到需求界面**
   - 现状：导航按钮 `onClick={() => onTab(tab.id)}`（`WorkspaceSidebar.tsx:129-145`）→ 控制器 `selectTab`（`useWorkspaceController.ts:530-536`，`setActiveTab(tab)` + `setSearchParams({tab}, {replace:true})`）；另有 URL→activeTab 同步副作用（`useWorkspaceController.ts:177-180`）按 `tabParam` 经 `resolveWorkspaceTab` 回填 activeTab，而 `DEFAULT_WORKSPACE_TAB = 'requirement'`（`types.ts:3`）。
   - 疑似根因：当点击「脚本」后 URL 同步存在时序窗口，同步副作用可能把瞬时为空/失效的 `tab` 参数解析回默认值 `requirement`，覆盖用户刚选的 `scripts`，表现为「点不动 / 跳回需求」。需复现确认后修复，使点击稳定生效且不被回退。
   - 约束：`useWorkspaceController.ts` 当前 599 行（已达 600 行上限），修复需控制行数。

5. **事件流列表是否支持动态加载**
   - 现状：`EventList`（`src/renderer/components/PlanLists.tsx:269-294`）一次性 `events.map` 渲染全部事件为 DOM 节点，无窗口化/分批；事件来源虽在快照层有上限（页面提示「最近 80 条」，`WorkspacePage.tsx:431`），但前端不做任何裁剪或增量加载，事件增多后 DOM 规模与渲染压力随事件数线性增长。
   - 目标：为 `EventList` 增加增量渲染（默认只渲染最近一批，提供「加载更多」分批展开更早事件），不引入新的运行时依赖，使列表在事件较多时仍保持流畅。

范围界定：

- 仅修复上述五项前端展示/交互问题；不改动计划生成、任务执行、验收等后端循环流程，不改动脚本编辑器校验（`validateScriptDraft`）、MCP/循环配置与持久化行为。
- 不调整 Plan 卡片其余字段、intake 编辑/删除/中断/恢复/追加任务流程、项目切换与搜索跳转等既有交互。
- 文件行数约束（`scripts/check-file-lengths.js`，默认上限 600）：`useWorkspaceController.ts`（599，已满）、`PlanLists.tsx`（497）、`WorkspacePage.tsx`（539）、`workspace.css`（523）余量紧张，改动须控制在限额内；确实超出时按既有先例在 `fileLimits` 中为该文件单独放宽上限并注明原因，但首选通过精简控制行数。
