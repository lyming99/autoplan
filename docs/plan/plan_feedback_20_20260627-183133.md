# 反馈 #20 开发计划：发布 beta 版本并补充说明

## 背景

近期 AutoPlan 已完成多轮功能更新，需要通过 `scripts/release.py` 发布一个 beta 版本，利用现有 GitHub Actions 发布流程生成安装包。该 beta 版本定位为快速分发版本：不做完整人工测试，AI 完成问题修复后，只要编译通过即可发布，以缩短测试与发布周期。

本次计划重点不是新增业务能力，而是规范 beta 版本发布文案、Release notes 与发布操作，确保下载用户明确知道：beta 是未充分测试版本，如下载测试或使用中发现问题，可直接在 GitHub Issue 界面反馈；后续软件/AI 流程会从 Issue 中发现问题并继续修复。

## 任务列表

- [x] P001: 确认 beta 版本号与发布参数 <!-- scope: package.json,scripts/release.py -->
  - 验收要点：确认目标 beta 版本号采用可识别的预发布格式，例如 `0.2.1-beta.1`，Git tag 对应为 `v0.2.1-beta.1`。
  - 验收要点：`package.json` 中 `version` 与计划发布的 tag 保持一致，避免 Release 产物版本号和 Git tag 不一致。
  - 验收要点：确认 `scripts/release.py` 已支持通过 `--tag` 和 `--notes-file` 或 `--notes` 传入 beta 发布说明并推送 tag。
  - 验收要点：如发布脚本已有能力，不为 beta 发布新增重复脚本；仅在参数、提示或文档存在缺口时补充最小变更。

- [x] P002: 补充 README beta 版本说明 <!-- scope: README.md -->
  - 验收要点：README 新增或更新 beta 版本说明，明确 beta 是未完整测试版本，不等同于稳定正式版。
  - 验收要点：README 说明 beta 发布策略为 AI 解决问题后编译通过即可发布，用于减少人工测试和等待时间。
  - 验收要点：README 明确下载测试或使用中发现问题时，用户可直接到 GitHub Issue 界面提交反馈。
  - 验收要点：README 说明后续软件/AI 流程会持续从 Issue 中发现问题、生成修复计划并处理，不承诺 beta 版本零缺陷。

- [x] P003: 准备 beta Release notes <!-- scope: docs/release-notes,docs/release.md -->
  - 验收要点：新增本次 beta 版本 Release notes 文件，文件名与 tag 对齐，例如 `docs/release-notes/v0.2.1-beta.1.md`。
  - 验收要点：Release notes 开头明确标注“Beta 未测试版本”或等价说明，避免用户误认为是稳定版本。
  - 验收要点：Release notes 包含近期更新摘要、beta 风险提示、Issue 反馈入口说明和“编译通过即发布”的发布策略说明。
  - 验收要点：`docs/release.md` 的发布说明补充 beta 发布流程，推荐使用 `--notes-file` 传入 beta Release notes。

- [x] P004: 使用 release 脚本推送 beta tag <!-- scope: scripts/release.py,git tags,GitHub Release -->
  - 验收要点：在工作区干净、版本号与 Release notes 准备完成后，通过 `python scripts/release.py --tag <beta-tag> --notes-file docs/release-notes/<beta-tag>.md --yes` 推送 beta tag。
  - 验收要点：脚本执行前不绕过既有发布前置检查，包括 Git 仓库检查、工作区状态检查、tag 格式检查、本地和远端同名 tag 检查。
  - 验收要点：推送后 GitHub Actions Release workflow 被 tag 触发，且 Release 正文使用 beta Release notes 内容。
  - 验收要点：若脚本或远端推送失败，保留错误信息并停止发布，不手工创建不一致的 tag 或 Release。

- [x] P005: 完整验收 <!-- scope: validation -->
  - 验收要点：最终验收命令为 `npm run build`，通过标准为 TypeScript 与 Vite 构建均成功，无阻断发布的编译错误。
  - 验收要点：确认 README 与 Release notes 均包含 beta 未测试、AI 修复、编译通过即发布、Issue 反馈和后续自动发现修复问题的描述。
  - 验收要点：确认 `package.json` 版本号、Git tag、Release notes 文件名和 GitHub Release 标题/正文中的 beta 版本一致。
  - 验收要点：确认 GitHub Actions 三端构建任务已启动；若任一平台构建失败，则该 beta 发布不得标记为可下载完成状态。
  - 验收要点：确认 GitHub Release 附件可见后，在 Release 正文保留 beta 风险提示与 Issue 反馈入口。

## 总体验收标准

- 文档范围：`README.md` 必须面向普通下载用户说明 beta 版本风险、发布节奏和 Issue 反馈方式；`docs/release.md` 必须面向维护者说明 beta 发布流程；本次 `docs/release-notes/<beta-tag>.md` 必须可直接作为 GitHub Release 正文。
- 发布范围：本次仅发布 beta 版本，不扩大为稳定正式版；版本号、tag、Release notes 和构建产物版本必须一致。
- 发布策略：beta 版本允许不做完整人工测试，但必须完成编译验收；AI 修复问题后，只要 `npm run build` 通过即可使用 release 脚本发布。
- 最终验收命令：`npm run build`。
- 发布命令：`python scripts/release.py --tag <beta-tag> --notes-file docs/release-notes/<beta-tag>.md --yes`，其中 `<beta-tag>` 必须替换为实际 beta tag，例如 `v0.2.1-beta.1`。
- 通过标准：构建命令返回 0；release 脚本成功推送 tag；GitHub Actions Release workflow 被触发；GitHub Release 正文包含 beta 风险提示；下载测试问题可通过 Issue 入口提交。
