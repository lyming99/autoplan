# 反馈 #21 开发计划：需求与反馈输入框内容持久化

## 背景

AutoPlan 工作区的「需求」与「反馈」两个界面各有一个输入框（Composer 文本域），用户在输入需求或反馈时，如果发生以下情况，已输入但未发送的文字会丢失：

- 在「需求」「反馈」「任务与计划」等标签页之间切换：`WorkspacePage.tsx` 用 `{activeTab === 'requirement' ? ... : null}` 条件渲染各面板，切换标签会卸载 Composer，其内部 `body` 状态随之销毁。
- 切换到其他项目后返回：页面组件按 `projectId` 重新挂载，输入框被重置为空。
- 重启或刷新应用：内存中的输入内容全部丢失。

反馈 #21 要求：需求、反馈界面的输入框内容要持久化。本次计划的目标是让这两个输入框的**文本内容**在标签切换、项目切换、应用重启后都能保留，发送成功后再清空。

代码库已有持久化先例：`useWorkspaceController.ts` 通过 `window.localStorage`（键名 `autoplan.scopeFileOpenSettings`）在 `useState` 初始化函数中读取、在 `useEffect` 中写入「作用域文件打开方式」。本次持久化沿用同一套模式，并把草稿按**项目维度**隔离存储，避免项目 A 的草稿文字出现在项目 B 的输入框里。

设计要点与范围界定：

- 持久化对象仅为 Composer 的文本域 `body`（需求/反馈两类各一份）。附件、CLI 后端、思考深度、「创建为草稿」勾选等暂不在持久化范围内，附件本就是一次性的待上传状态。
- 持久化键按项目隔离：`autoplan.composerDrafts.<projectId>` → `{ requirement: string, feedback: string }`。
- 约束：`useWorkspaceController.ts` 当前 595 行，`scripts/check-file-lengths.js` 默认上限 600 行。因此草稿状态与 localStorage 读写逻辑必须抽取到新的 hook 与 `workspaceForms.ts` 纯函数中，不能直接堆进控制器，否则会触发长度检查失败。

- [x] P002: 新增 useComposerDrafts hook 管理按项目持久化的草稿 <!-- scope: src/renderer/hooks/useComposerDrafts.ts -->
  - 验收要点：新建 `src/renderer/hooks/useComposerDrafts.ts`，导出 `useComposerDrafts(projectId)`，与现有 `useSnapshot.ts` 同属 hooks 目录风格。
  - 验收要点：`useState` 初始化函数调用 `loadComposerDrafts(projectId)` 读取当前项目草稿；当存储键为 `null`（projectId 非法）时使用 `emptyComposerDrafts()` 且不读写 localStorage。
  - 验收要点：在 `[projectId]` 变化的 `useEffect` 中重新加载对应项目的草稿，使切换项目后输入框显示该项目的草稿、不串项目。
  - 验收要点：在草稿变化的 `useEffect` 中将最新草稿写入 `window.localStorage`（参照 `autoplan.scopeFileOpenSettings` 的写入写法）；写入前对 `window`/`localStorage` 存在性做保护，避免在异常环境报错。

  - 验收要点：对 `window.localStorage` 的访问全部包裹在 try/catch 中，存储不可用时不影响输入框正常使用（仅退化为内存态）。

## 总体验收标准

- 功能范围：仅持久化需求与反馈两个输入框的**文本内容**；附件、CLI 后端、思考深度、「创建为草稿」勾选不在本次范围内，行为应与改动前一致。
- 持久化范围：文字需在「标签切换」「项目切换」「应用重启」三种场景下保留；切换项目时按项目隔离，互不串扰；发送成功后清空并持久化清空结果。
- 存储约定：使用 `window.localStorage`，键名 `autoplan.composerDrafts.<projectId>`，值为 JSON `{ requirement: string, feedback: string }`；与既有 `autoplan.scopeFileOpenSettings` 同一存储命名空间，遵循同一读写模式；`localStorage` 不可用时优雅退化为内存态，不阻断输入。
- 代码约束：改动不得触发 `scripts/check-file-lengths.js` 报错；草稿状态与读写逻辑集中在新增的 `useComposerDrafts.ts` 与 `workspaceForms.ts` 中，控制器仅作最小接入。
- 最终验收命令：`npm run build` 与 `npm run check`。
- 通过标准：两个命令均返回 0；人工验证四项全部满足；输入框内容按项目持久化、发送成功后清空。
