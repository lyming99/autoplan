# 需求 #39 开发计划：需求/反馈项绑定 Plan 并支持全文预览

## 背景

需求 #39 要求在需求、反馈界面中，每个记录项都能展示其绑定的 Plan 计划，并支持直接打开该计划的全文预览。当前系统已在需求/反馈记录上维护 `linked_plan_id`、计划状态与任务进度，并在任务页的 Plan 列表中具备 Plan 全文阅读能力；本次开发应复用既有读取接口和 Markdown 预览能力，将绑定计划信息和预览入口前移到需求、反馈列表项。

## 设计约束

- 需求与反馈记录只展示已绑定的 Plan；未生成计划的记录保持现有状态展示，不出现误导性预览入口。
- Plan 全文预览必须复用现有 `plans:read` / `window.autoplan.readPlan` 能力，避免新增重复的文件读取通道。
- 预览弹窗应能在需求页、反馈页和任务页统一打开，读取中、读取失败、刷新、关闭、Markdown 渲染等行为保持一致。
- 绑定计划展示应包含可识别信息，例如 Plan 标题、文件路径、状态、任务进度；当绑定 ID 存在但计划快照缺失时，应有降级提示且不能触发错误预览。
- 不改变计划生成、任务执行、中断、恢复、追加任务等既有业务语义；需求/反馈项原有编辑、删除、追加任务按钮仍可正常使用。
- 不要把运行测试、回归、构建拆成普通开发任务；最终验证统一放入最后的“完整验收”节点。

## 总体验收标准

- 数据完整：需求与反馈记录能稳定关联到绑定 Plan 的 ID、标题、文件路径、状态和任务进度，历史或异常数据有安全降级。
- 入口明确：需求页、反馈页中每个已绑定 Plan 的项都有清晰的计划信息与全文预览入口，未绑定项不出现误导操作。
- 预览一致：从需求、反馈、任务页打开的 Plan 全文预览使用同一读取流程和弹窗体验，支持读取中、刷新、失败提示、关闭和 Markdown 正文展示。
- 交互无回退：需求/反馈项原有编辑、删除、中断、恢复、追加任务、附件展示和状态标签功能保持可用。
- 兼容稳定：绑定 Plan 缺失、文件读取失败、长标题/长路径、不同计划状态和任务进度为空等情况均可读且不崩溃。
- 最终验收命令：执行 `npm run check && npm run smoke`；验收范围为快照数据、类型定义、需求/反馈列表展示、Plan 阅读弹窗复用、读取失败降级和现有 smoke 主流程；通过标准为命令退出码为 0，且手动验证需求 #39 的核心路径全部通过。
