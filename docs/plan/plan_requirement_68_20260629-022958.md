# 需求 #68 开发计划：增加 oh-my-pi CLI 支持

## 背景

需求原话：「增加 oh-my-pi cli支持」。oh-my-pi（命令 `omp`，作者 can1357，仓库 github.com/can1357/oh-my-pi、站点 omp.sh）是一个面向终端的 AI Coding Agent CLI，与 AutoPlan 既有接入的 codex / claude / opencode 同类。本需求要求把 **oh-my-pi 作为第 4 个 Agent CLI 后端**接入 AutoPlan，使其能用于「生成计划」与「执行任务」两条链路，并在 UI（设置、Composer、计划/任务展示）与 MCP 工具中作为可选后端出现。

现状梳理（结合代码）：

1. **Agent CLI 后端是一个可扩展的 provider 体系，目前只内置 codex / claude / opencode 三种**。provider 集合定义在 `src/agentCli.js:6` 的 `AGENT_CLI_PROVIDERS = new Set([DEFAULT_AGENT_CLI_PROVIDER, 'claude', 'opencode'])`（`DEFAULT_AGENT_CLI_PROVIDER='codex'`，`agentCli.js:5`）；显示名映射在 `AGENT_CLI_DISPLAY_NAMES`（`agentCli.js:9`，仅 codex/claude/opencode）。每个 provider 在 `agentCliSpawnSpec`（`agentCli.js:187`）里按 `normalizedProvider` 分流构造 spawn 规格——claude（`agentCli.js:190`，`promptSource:'stdin'`、`lastFileSource:'claude-stream-json'`）、opencode（`agentCli.js:206`，`promptSource:'argument'`、`lastFileSource:'stdout'`），其余落到末尾的 codex 默认分支（`agentCli.js:221`，`useShell:true`、`promptSource:'stdin'`、`lastFileSource:'cli'`）。新增 oh-my-pi 即在此集合与 `agentCliSpawnSpec` 增加一个分支。

2. **oh-my-pi 的非交互形态与 claude/opencode 高度同构，可无新机制接入**。omp 官方文档（pi.dev/docs/latest/usage、omp.sh）明确：`omp -p` / `--print` 为「单次问答后退出」的非交互模式，**读取 stdin prompt、把最终回复写入 stdout 后退出**，无 TUI、无权限确认挂起（print 模式为 CI/脚本设计，不以沙箱/确认循环阻塞）。这与 codex/claude 的 `promptSource:'stdin'` 一致；其「stdout 即最终回复」则与 opencode 的 `lastFileSource:'stdout'`（`agentCli.js:355` 的 `fs.writeFileSync(lastFile, stdoutOutput)` 写回 last 文件）完全同构。因此 oh-my-pi 的 spawn 规格可照搬 opencode 的「stdout 捕获 + stdin 投递」范式，命令默认为 `omp`。

3. **会话复用体系对「无会话 provider」已有原生降级路径，oh-my-pi 可零分支做成无状态**。`src/loop/agentCliConfig.js:92` 的 `AGENT_CLI_SESSION_PROVIDERS = new Set([DEFAULT_AGENT_CLI_PROVIDER, 'claude', 'opencode'])` 列出支持会话复用的 provider；`agentCliProviderSupportsSession`（`agentCliConfig.js:305`）据此判定，不在集合内的 provider 在 `agentCliSessionContextFields`（`agentCliConfig.js:398`，命中 `if (provider && !agentCliProviderSupportsSession(provider)) return compactDefinedFields({ agentCliProvider: provider })`）与 `clearUnsupportedAgentCliSessionFields`（`agentCliConfig.js:591`）中被自动当作「无会话」处理——不产出任何会话字段、不触发 resume。执行链路里 `runCodex`（`src/loopService.js:898`）的 `isCodexProvider`/`isClaudeProvider`/`isOpenCodeProvider`（`loopService.js:908`/`909`/`910`）三条会话分支、`executeTask`（`src/loop/taskExecution.js:178`）的 `isTaskCodexProvider`/`isTaskClaudeProvider`/`isTaskOpenCodeProvider`（`taskExecution.js:184`/`185`/`186`）三态分支，对 oh-my-pi 一律为 false，`startedSessionContext` 落到末尾 `: {}`（`taskExecution.js:218`）、批次并发落到非 opencode 的并行分支（`taskExecution.js:143`）。**即：只要不把 oh-my-pi 加入 `AGENT_CLI_SESSION_PROVIDERS`，整个 loop/runtime/taskEvents/loopService 的会话分支无需任何新增 provider 专属逻辑**，每次「生成计划/执行任务」都是一次全新的 `omp --print` 调用——这与 omp `-p` 单次问答的语义天然吻合。会话复用（omp `--resume`）留作后续增强，不在本需求范围。

4. **provider 的「显示名」与「默认命令」在前后端各有一份，需逐处补齐**。后端：`agentCliConfig.js:447` 的 `agentCliProviderDisplayName`（claude/opencode/codex 三分支）、`agentCli.js:25` 的 `defaultAgentCliCommand`（`return normalizeAgentCliProvider(provider)`——codex/claude/opencode 的 provider 名即命令名，oh-my-pi 是首个 provider 名 `oh-my-pi` 与命令名 `omp` 不一致者，需在此做一处映射）、`src/mcpTools.js:16` 的 `AGENT_CLI_PROVIDERS = Object.freeze(['codex','claude','opencode'])`（MCP `create_project` 等工具的 provider 白名单，`COMMON_CLI_PROPERTIES.agentCliProvider.enum`，`mcpTools.js:58`）。前端：`src/renderer/components/shared.tsx:38` 的 `agentCliProviderLabel`、`shared.tsx:97` 的 `normalizeAgentCliProvider`、`shared.tsx:100` 附近的默认命令解析、`src/renderer/types.ts:4` 的 `AgentCliProvider` 类型联合、`src/renderer/utils/workspaceForms.ts:37` 的 `agentCliOptions`、`workspaceForms.ts:56` 的 `agentCliOptionDetails`、`workspaceForms.ts:217` 的默认命令解析、`src/renderer/components/workspace/WorkspaceSettingsView.tsx:33` 的 `agentCliNavLabel`/`agentCliNonCodexHint`（`WorkspaceSettingsView.tsx:39`）及命令提示分支（`WorkspaceSettingsView.tsx:199`）。这些点位在加入 opencode 时已逐一打过补丁，本需求对 oh-my-pi 依样补齐即可。

5. **CLI 错误/日志分类有两处 provider 名正则需同步**。`src/loop/validation.js:272` 的失败分类正则 `/(?:Agent CLI|Codex CLI|Claude CLI|OpenCode CLI)/i`（用于把 CLI 失败归为 `agent` 类）；`src/renderer/components/CodexLog.tsx:206` 的日志行判定 `hasProviderLabel`（`text.includes(providerLabel) || /\b(?:Agent|Codex|Claude|OpenCode)\b/i.test(text)`）。oh-my-pi 的可读错误文案为 `Oh My Pi CLI`（由 `AGENT_CLI_DISPLAY_NAMES['oh-my-pi']='Oh My Pi'` + `' CLI'` 拼成，`agentCli.js:582`/`591`），需把这两处正则补上 `Oh My Pi`，否则 oh-my-pi 的失败不会被归为 `agent_failure`、日志行不会带 provider 标签。

6. **既有链路成熟可复用、且必须不回退**。`runAgentCliAttempt`（`agentCli.js:232`）已能按 `spawnSpec` 的 `promptSource`/`lastFileSource`/`useShell` 统一处理投递、捕获、超时、取消、可读错误；`agentCliExecutionSpec`（`agentCli.js:510`）的 Windows `.bat/.cmd` 解析（`resolveWindowsCommand`，`agentCli.js:527`）对 `omp`/`omp.exe`/`omp.cmd` 等 PATH 解析通用。oh-my-pi 分支只需复用，不改这些公共逻辑。codex/claude/opencode 既有行为、思考深度（仅 codex）、会话复用、事件流、MCP/项目/需求/计划/任务链路、工作区导航均保持不变。

7. **受影响文件已逼近行数上限，需先行调高**。`scripts/check-file-lengths.js` 的 `fileLimits`（`check-file-lengths.js:9`）中：`src/loop/agentCliConfig.js` 当前 699/700（补 1 行显示名即超）、`scripts/smoke-test.js` 当前 2990/3000（补 oh-my-pi 后端 smoke 约 90+ 行即超）、`src/agentCli.js` 610/640（补 spawn 分支约 20 行后偏紧）。需按 #67 的 P002 范式**先单独调高三处上限**，避免后续实现任务触发 `npm run check:lengths` 失败。

综上，本需求的核心工作为：在 `agentCli.js` 注册 `oh-my-pi` provider（显示名 Oh My Pi、默认命令 omp）并新增 `--print` stdin/stdout 无状态 spawn 分支；在前后端逐处补齐显示名/默认命令/类型/选项/设置 UI/MCP 白名单/错误与日志正则；**不**接入会话复用（沿用既有无会话降级）；并补充单测与 smoke 后端用例。

## 设计约束

- **provider 键定为 `oh-my-pi`，显示名 `Oh My Pi`，默认可执行命令 `omp`**：与需求/仓库名一致、便于 DB/事件可追溯；命令名与 provider 名不一致是首个特例，**唯一**需要在 `defaultAgentCliCommand`（`agentCli.js:25`）加一处映射 `normalized === 'oh-my-pi' ? 'omp' : normalized`（前端 `shared.tsx`/`workspaceForms.ts` 的默认命令解析各加一处对称映射），其余 provider 不受影响。用户仍可在设置里覆盖为自定义命令路径（如 `/usr/local/bin/omp`），覆盖逻辑沿用 `normalizeAgentCliCommand(command) || defaultAgentCliCommand(provider)`（`agentCli.js:189`）。
- **oh-my-pi 为无状态单次后端，不加入 `AGENT_CLI_SESSION_PROVIDERS`**：spawn 规格固定 `args=['--print']`、`promptSource:'stdin'`（prompt 经 stdin 投递，规避命令行长度/转义问题，与 codex/claude 一致）、`lastFileSource:'stdout'`（把 omp 最终回复写入 last 文件，与 opencode 同构，复用 `agentCli.js:355` 的 stdout→lastFile 写回）、`useShell:false`（omp 为原生二进制，与 opencode 一致；Windows 下由 `resolveWindowsCommand` 解析 PATH）。**不**新增 session 字段、**不**新增 codex/claude/opencode 之外任何会话分支——既有 `agentCliProviderSupportsSession` 返回 false 即让全链路降级为「每次全新调用」。任务并发沿用非 opencode 的并行分支（无共享会话，天然安全）。
- **`--print` 模式无需补权限/沙箱标志**：omp `-p`/`--print` 为 CI/脚本设计的非交互模式，无 TUI、无确认挂起（官方 pi.dev/docs/latest/usage）；且 omp 本身不做沙箱、以当前用户权限运行（无需也不存在 `--dangerously-skip-permissions` 类标志）。命令仅 `omp --print` + stdin prompt，不臆造其他标志；若 last 文件混入 ANSI 控制序列（验收时观察），作为后续小优化项（omp 的 color 关闭标志）单独处理，不在本计划强行加未确认的标志。
- **思考深度仅 codex 生效，oh-my-pi 一律 `null`**：既有 `provider === DEFAULT_AGENT_CLI_PROVIDER` 守卫（`agentCliConfig.js:136`/`176`/`238`、`agentCliOperationFields` `agentCliConfig.js:198`）已自动让非 codex provider 的 `codexReasoningEffort` 为 null，无需新增分支；UI 上「思考深度」仅对 codex 显示（既有 `isCodexProvider` 判定），oh-my-pi 与 claude/opencode 一样隐藏该字段。
- **显示名/默认命令/类型/选项/UI/白名单/正则逐处补齐，且与 opencode 接入保持同构**：后端 `agentCliConfig.js:447`（`agentCliProviderDisplayName` 加 `'Oh My Pi'`）、`mcpTools.js:16`（白名单数组加 `'oh-my-pi'`）、`validation.js:272`（正则加 `Oh My Pi CLI`）；前端 `shared.tsx:38/97/100`（label/provider 归一/默认命令）、`types.ts:4`（类型联合加 `'oh-my-pi'`）、`workspaceForms.ts:37/56/217`（两个选项表 + 默认命令解析）、`WorkspaceSettingsView.tsx:33/39/174/199`（导航 label/非 codex 提示/卡片标题文案/命令提示分支）、`CodexLog.tsx:206`（正则加 `Oh My Pi`）。文案与描述风格与既有 opencode 条目一致。
- **IPC/数据库/快照不新增**：`agent_cli_provider` 列为 `TEXT`、`normalizeAgentCliProvider` 已能接受任意集合内字符串，无需 schema 迁移；`src/loop/snapshots.js:373` 的会话列分支仅在 `provider==='opencode'` 取特值、oh-my-pi 落默认（空会话），无需改动；preload/IPC 不涉及（provider 经既有 loop 配置/需求/反馈字段透传）。
- **文件长度约束（关键风险）**：`npm run check` 含 `scripts/check-file-lengths.js`，受影响文件逼近上限——`agentCliConfig.js` 699/700、`smoke-test.js` 2990/3000、`agentCli.js` 610/640。本计划单独设一个任务（P002）**先行调高三处配置上限**（沿用该配置文件 per-file 覆盖范式并补注释说明原因），其余实现任务不触碰 `check-file-lengths.js`，避免并发改同一文件。
- **既有链路不得回退**：codex/claude/opencode 三后端的命令拼装、stdin/stdout/stream-json 捕获、会话复用、思考深度、超时/日志/取消、事件流、手动运行、并发批次策略、MCP 工具、项目/需求/计划/任务链路与工作区导航均保持不变；未选 oh-my-pi 的项目行为与现状逐字一致。
