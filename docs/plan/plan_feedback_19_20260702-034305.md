- 涉及模块：AI 配置管理、聊天页配置状态、对话创建、主进程 Chat/AI Config IPC、回归测试

## 需求与背景

用户已经在「设置 -> AI 对话」中创建了 AI 配置，但切到「AI 对话」页后仍看到“尚未配置 AI 接口”，并且因为 `ChatView` 在未配置状态下提前返回，左侧“新建对话”入口也不会渲染，导致对话无法创建。

基于当前代码定位，设置页的新多配置链路写入的是 `ai_configs` 表：`WorkspaceSettingsView.tsx` 调用 `window.autoplan.aiConfigCreate`，主进程 `main.js` 的 `ai-config:create` 继续调用 `src/chat/aiConfigService.js`。聊天页状态却仍由 `useChat.ts` 初始化时调用旧接口 `chatGetConfig()` 决定，而 `main.js` 的 `chat:getConfig` 只读取旧的全局 `settings` 表 `chat.*` 键。于是即使项目级 `ai_configs` 已有可用 API Key，只要旧 `chat.apiKey` 为空，`ChatView.tsx` 仍会用 `config.hasApiKey === false` 显示“尚未配置 AI 接口”并阻断对话 UI。

服务端实际发送消息时，`chatController.js` 已经通过 `resolveAiConfigForConversation` 按 `conversation.ai_config_id -> 项目首条 ai_configs -> 内置默认` 解析配置。因此本次修复重点是让聊天页的“是否已配置”判断和对话创建入口使用同一套项目级 AI 配置事实源，并补齐对话创建绑定/回退的回归测试，避免旧 `chat.*` 状态继续误判。

## 设计约束

- 聊天页不得再用旧全局 `chatGetConfig()` 的 `hasApiKey` 作为项目级 AI 配置是否可用的唯一判断。
- 未配置 API Key 时可以提示用户去设置，但不能因为空状态提前返回而隐藏对话列表和“新建对话”按钮。
- 新建对话默认应能绑定当前项目首个已有 API Key 的 AI 配置；如未显式绑定，后端配置解析仍应能回退到项目首条配置。
- 修改应保持现有多配置 CRUD、API Key 脱敏、不向渲染层返回明文密钥的边界不变。
- 测试应覆盖“旧 `chat.apiKey` 为空但项目 `ai_configs` 已配置”场景，防止聊天页继续误报“尚未配置 AI 接口”。

## 总体验收标准

- 功能正确性：项目级 AI 配置创建成功后，聊天页以 `ai_configs` 为配置事实源，不再被旧 `chat.*` 全局设置误判为未配置。
- 对话可用性：无论是否已有会话，聊天页都保留会话侧栏和“新建对话”入口；已配置 API Key 后可以创建会话并发送消息。
- 配置绑定：新建会话优先绑定当前项目可用配置；未绑定时后端仍按项目首条配置回退；跨项目配置不会被误绑定。
- 安全边界：渲染层只接收脱敏密钥状态，不返回明文 API Key；服务端发送 LLM 请求时才读取真实密钥。
- 最终验收命令：`npm run check && npm test && npm run build`；范围覆盖静态检查、单元测试、渲染层回归测试和构建；通过标准为命令退出码 0 且手动验收路径符合预期。
