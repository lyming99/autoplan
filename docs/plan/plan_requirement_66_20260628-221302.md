# 需求 #66 开发计划：模型配置档案与一键切换（类 cc switch）

## 背景

需求原话：「能否像 cc switch 那样，配置模型、切换模型？」cc switch（关尔/cc-switch）是一款用于**管理多套 AI 供应商预设并一键切换激活项**的工具：每套预设包含「名称 + 供应商协议（Anthropic/OpenAI 兼容等）+ API Key + Base URL + 模型名」，用户在多个预设之间一键切换「当前激活」的那一套，被激活的预设即生效。用户希望 AutoPlan 也具备同类能力——在应用内配置多套模型/供应商档案并快速切换。

现状梳理（结合代码）：

1. **AutoPlan 通过 spawn agent CLI 执行任务，但「模型 / API Key / Base URL」不由 AutoPlan 管理**。`src/agentCli.js` 的 `runAgentCliAttempt` 以 `env: env || process.env` 启动 codex/claude/opencode 子进程；`env` 由 `src/loopService.js` 在 `runCodex` 处组装（`src/loopService.js:1136` 的 `env: workspaceToolEnv(workspace)`），而 `workspaceToolEnv`（`src/loop/workspaceFiles.js:104`）只注入工作区隔离相关的目录变量（`PUB_CACHE`/`GRADLE_USER_HOME`/`XDG_*`/`APPDATA` 等），**不含任何 API Key / Base URL / 模型名**。这些凭据目前完全依赖 CLI 自身认证（`~/.claude`、`~/.codex` 等），AutoPlan 既看不到也无法切换。

2. **当前可配置的只有「用哪个 CLI 二进制」**。项目状态列里保存 `agent_cli_provider`（codex/claude/opencode）、`agent_cli_command`、`codex_reasoning_effort`（见 `src/loop/agentCliConfig.js` 的 `AGENT_CLI_PROVIDER_COLUMNS`/`CODEX_REASONING_EFFORT_COLUMNS` 与 `src/renderer/components/workspace/WorkspaceSettingsView.tsx` 的「CLI 后端」面板）。这是「CLI 后端」维度，与「模型/供应商」维度正交——用户无法在不改动本机 CLI 配置的前提下，让 AutoPlan 在不同模型/供应商之间切换。

3. **已有可复用的「配置模块 + IPC + 脱敏」范式**。`src/mcpConfig.js` 的 `saveMcpSettings(db, input)` 按 `mcp.` 前缀落库、`normalizeMcpAuthToken` 允许清空、`src/loop/snapshots.js` 的 `mcpStatusSnapshot` 以 `hasAuthToken`/`authTokenMasked`（末 4 位）脱敏下发——这是需求 #55 刚落地的成熟模式，本需求的「模型档案」可完全沿用同一套范式（settings 表前缀存储 + 脱敏快照 + 显式确认保存）。

4. **设置存储原语齐备**。`src/database.js` 提供 `getSetting(key, fallback)`/`setSetting(key, value)`/`getSettings(prefix)`（`database.js:330`），无需新增数据表，模型档案以 `model.` 前缀写入既有 `settings` 表即可。

5. **前端设置面板为 nav 分页结构**。`WorkspaceSettingsView.tsx` 有 `loop`/`cli`/`scope`/`mcp` 四个分页，MCP 面板已抽成独立子组件 `McpControlPanel.tsx`（`WorkspaceSettingsView.tsx:15` 导入）。新增「模型」分页可复用该子组件模式，避免视图文件超出行数上限。

综上，本需求的核心工作为：新增一个**模型档案配置模块**（多套预设的 CRUD + 激活项 + 脱敏 + env 派生），把激活档案在 spawn agent 时注入为环境变量/模型参数，并通过 IPC + 新增「模型」设置面板暴露「配置 + 一键切换」交互。

## 设计约束

- **复用 settings 表 + 前缀存储，不新增数据表**：模型档案以 `model.profile.<id>.{name,protocol,apiKey,baseUrl,model,createdAt,sortOrder}` 与 `model.activeProfileId` 写入 `settings` 表，读取用 `db.getSettings('model.')`。命名风格沿用 `mcp.` 前缀范式（`src/mcpConfig.js`）。档案为**应用全局**（非每项目），激活项对所有 loop/项目生效——与 cc switch「全局切换」一致。
- **以 `src/modelProfiles.js` 为新配置模块，沿用 `src/mcpConfig.js` 范式**：导出 `loadModelProfiles(db)`（返回 `{ profiles, activeProfileId }`）、`saveModelProfile(db, input)`（新增/更新，返回是否变化与落库 id）、`deleteModelProfile(db, id)`、`setActiveModelProfile(db, id)`、`activeModelProfileEnv(db)`（派生激活档案的环境变量对象）、`normalizeModelApiKey`/`maskApiKey` 等纯函数。归一化/脱敏逻辑集中在此文件，不在多处重复。
- **API Key 默认脱敏，绝不明文下发渲染层**：快照中档案的 `apiKey` 一律替换为 `hasApiKey: boolean` + `apiKeyMasked`（仅末 4 位，如 `····1234`，无 key 时为空）——完全沿用 `mcpStatusSnapshot` 对 `authToken` 的脱敏处理（需求 #55 P002）。渲染层表单的密钥输入框**默认留空**（占位提示「已设置密钥（····1234）」），用户输入新值即覆盖、点「清空」即下发空串，不再从快照回填明文。
- **允许清空 API Key**：`normalizeModelApiKey` 传入空值时返回空串**不抛错**（沿用 #55 对 `normalizeMcpAuthToken` 的修正）；保存区分「设置 key（trim 后非空）」与「清除 key（显式空串）」两种意图。`saveModelProfile` 对「仅更新部分字段」要支持：未显式传 `apiKey` 时保留原值、显式传空串时清空。
- **激活档案的生效机制为「环境变量注入 + 模型参数注入」，不写全局 CLI 配置文件**：AutoPlan 已用 `workspaceToolEnv` 把 CLI 的配置/缓存目录隔离到工作区运行时目录（`workspaceFiles.js:104`），因此**不能**去写用户真实的 `~/.claude`/`~/.codex`（会污染用户本机配置、且与工作区隔离冲突）。正确做法是在 `loopService.runCodex` 组装 agent env 处（`loopService.js:1136`）把 `activeModelProfileEnv(db)` 合并入 `workspaceToolEnv(workspace)`：按档案 `protocol`（`anthropic` → `ANTHROPIC_API_KEY`/`ANTHROPIC_BASE_URL`；`openai` → `OPENAI_API_KEY`/`OPENAI_BASE_URL`）注入对应 env；模型名则按当前 `agentCliProvider` 经 CLI 参数注入（claude → `--model <model>`；codex → 追加 `-c model="<model>"`，与既有 `codexReasoningConfigArgs` 的 `-c` 写法一致；opencode → 尽力而为/沿用其 provider 配置，不阻塞）。各 CLI 实际识别的 env/参数以 CLI 官方文档为准，实现时核对，未识别项不报错。
- **模型档案与 CLI 后端正交，互不污染**：既有「CLI 后端」面板（provider/命令/Codex 深度）保持不变；新增「模型」面板独立管理档案与激活项。激活档案在 spawn 时叠加在所选 CLI 之上。既有 `agentCliConfig`/会话恢复/事件流/loop 配置/MCP/项目链路**不得回退**。
- **切换为「显式确认」动作**：「设为激活」与「保存含 key 的档案」均为明确按钮触发（文案/二次确认），避免误切换或误覆盖 key；切换成功后 spawn agent 即按新档案生效，无需重启应用（下一轮 loop 的 agent 调用即用新 env）。
- **随机 key 生成在渲染层完成**：「生成随机密钥」按钮用 `window.crypto.getRandomValues` 生成高熵随机串（如 32 字节 base64url）填入输入框，无需主进程往返（沿用 #55 P004/P005 决策）。
- **IPC 命名与校验沿用既有风格**：新增通道 `models:list`/`models:save`/`models:delete`/`models:setActive`，失败抛中文错误；写操作返回 `loop.snapshot(projectId)` 以刷新 UI。preload 在 `window.autoplan` 上以一行式 `ipcRenderer.invoke` 暴露，命名与 `saveMcpConfig`/`configureLoop` 一致。
- **文件长度约束**：`npm run check`（含 `scripts/check-file-lengths.js`）对单文件行数有限（`src/main.js` 800、`src/renderer/types.ts` 980，默认 600）。`WorkspaceSettingsView.tsx` 现已接入 MCP 子组件，「模型」面板同样抽成独立子组件 `ModelControlPanel.tsx`（镜像 `McpControlPanel.tsx`）；档案表单状态/校验下沉到 `workspaceForms.ts`，配置/脱敏逻辑集中在 `src/modelProfiles.js`，避免单文件超限。
- **不要把运行测试/回归/构建拆成普通开发任务**；最终检查统一放入最后的「完整验收」节点。
