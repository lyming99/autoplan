# 开发计划：需求 #18 执行器模块与悬浮控制按钮

## 需求分析

当前 AutoPlan 已有循环执行、任务执行和脚本模块。脚本模块支持手动、钩子、定时运行，适合自动化片段，但它不是面向项目运行态的“执行器”。本需求更接近为每个项目配置一组可自定义命令的执行器，支持启动、停止、刷新三类动作，并在工作区顶部标题栏下方右上角提供 3 个悬浮按钮，便于 Flutter 等项目通过插件、脚本或命令直接控制预览/运行进程。

推荐方案：新增独立 executor 数据模型和运行服务，不复用 scripts 表承载运行态，避免和 AutoPlan 自身 plan/task 生命周期混淆。执行器动作底层复用现有 `runShell`、runtime operation、事件流、环境变量和日志目录能力；前端在设置页配置默认执行器，在顶部 UI 提供启动/停止/刷新按钮。

## 设计原则

- 执行器属于项目级能力，每个项目可配置自定义启动、停止、刷新命令，命令为空时对应按钮禁用。
- 执行器动作不影响 AutoPlan 循环的启动/停止状态，也不阻塞需求、反馈、Plan、任务执行。
- 执行器运行状态要进入 snapshot，UI 可展示 running/stopped/failed、最近退出码、最近日志、最后运行时间。
- Flutter 场景以命令/脚本适配为主，例如 `flutter run -d windows`、自定义 hot reload 脚本或插件命令；AutoPlan 不在第一版内绑定具体 Flutter SDK API。

## 影响范围

- 主进程：新增执行器 CRUD、启动、停止、刷新 IPC。
- 循环服务：新增 executor 运行态和状态写回，复用 `runShell` 与 runtime cancellation。
- 数据库：新增项目执行器表或等价项目级配置表，并兼容历史数据库迁移。
- 渲染层：扩展 snapshot 类型、设置页配置入口、顶部悬浮控制按钮和状态反馈。
- 测试：覆盖数据库迁移、命令派发、停止运行中进程、UI 按钮启用/禁用与动作调用。

## 任务拆解

- [ ] P001: 建立执行器数据模型与快照字段 <!-- scope: src/database.js,src/loop/snapshots.js,src/renderer/types.ts -->
  - 验收要点：
    - 数据库迁移新增项目级执行器配置，至少包含 `project_id`、名称、启动命令、停止命令、刷新命令、工作目录、启用状态、最近状态、最近退出码、最近日志、运行时间字段。
    - 历史数据库启动后自动补齐执行器表/字段，不破坏现有 projects、project_states、scripts、plans 数据。
    - `snapshot` 返回当前项目执行器配置和最近运行态，渲染层类型中有明确的 `Executor`、`ExecutorAction`、`ExecutorRunStatus` 定义。
    - 未配置执行器时 snapshot 返回空配置或默认空对象，前端可据此禁用按钮而不是抛错。

- [ ] P002: 实现执行器动作运行服务 <!-- scope: src/loop/executors.js,src/loopService.js,src/loop/runtime.js -->
  - 验收要点：
    - 新增执行器服务封装 `start`、`stop`、`refresh` 三类动作，动作命令从项目执行器配置读取。
    - 启动与刷新动作通过现有 `runShell` 执行，继承项目工作区、项目环境变量、日志写入和 active operation 更新能力。
    - 停止动作优先终止运行中的执行器进程；如配置了停止命令，则在无运行进程或终止后按配置执行停止命令。
    - 执行器动作写入事件流，事件 meta 包含 executorId、action、exitCode、log、timedOut、commandConfigured 等可排查信息。
    - 执行器动作失败不会影响 AutoPlan plan/task 状态，也不会改变循环 `running` 字段。

- [ ] P003: 暴露执行器 IPC 与预加载 API <!-- scope: src/main.js,src/preload.js,src/renderer/types.ts -->
  - 验收要点：
    - 主进程提供保存执行器配置、启动执行器、停止执行器、刷新执行器的 IPC handler。
    - IPC 入参校验 projectId、executorId/action，项目不存在、命令为空、工作区为空时返回可读错误。
    - `preload` 暴露 `saveExecutor`、`startExecutor`、`stopExecutor`、`refreshExecutor` 等 API，命名与现有 `startLoop`、`runScript` 风格一致。
    - TypeScript `AutoplanApi` 补齐新增方法签名，调用方无需使用临时类型断言。

- [ ] P004: 增加执行器配置界面 <!-- scope: src/renderer/components/workspace/WorkspaceSettingsView.tsx,src/renderer/utils/workspaceForms.ts,src/renderer/styles/workspace.css -->
  - 验收要点：
    - 设置页新增“执行器”分组，可编辑名称、启动命令、停止命令、刷新命令、工作目录、启用状态。
    - 工作目录支持为空时使用项目工作区，并说明可填写相对路径或绝对路径。
    - 保存配置后 snapshot 同步刷新，切换项目时表单显示对应项目执行器配置。
    - 命令输入使用等宽字体，长命令在桌面宽度和窄屏下不溢出。
    - 未启用或命令为空时给出明确状态提示，不要求用户必须配置 Flutter。

- [ ] P005: 在顶部标题栏下方加入启动/停止/刷新悬浮控制 <!-- scope: src/renderer/pages/WorkspacePage.tsx,src/renderer/components/icons.tsx,src/renderer/styles/workspace.css -->
  - 验收要点：
    - 工作区主区域顶部标题栏下方右上角出现 3 个悬浮图标按钮：启动、停止、刷新。
    - 按钮使用现有图标体系，提供 title/aria-label，视觉上不遮挡搜索框、状态 pill、页面内容和移动端布局。
    - 启动按钮调用执行器 start，停止按钮调用 executor stop，刷新按钮调用 executor refresh；运行中按钮显示 loading/disabled 状态，避免重复点击造成并发动作。
    - 未配置对应命令或执行器未启用时按钮禁用，并通过 tooltip/title 说明原因。
    - 悬浮控制只控制项目执行器，不改变 AutoPlan 循环启动/停止按钮语义。

- [ ] P006: 补充执行器测试代码 <!-- scope: src/loop/executors.test.js,src/renderer/pages/WorkspacePage.test.tsx,scripts/smoke-test.js -->

    - 通过标准：命令全部退出码为 0；新增/既有 smoke 场景全部通过；配置执行器后可从顶部按钮触发启动/停止/刷新并在事件流和日志中看到对应记录；未配置命令时按钮禁用且无未捕获异常。

## 总体验收标准

1. 每个项目可保存独立执行器配置，重启应用后配置仍存在。
2. 执行器支持自定义启动、停止、刷新命令，命令可用于 Flutter 项目的插件、脚本或任意本地命令方案。
3. 顶部标题栏下方右上角始终可见启动/停止/刷新 3 个悬浮按钮，且按钮状态与执行器配置、运行态一致。
4. 执行器运行日志、错误、退出码可追踪，不影响 AutoPlan 自身循环、任务执行和验收流程。
5. 自动化验收 `npm run check && npm run smoke` 全部通过。
