# 反馈 #14 开发计划：OpenCode CLI 计划生成格式与内容失真治理

## 背景与目标

反馈指出：选用 OpenCode CLI 作为后端生成计划时，无法像 Claude CLI 那样严格遵循 AutoPlan 的计划格式（`- [ ] P0NN: ... <!-- scope -->`），也没有依据给定的需求/反馈正文生成，而是自行读取项目内容后"自行发挥"生成计划。

基于当前代码梳理的根因：

1. **OpenCode 是完全 agentic 的 CLI，默认配置鼓励自主探索。** `src/agentCli.js` 中 OpenCode 经 `opencode run --format default --dangerously-skip-permissions` 执行，使用内置通用编码 agent（自带 system prompt、全量工具，并读取 `AGENTS.md`/`.claude`）。Claude 走 `--print` 单轮模式，忠实执行 prompt；OpenCode 则倾向主动读取 README、目录树与源码再自行总结，从而忽略 prompt 里的反馈正文与格式硬约束。
2. **计划生成 prompt 本身反向激励探索。** `src/loop/planGeneration.js`（`generatePlanForIntake`）的 prompt 写有"先阅读相关模块源码（README、目录树、相关文件）"，且正文较短（<20 字）时会自动注入 README 摘要与根目录概览——这恰恰把 OpenCode 引向"读项目"而非"用反馈正文"。
3. **长 prompt 投递存在语义错误。** `src/agentCli.js` 的 spillover 机制在 prompt 超命令行长度上限时，把完整 prompt 落盘，位置参数只给一句"完整指令见随附的 prompt 文件…"，并用 `-f <file>` 投递。但 OpenCode 官方文档（opencode.ai/docs/cli）明确 `--file`/`-f` 的语义是"File(s) to attach to message"（把文件作为消息附件），而非"从文件读取 prompt"。结果 OpenCode 收到的用户消息是一句含糊指针，真正的指令仅作为附件被弱引用，agentic 模型更易忽略并转向自主探索。
4. **缺少产物格式校验兜底。** `recoverPlanFromStdout` 只按首个 `## ` 标题截取 stdout 落盘，不校验是否含 `## 任务拆解` 与合规任务行；OpenCode 产出畸形内容时会被静默落库（`syncPlanTasks` 后表现为空/错乱任务）。

目标：用 OpenCode 原生的 `--agent` 机制约束其计划生成行为（专用 agent + 受限工具 + 强 system prompt），修正 prompt 的 provider 感知与长 prompt 投递语义，并补齐产物格式校验，使 OpenCode 后端的计划与 Claude 后端同样严格遵循 AutoPlan 格式、且忠实于需求/反馈正文。

## 任务拆解

- [x] P001: 新增 OpenCode 计划生成专用 agent 生成器 <!-- scope: src/loop/opencodeAgent.js -->
  - 验收要点：新增模块导出常量（如 `AUTOPLAN_OPENCODE_PLAN_AGENT = 'autoplan-plan'`）与生成函数，幂等把专用 agent 写入 `<workspace>/.opencode/agent/autoplan-plan.md`，文件名使用 AutoPlan 命名空间以避免与用户既有 agent 冲突；目录不存在时自动创建。
  - 验收要点：agent frontmatter 声明 `mode: primary`，`permissions` 仅允许 `read,edit,glob,grep`（覆盖写计划文件所需），显式拒绝 `bash,webfetch,websearch,task,todowrite,lsp,skill`，从工具层抑制自主探索/联网/子代理。
  - 验收要点：agent 的 system prompt 明确：仅依据用户消息中提供的需求/反馈正文与显式列出的附件/文件生成计划；最小化读取仓库（仅限为判断 scope 必要的少量文件）；严格产出 `## 任务拆解` 章节与 `- [ ] P0NN: 标题 <!-- scope: ... -->` 任务行；必须用 write/edit 工具把完整 plan 写入指定输出文件路径；不得反问、不得改业务代码、不得自主重写计划主题。

- [x] P002: 在 OpenCode 计划生成调用中接入 --agent <!-- scope: src/agentCli.js,src/loopService.js -->
  - 验收要点：`opencodeCliArgs`/`agentCliSpawnSpec` 支持 `agent` 选项，计划生成场景下在 `run` 子命令参数中追加 `--agent autoplan-plan`（位于 `run`、`--format`、`--dangerously-skip-permissions`、`--session`/`--title` 之后、位置 prompt 之前），不破坏既有参数顺序。
  - 验收要点：`runCodex` 在 provider 为 opencode 且为计划生成操作（label 为 `generate-plan`、或以 `gen-requirement-`/`gen-feedback-` 开头，或 `operation.intakeType` 存在）时，spawn 前调用 P001 生成器确保 agent 文件已落盘，并把 agent 名称传入 agentCliOptions；非计划生成操作（任务执行/修复）不注入该 agent，沿用现有行为。
  - 验收要点：任务执行/修复阶段仍按现有逻辑复用 OpenCode session（`--session`/`--title`），`--agent` 注入仅限计划生成阶段，不破坏会话复用、session 回退新建与失败链路。

- [x] P003: 修正 OpenCode 长 prompt 投递语义 <!-- scope: src/agentCli.js -->
  - 验收要点：纠正对 `-f/--file` 的误解：在代码注释中明确其语义为"把文件作为消息附件"而非"从文件读取 prompt"，依据 opencode.ai/docs/cli 的 `run --file` 说明。
  - 验收要点：当确实需要 spillover 时，位置参数的指针消息改为权威指令（明确告知"你的完整唯一指令已作为附件 <路径> 提供，必须完整读取该附件并严格按照其内容执行，禁止转而探索仓库或重写任务主题"），使 OpenCode 把附件当作唯一权威指令而非含糊提示。
  - 验收要点：保持对 codex/claude/oh-my-pi（promptSource=stdin）无影响；spillover 仅作用于 promptSource=argument 的 opencode 路径；落盘临时文件清理逻辑保持不变。

- [x] P004: 计划生成 prompt 改为 provider 感知 <!-- scope: src/loop/planGeneration.js -->
  - 验收要点：`generatePlanForIntake` 在 provider 为 opencode 时，弱化"自主阅读源码/README/目录树"的措辞，改为"优先依据给定需求/反馈正文；仅可为推断 scope 读取少量明确文件"，避免把 agentic 后端引向整仓探索。
  - 验收要点：opencode 后端下跳过"短正文自动注入 README 摘要与根目录概览"的逻辑（该注入既膨胀 prompt、易触发 spillover，又强化"读项目"倾向）；claude/codex/oh-my-pi 行为不变。
  - 验收要点：provider 判断复用 `effectiveAgentCliConfig`，与 P002 的计划生成判定保持一致，避免两处判定漂移。

- [x] P005: 计划产物格式校验与失败兜底 <!-- scope: src/loop/planGeneration.js -->
  - 验收要点：新增校验函数，判断落盘/恢复的 plan 内容是否同时满足：含 `## 任务拆解` 二级标题，且至少一行匹配 `- [ ] P0NN: ... <!-- scope: ... -->`（编号连续性可由 `syncPlanTasks` 后续处理，本处仅校验格式存在性）。
  - 验收要点：校验不通过时不落库畸形 plan：记录 `plan.format.invalid` 事件（含 provider、日志路径、plan 文件路径），并按既有 `plan.generate.failed` 失败链路处理（触发 `on:fail`、`generate_fail_count` 自增），而非把畸形内容静默写入并 syncPlanTasks。
  - 验收要点：校验在 stdout 兜底落盘与 opencode 直接写盘两条路径上均生效；claude/codex 正常产物不被误判为非法。

## 总体验收标准

最终验收命令：

```bash
npm test
npm run check
npm run build
```

验收范围：
- OpenCode agent 生成：`.opencode/agent/autoplan-plan.md` 的 frontmatter（mode/permissions）与 system prompt 内容。
- 调用接入：`opencode run ... --agent autoplan-plan` 仅在计划生成阶段注入；任务执行/修复与 session 复用不受影响。
- prompt 投递：opencode 下不自动注入 README/目录概览、措辞弱化探索；长 prompt spillover 指针为权威指令，`-f` 作为附件语义被正确理解。
- 产物校验：`## 任务拆解` + 合规任务行存在性校验，畸形 plan 不落库并触发 `plan.format.invalid` / 失败链路。
- 回归：claude/codex/oh-my-pi 计划生成、任务执行、人工验收、快照、搜索与 MCP 主流程无回归。

通过标准：
- OpenCode 后端生成的计划严格遵循 AutoPlan 任务拆解格式，且内容基于给定需求/反馈正文，不再出现"自行读取项目生成计划"。
- 缺格式或缺 scope 的畸形产出被校验拦截并按失败处理，不出现畸形 plan 落库。
- Claude/Codex/OpenCode 三类后端在计划生成、任务执行、会话复用上的既有行为保持一致，无回归。
- `npm test`、`npm run check`、`npm run build` 全部通过，无新增 TypeScript、Node 语法、构建或文件长度检查错误。
