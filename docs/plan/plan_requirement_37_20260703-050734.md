# 需求 #37：对话队列发送（AI 回复中可继续输入并排队）

> 生成时间：2026-07-03
> 需求：当前 AI 回复过程中对话输入被禁用，用户无法继续输入/发送。希望支持「队列发送」：AI 回复时输入框保持可用，新发送的消息进入队列按顺序处理；队列状态可见（排队中/处理中）、可管理（取消/编辑/清空、停止当前回复并处理下一条）；并在失败/刷新等场景下尽量保留未处理队列、不丢失内容。

---

## 需求分析与方案

### 当前现状（基于代码分析）

对话（Chat）链路横跨主进程 → preload → 渲染层，发送在多个环节被「流式中」硬性阻断：

| 环节 | 文件 | 现状（阻断点） |
| --- | --- | --- |
| 渲染：输入框禁用 | `src/renderer/components/workspace/ChatView.tsx` | `inputDisabled = isStreaming || !activeConversationId || !hasAiApiKey;`（`ChatView.tsx:417`）；`disabled={inputDisabled}`（`ChatView.tsx:603`）；流式时发送按钮被替换为停止按钮（`ChatView.tsx:692-710`），无法发送 |
| 渲染：发送早退 | `ChatView.tsx` `handleSend` | `if (!text \|\| isStreaming \|\| !activeConversationId \|\| !hasAiApiKey) return;`（`ChatView.tsx:440`）→ 流式中直接丢弃 |
| 渲染：Hook 发送守卫 | `src/renderer/hooks/useChat.ts` `sendMessage` | `if (!text \|\| stateRef.current.isStreaming \|\| stateRef.current.awaitingResponse \|\| !cid \|\| !projectId) return;`（`useChat.ts:568`）→ 流式中直接丢弃 |
| 主进程：IPC 入口 | `src/main.js` `chat:send` | 调 `controller.send(message)`，返回 `{accepted:true, conversationId}`（`main.js:552-568`） |
| 主进程：控制器 | `src/chat/chatController.js` `send` | `if (active) return;`（`chatController.js:56`）→ 已在生成中直接忽略重复发送，无队列 |
| 主进程：控制器生命周期 | `main.js` `getOrCreateChatController` / `onDone` | 每会话一个 controller（`chatControllers` Map）；`onDone` 回调 `chatControllers.delete(conversationId)`（`main.js:716`）→ 生成一结束即销毁，无续跑下一条的时机 |
| 持久化 | `chat_messages.status`（`database.js:224`，`TEXT` 无 CHECK 约束） | 已有 `done/streaming/aborted/error`；`buildMessages` 仅排除 `streaming`（`chatController.js:486-489`），其余都进 LLM 上下文 |
| 队列状态/事件 | 无 | 仅有 `chat:chunk`（`text_delta/tool_*/thinking_*/error/status`）与 `chat:done`，无队列状态通道 |

**关键结论**：当前是「流式中全链路丢弃发送」的模型，没有任何队列概念。要支持队列发送，需把**发送语义从「立即执行」改为「入队 + 顺序调度」**，且队列应放在**主进程（controller）**而非渲染层——这样队列天然跨渲染层刷新存活（Electron 主进程不随窗口刷新重启），并可落库跨应用重启恢复。

### 设计方案

**1. 主进程队列核心（`src/chat/chatQueue.js`，新文件）**
- 纯模块（依赖注入 `db` + `conversationId` + `projectId` + `emit` 回调），管理每会话 FIFO：`enqueue(message)` / `peekNext()` / `markProcessing(id)` / `releaseProcessing()` / `getQueue()` / `cancelItem(id)` / `editItem(id,text)` / `clear()` / `hasQueued()`。
- 队列项状态机：`queued`（排队中/等待发送）→ `processing`（处理中）→ 出队。
- 持久化：`enqueue` 把用户消息写为 `chat_messages` 行 `status='queued'`；`markProcessing` 将其 `UPDATE ... status='done'`（使其作为「刚发送的用户消息」进入 LLM 上下文）；`cancelItem`/`clear` 删除对应 `queued` 行。
- `loadPersisted()`：从 `chat_messages` 读出 `status='queued'` 行按 `id ASC` 重建 FIFO，用于刷新/重启后恢复。

**2. 控制器接入（`src/chat/chatController.js`，改造）**
- `send(message)` 改为委托 `queue.enqueue` 后调用 `pump()`；不再因 `active` 丢弃。
- 新增 `pump()`：`active` 或队列空则返回；否则 `peekNext`+`markProcessing` → `emit` 队列快照 → 以该消息内容启动 `runAgentLoop`。
- `finishGeneration` 在 `emitDone` 前调用 `pump()`：上一条结束后立即调度下一条（done/aborted/error 均触发）。
- `buildMessages` 额外排除 `status='queued'`（未派发的排队消息不得提前进入上下文）；正在处理项已被翻为 `done`，自然进入。
- 暴露 `getQueue()/hasQueued()/isActive()`；管理动作委托队列模块；新增 `onQueue` 依赖回调用于广播快照。
- 保留 `send/stop/getHistory/clearHistory/getConfig/...` 既有对外契约不变。

**3. 主进程 IPC + 生命周期（`src/main.js` / `src/preload.js`）**
- `chat:send` 调 `controller.send`（现语义=入队），返回 `{accepted, conversationId, enqueuedId}`。
- 新增 `chat:queueList / chat:queueCancel / chat:queueEdit / chat:queueClear`，均带 `conversationId/projectId` 校验（复用 `conversationInProject`）。
- `getOrCreateChatController`：创建后调用 `queue.loadPersisted()` + `pump()` 恢复历史排队；`onDone` 回调仅在 `!controller.isActive() && !controller.hasQueued()` 时 `chatControllers.delete`（有排队则保留 controller 续跑）。
- `onQueue` 回调 → `mainWindow.webContents.send('chat:queue', snapshot)`；preload 暴露 `onChatQueue` 及上述 4 个队列方法。
- `stop()`（中止当前）→ `finishGeneration({status:'aborted'})` → `pump()` 立即处理下一条（满足「停止当前并立即处理下一条」）。

**4. 渲染层（`useChatQueue.ts` / `useChat.ts` / `ChatView.tsx` / `ChatQueueView.tsx` / `chat.css` / `types.ts`）**
- `useChatQueue.ts`（新）：订阅 `onChatQueue` 维护队列快照 `{items:[{id,content,state}], count}`；暴露 `cancelQueueItem/editQueueItem/clearQueue`（走新 IPC）。`useChat` 组合之并下发给 `ChatView`。
- `useChat.ts` `sendMessage`：**移除 `isStreaming/awaitingResponse` 守卫**，只要 `text+cid+projectId` 即发送（服务端负责入队），乐观把该消息以 `status:'queued'` 加入列表。
- `ChatView.tsx`：`inputDisabled` 去掉 `isStreaming`（流式中输入框仍可用）；`handleSend` 去掉 `isStreaming` 早退；发送按钮恒显，流式时与停止按钮并存；渲染队列项（`排队中/处理中` 徽标）+ 队列计数；每条未处理项提供「取消」「编辑」。
- 队列 UI 与样式抽到 `ChatQueueView.tsx` + `chat.css`（守住文件长度）。
- `types.ts`：`ChatMessage.status` 增加 `'queued'`；新增 `ChatQueueSnapshot/ChatQueueItem`；`AutoplanApi` 增 `onChatQueue` + 4 个队列方法；`WorkspaceChatState` 增 `queue` 与管理动作。

**5. 异常与持久化**
- 生成失败（`error`）/中止（`aborted`）/达上限（`max_rounds`）均经 `finishGeneration` → `pump()` 继续下一条；不阻断队列。
- 渲染层刷新：主进程 controller 存活，队列不丢；应用重启：`loadPersisted()` 从 `chat_messages(status='queued')` 恢复。
- 队列项失败时不删除用户已输入内容（`queued` 行保留直到 `markProcessing` 翻为 `done`；取消才物理删除）。

- [x] P001: 抽取主进程消息队列核心模块 chatQueue.js（FIFO + 状态机 + 持久化 + 管理） <!-- scope: src/chat/chatQueue.js -->

  验收要点：
  - 新建 `src/chat/chatQueue.js`，导出 `createChatQueue({ db, conversationId, projectId, emit })`，返回 `{ enqueue, peekNext, markProcessing, getQueue, cancelItem, editItem, clear, hasQueued, loadPersisted }`。
  - `enqueue(message)`：trim 校验空串返回 `null`；否则向 `chat_messages` 写入 `role='user', status='queued'` 行（复用 `nowIso`），记录到内存 FIFO，返回 `{ id, content, state:'queued' }`。
  - 队列项状态仅 `queued`/`processing`；`peekNext()` 返回 FIFO 首个 `queued` 项（不消费）；`markProcessing(id)` 将该项 `state` 置 `processing` 并 `UPDATE chat_messages SET status='done' WHERE id=?`（使其进入 LLM 上下文）。
  - `getQueue()` 返回 `[{id,content,state}]` 快照（仅 `queued`+`processing`，按入队顺序）；`hasQueued()` 在存在 `queued` 项时为 `true`。
  - `cancelItem(id)`：仅对 `queued` 项生效——从 FIFO 移除并 `DELETE FROM chat_messages WHERE id=? AND status='queued'`；`processing` 项或不存在返回 `false`，不误删。
  - `editItem(id, text)`：仅对 `queued` 项更新 FIFO 内容与 `UPDATE chat_messages SET content=? WHERE id=? AND status='queued'`；非法返回 `false`。
  - `clear()`：移除全部 `queued` 项并 `DELETE FROM chat_messages WHERE conversation_id=? AND project_id=? AND status='queued'`；不影响 `processing` 项与历史。
  - `loadPersisted()`：`SELECT id,content FROM chat_messages WHERE conversation_id=? AND project_id=? AND status='queued' ORDER BY id ASC` 重建 FIFO（用于刷新/重启恢复）。
  - 每个变更方法在成功后调用 `emit(getQueue())`，以便上层广播；模块为纯函数式（无 React/Electron 依赖），可被 `node --test` 单测。
