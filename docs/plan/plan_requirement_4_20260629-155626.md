# 需求 #4 开发计划：任务执行失败重试与递增退避等待

## 背景

AutoPlan 当前任务执行流程（`processPlan` / `runTask` / `executeTaskBatch`）中，当 `executeTask` 返回非零退出码时，任务直接记录失败、不做重试。循环由定时器驱动，下一次 tick 可能重新拾取该 pending 任务，但等待间隔由 `interval_seconds`（默认 5s）决定，而非面向本次失败的有序退避。

本需求为任务执行失败引入**递增退避重试机制**：首次失败等待 5s 重试，第二次等 10s，第三次等 20s，第四次及以后等 30s（上限 30s），最多重试 4 次（即总计 1 次初始执行 + 4 次重试 = 5 次尝试）。全部重试耗尽后任务标记为最终失败。

### 现状关键路径

- **循环自动执行**：`runOnce`（`loopService.js:346`）→ `processPlan`（`taskExecution.js:25`）→ `executeTask` → 成功则 `completeTask`，失败则 `recordTaskFailure` 将任务重置为 `pending`（下次 tick 再试）。重试无延迟、无上限。
- **手动执行**：`runTask`（`loopService.js:412`）→ `executeTask` → 成功则 `completeTask`，失败即止。
- **并发批次**：`runTaskBatches`（`loopService.js:449`）→ `executeTaskBatch`（`taskExecution.js:104`）→ 串行模式（OpenCode）失败即 `break` 停止后续任务，并行模式各任务独立失败。
- **验收重试**：`validatePlan`（`validation.js:66`）已有 2 次修复重试循环（`:147-187`），但无等待延迟。

### 设计约束

- **退避序列固定**：`[5s, 10s, 20s, 30s]`（硬编码常量），首项 5s、上限 30s。调用方可读取该常量但不可运行时修改。
- **重试上限为退避序列长度**（4 次重试 = 5 次总尝试）。全部失败后任务标记为最终失败（`recordTaskFailure`），不再回到 pending 等待下次 tick。
- **退避等待发生在同一次 `processPlan`/`runTask`/`executeTaskBatch` 调用内**，不跨越循环 tick。等待通过 `setTimeout` + Promise 实现（`await new Promise(r => setTimeout(r, ms))`），不阻塞事件循环。
- **退避等待期间 `runtime.busy` 保持为 true**（阻止同项目并发执行），等待可被 `stopProjectRuntime`/`stopRuntimeTask` 中断（kill 子进程 + 清理 timer）。
- **环境阻塞性失败（`environmentBlocked`）不重试**：权限拒绝、命令缺失、超时等环境问题重试无意义，直接标记失败。
- **验收流程（`validatePlan`）已有独立重试机制，本需求不重复改动**；但验收修复循环中 `runCodexWithPlanGuard` 的调用如需重试，可由本需求统一提供的退避工具复用。
- **不回退既有能力**：循环轮询 / `runOnce` / `processPlan` / 任务事件（`taskLifecycleEvent`）/ `on:fail` 钩子 / CLI 后端切换 / MCP 工具链路 / 并发控制 / 脚本模块均不受影响；仅在执行失败路径插入退避重试。
- **不要把运行测试/回归/构建拆成普通开发任务**；最终检查统一放入最后的「完整验收」节点。

- [x] P002: `processPlan` 集成退避重试（循环自动执行路径） <!-- scope: src/loop/taskExecution.js -->
  - 验收要点：修改 `processPlan`（`:35-44`）：在 `executeTask` 返回非零退出码后，不直接放弃，进入退避重试循环。
  - 验收要点：重试逻辑：`let attempt = 0; let result; do { result = await service.executeTask(...); if (result.exitCode === 0) break; attempt++; if (attempt <= TASK_RETRY_BACKOFF_SECONDS.length && !isEnvironmentBlocking(result)) { await sleep(TASK_RETRY_BACKOFF_SECONDS[attempt-1] * 1000); } } while (result.exitCode !== 0 && attempt <= TASK_RETRY_BACKOFF_SECONDS.length && !isEnvironmentBlocking(result));`
  - 验收要点：全部重试耗尽或环境阻塞时，`recordTaskFailure` 照常调用（`:127-131` 的 `executeTask` 内部已处理失败记录与 `on:fail` 钩子），任务不再回到 pending；成功时 `completeTask` 照常调用。

## 总体验收标准

- **验收命令**：`npm run check && npm test`
- **验收范围**：`src/loop/taskExecution.js`（退避工具函数 + processPlan + executeTaskBatch 改动）、`src/loopService.js`（runTask 改动）、新增/补充的测试文件 `src/loop/taskExecution.test.js`、`src/loop/acceptance.test.js`
- **通过标准**：
  1. `npm run check` 零错误（类型检查通过，文件行数在限制内）
  2. `npm test` 全部通过，无失败用例
  3. 退避序列常量 `TASK_RETRY_BACKOFF_SECONDS` 导出为 `[5, 10, 20, 30]`
  4. 环境阻塞性错误（`environmentBlocked: true`）不触发重试
  5. 全部重试（4 次）耗尽后任务标记为最终失败（status 不为 pending）
  6. 成功重试后 `completeTask` 正常调用，任务状态为 completed
  7. 循环自动执行、手动执行、并发批次三条路径均覆盖退避重试
  8. 既有功能（启停、验收、钩子、事件、CLI 后端）无回退
