# 需求 #1 开发计划：脚本模块定时任务、环境变量配置与脚本卡片启动按钮

## 背景

AutoPlan 工作区「执行」分组下的**脚本模块**（`src/renderer/components/workspace/WorkspaceScriptsView.tsx`）以卡片宫格展示自定义脚本，每张卡片目前只有**启用/禁用开关**与**点击打开编辑弹窗**两个入口（`ScriptCard`，`:277`），**没有「直接运行」按钮**——要执行脚本必须先进编辑弹窗点底部「运行」（`ScriptEditorModal.handleRun`，`ScriptEditorModal.tsx:191`，调用 `window.autoplan.runScript`）。运行链路本身是现成的：`scripts:run` IPC（`src/main.js:422`）→ `loop.runScriptManually`（`src/loopService.js:1507`）→ `scriptHooks.runScriptOnce`（`src/loop/scriptHooks.js:92`）→ `service.runShell`（`src/loopService.js:1424`），运行结果回写 `scripts.last_status/last_exit_code/last_duration_ms/last_log/last_run_at` 并记 `script.run.succeeded/failed` 事件。

脚本的**触发方式**当前只有两种：`trigger_mode ∈ {'hook','manual'}`（`src/renderer/types.ts:559`、`src/main.js:328` 的 `SCRIPT_TRIGGER_MODES`）。`hook` 挂到循环五个阶段（`plan:after / task:after / validation:before / loop:end / on:fail`）由 `runHookScripts`（`src/loop/scriptHooks.js:17`）在 `runOnce` 各阶段触发；`manual` 仅靠人工点「运行」。**缺少「按时间计划自动执行」的能力**——即定时任务（cron）。

执行环境方面：脚本与 CLI 任务统一通过 `runShell`/`runCodex` 执行，环境变量由 `workspaceToolEnv(workspace)`（`src/loop/workspaceFiles.js:104`）派生（注入 `.autoplan-runtime` 缓存目录、`CI/PUB_CACHE/XDG_*` 等），脚本另按 `context_inject` 注入 `AUTOPLAN_*` 上下文变量（`scriptHooks.contextEnvVars`，`:254`）。**没有任何「用户自定义环境变量」的配置入口**：设置面板（`WorkspaceSettingsView.tsx`）只有「循环控制 / CLI 后端 / scope 文件 / MCP 接入」四个 pane（`SETTINGS_NAV`，`:19`），用户无法为脚本或 CLI 注入 `MY_TOKEN=xxx` 这类自定义变量。

本需求在三处增量扩展脚本模块：
1. **定时任务**：脚本新增「定时」触发方式，按 5 字段 cron 表达式自动执行（独立于循环是否运行）。
2. **环境变量配置**：设置面板新增「环境变量」pane，按项目维护键值对，注入到脚本与 CLI 执行环境。
3. **卡片启动按钮**：脚本列表卡片新增「启动执行」按钮，无需打开弹窗即可运行/停止。

现状补充（供实现沿用既有风格）：`project_states` 表（`src/database.js:62`）保存每项目循环配置（`interval_seconds/validation_command/agent_cli_*`），`loop.configure`（`src/loopService.js:191`）按 `projectStateColumns()`（`:230`）守卫列后增量更新；快照 `state` 由 `service.status(projectId)` 直接展开（`src/loop/snapshots.js:25`），**新增列会自动随行下发**，无需改查询。`scripts` 表（`src/database.js:184`）用 `ensureColumn` 增量补列（如 `source_type`，`:263`）。`src/main.js` 的脚本 IPC 用 `normalizeScriptFields`（`:354`）+ `SCRIPT_COLUMN_LIST/SCRIPT_SET_ASSIGNMENTS`（`:375`）统一透传列。渲染层 draft 表单集中在 `src/renderer/utils/workspaceForms.ts`（`LoopFormState:84`、`loopFormFromProjectState:225`、`loopConfigurePayloadFromForm:236`；`ScriptDraftState:402`、`createScriptDraft:444`、`scriptCreateInputFromDraft:502`）。控制器 `useWorkspaceController.ts` 持有 `loopForm`/`submitLoopConfig`（`:358`），脚本动作 `toggleScript`/`syncScripts` 在 `WorkspacePage.tsx:210`。设计稿 `ui/scripts.html`、`ui/settings.html` 已存在并镜像对应视图。

## 设计约束

- **定时任务以 cron 表达式驱动，分钟粒度，独立于循环运行态**：脚本 `trigger_mode` 新增 `'schedule'`，配 `schedule_cron`（5 字段标准 cron：`分 时 日 月 周`，支持 `*`、`,`、`-`、`/`）。调度由 `loopService` 持有的**单一全局调度器**（`setInterval`，周期 60s，`timer.unref()` 不阻塞退出）驱动：每 tick 遍历所有项目，筛出 `enabled=1 AND trigger_mode='schedule'` 且 cron 命中当前分钟、且 `last_run_at` 不在本分钟内（避免一分钟内重复触发）的脚本，复用 `scriptHooks.runScriptOnce`（`trigger:'schedule'`）执行，自动回写 `last_*` 并沿用既有 `script.run.succeeded/failed` 事件（meta 带 `trigger:'schedule'`）。定时脚本在**循环停止时也会按计划运行**（定时任务与循环生命周期解耦），但**禁用的脚本（`enabled=0`）绝不触发**；调度器由 `main.js` 创建 `loop` 后调 `loop.startScheduler()`、`will-quit` 调 `loop.stopScheduler()` 启停。cron 求值器为 `scriptHooks` 内新增的纯函数（无外部依赖），无效 cron 表达式在该脚本被选中执行时记一条失败事件、不抛出打断调度器。
- **环境变量按项目存储为 JSON、注入到 runShell 与 runCodex 基线环境**：新增 `project_states.env_vars TEXT`（JSON 数组，如 `[{"name":"FOO","value":"bar"}]`，空串/NULL 视作无变量），经 `ensureColumn('project_states','env_vars','TEXT')` 对既有库幂等迁移。`loopService` 新增 `projectEnvVars(projectId)` 解析为 `{ [name]: value }`（JSON 解析失败安全降级为空对象）。注入点：`runShell` 的 `baseEnv` 与 `runCodex` 传给 `runAgentCliAttempt` 的 `env` 由 `workspaceToolEnv(workspace)` 改为 `{ ...workspaceToolEnv(workspace), ...this.projectEnvVars(projectId) }`——**用户变量覆盖 workspace 派生值，但脚本的 `AUTOPLAN_*`（`operation.extraEnv`）仍优先于用户变量**（保持 `extraEnv` 在合并链最后）。保存走既有 `loop:configure` → `loop.configure`（`config.envVars` 透传，按 `projectStateColumns()` 守卫后写入 `env_vars` 列）。变量值本地明文存 SQLite（与 MCP authToken 同级，均为本机本地数据），快照经 `state` 展开自动下发。
- **卡片启动按钮复用既有运行/停止链路**：`ScriptCard`（`WorkspaceScriptsView.tsx:277`）新增 `onRun` 回调；`WorkspacePage` 新增 `runScript(script)` 动作（命名与 `toggleScript:210` 一致），调 `window.autoplan.runScript({ projectId, scriptId })`（返回 `ScriptRunResult`，含 `snapshot`）后经既有 `syncScripts` 回灌快照。按钮态：非运行时显示「运行」（`play` 图标，复用 `ScriptEditorModal` 已用的 `Icon name="play"`），`last_status==='running'` 时切换为「停止」（`stop` 图标，调 `window.autoplan.stopScript`）。点击运行时 `event.stopPropagation()` 不触发卡片「打开编辑弹窗」；禁用脚本（`enabled=0`）仍可手动运行（运行与启停正交，沿用弹窗「运行」语义）。
- **新增内容尽量复用既有结构，控制文件行数**：`npm run check`（含 `scripts/check-file-lengths.js`）默认单文件上限 600 行。`WorkspaceSettingsView.tsx` 现约 346 行——新增「环境变量」pane 的键值对编辑器若使文件逼近上限，须把「键值对行编辑器」抽为同文件内的小组件（参考既有 `SettingsActions`/`FilterTab` 的内嵌子组件写法），不得外溢超限。`ScriptEditorModal.tsx` 现约 657 行（已超 600？以 `npm run check:lengths` 实测为准；若已超限，本需求的「定时」配置增量须同步精简或下沉工具到 `workspaceForms.ts`），新增「定时」触发段沿用既有 `segment`+`hook-grid` 的栅格样式。`WorkspaceScriptsView.tsx` 的卡片增量（运行按钮 + 定时角标）沿用 `.sc-toggle`/`.sc-trigger` 既有 token。
- **数据模型与类型同步**：`ScriptTriggerMode` 联合补 `'schedule'`；`Script`/`CreateScriptInput` 补 `schedule_cron`（`'schedule'` 时序列化为 cron 串，其它模式为 `null`）；`ScriptDraftState`/`createScriptDraft`/`scriptCreateInputFromDraft`/`validateScriptDraft` 同步补 `scheduleCron` 与 cron 校验。`LoopConfigInput`/`LoopFormState`/`ProjectState` 补 `envVars`（`Array<{name:string;value:string}>`），`loopFormFromProjectState`/`loopConfigurePayloadFromForm`/`loopFormsEqual` 同步序列化/反序列化/相等判定。`src/main.js` 的 `SCRIPT_TRIGGER_MODES` 加 `'schedule'`、`normalizeScriptFields` 透传 `schedule_cron`、`SCRIPT_COLUMN_LIST/SCRIPT_SET_ASSIGNMENTS` 加该列。preload 无新通道（定时字段随 `scripts:create/update`、envVars 随 `loop:configure` 透传），仅类型签名收紧。
- **不回退既有能力**：循环轮询/`runOnce`/任务执行/验收、`hook` 与 `manual` 两种既有触发方式及其 `runHookScripts` 阶段链路、需求/反馈 intake、事件流、CLI 后端（codex/claude/opencode/oh-my-pi）、MCP 工具链路、脚本编辑弹窗的全部既有能力（运行/停止/删除/复制/文件来源/上下文注入）、工作区导航与全局搜索、设计稿均不得回退；`workspaceToolEnv` 的既有派生变量（`.autoplan-runtime`/`CI`/`PUB_CACHE`/`XDG_*` 等）保持不变，用户环境变量仅在其之上叠加。
- **设计稿同步**：`ui/scripts.html` 与 `ui/settings.html` 已存在并镜像视图，本需求对卡片（运行按钮、定时角标）、编辑弹窗（定时触发段）、设置（环境变量 pane）的 UI 改动须同步回这两份设计稿，使其可在浏览器直接预览且与实现视觉一致（沿用既有 `--brand-*`/`--bg-*`/`--text-*`/`--r-*`/`--shadow-*` token 与 `icon-sprite`）。
- **不要把运行测试/回归/构建拆成普通开发任务**；最终检查统一放入最后的「完整验收」节点。

- [x] P002: 定时任务调度后端（cron 求值 + 全局调度器 + 复用 runScriptOnce + 事件） <!-- scope: src/loop/scriptHooks.js,src/loopService.js,src/loop/runtime.js -->
  - 验收要点：`src/loop/scriptHooks.js` 新增纯函数 cron 求值器（无外部依赖）：`parseCron(expr)` 校验并解析 5 字段标准 cron（`分 时 日 月 周`，支持 `*`、`,`、`-`、`/` 步长），非法表达式抛中文错误（如「cron 表达式格式无效」）；`isCronDue(parsed, date)` 判定给定时间是否命中；`dueScheduledScripts(scripts, now)` 综合判定：`enabled=1 && trigger_mode='schedule' && schedule_cron 合法 && isCronDue(now) && last_run_at 不在 now 所在分钟内`（用 `last_run_at` 去重避免同分钟重复触发）。三者导出供测试与调度器复用（沿用 `module.exports` 风格）。
  - 验收要点：`src/loopService.js` 新增调度器：`startScheduler()` 创建单一 `setInterval`（周期 60000ms，`timer.unref()` 不阻塞进程退出）存于实例（如 `this.scheduleTimer`），每 tick 调 `this.runScheduledScripts()`；`stopScheduler()` 清除该 timer（幂等）；`runScheduledScripts()` 遍历 `this.projects()`，对每个项目查 `SELECT * FROM scripts WHERE project_id=? AND enabled=1 AND trigger_mode='schedule'`，经 `dueScheduledScripts` 筛选后逐条 `await scriptHooks.runScriptOnce(this, script, script.hook_stage||'schedule', { trigger:'schedule', workspace })`——单条异常被 `runScriptOnce`/`recordRunFailure` 兜底，绝不冒泡打断调度器或同 tick 其它脚本；调度器在**循环停止时仍运行**（不依赖 `runtime.running`）。
  - 验收要点：定时执行复用既有落库与事件：经 `runScriptOnce`→`recordScriptRun` 回写 `last_status/last_exit_code/last_duration_ms/last_log/last_run_at`，记 `script.run.succeeded/failed` 事件且 meta 的 `trigger` 为 `'schedule'`（沿用 `scriptHooks.recordScriptRun:282` 既有 meta 结构）；非法 `schedule_cron` 的脚本被选中时记一条 `script.run.failed`（或 `script.schedule.error`）事件、`last_status='bad'`，不抛错、不影响同 tick 其它脚本与后续 tick。
  - 验收要点：`src/loop/runtime.js` 若需为调度器提供共享 helper（如统一的 interval/timer 句柄工具），沿用 `scheduleProjectRuntime`（`:142`）的 `setInterval`+`unref` 写法新增；不改动 `scheduleProjectRuntime`/`createProjectRuntime`/`stopProjectRuntime` 等既有循环运行态逻辑，循环轮询/`runOnce`/任务执行/`hook`+`manual` 触发链路无回退。
  - 验收要点：调度器与既有运行态正交：定时脚本经 `runShell` 注册到该项目 `runtime.activeOperations`（与手动运行一致），可被 `stopScript` 停止；不占用 `runtime.busy` 锁，不阻塞循环或手动运行；`runScheduledScripts` 对无定时脚本项目为空操作，开销可忽略。
