# 反馈 #10 开发计划：验收命令默认空、单条需求/反馈 CLI 配置与 Codex 思考深度

## 背景

当前 AutoPlan 已支持项目级 CLI 后端选择，并通过验收命令完成任务闭环。但反馈 #10 指出三个需要细化到输入与执行链路的行为：验收命令默认应为空；每条需求、每条反馈在输入框下方都应能单独选择 CLI 后端（Codex/Claude）；Codex 后端还应支持切换思考模式深度。目标是在不破坏项目级默认配置的前提下，为单条需求/反馈提供可覆盖的执行配置，并让计划生成、任务执行、修复与验收能稳定使用该配置。

## 设计约束

- 验收命令默认值必须为空字符串；历史已有命令的项目不应被清空，新建项目与初始化状态不再自动填入示例命令。
- 项目级 CLI 配置仍作为默认值；需求/反馈未单独配置时继承项目级配置，单独配置后优先使用单条记录配置。
- 需求输入框与反馈输入框下方都要展示 CLI 选择控件，交互位置、样式和保存行为保持一致。
- CLI 后端至少支持 `codex` 与 `claude`；非法值需要归一化为默认 Codex，避免进入执行链路。
- Codex 思考深度仅对 Codex CLI 生效；选择 Claude 时该控件应隐藏、禁用或明确提示不适用。
- 思考深度需要有明确枚举值与默认值，执行命令参数、日志展示和快照数据保持一致。
- 不应影响附件、粘贴图片、正文提交、计划任务 scope 解析、任务并发与 existing progress 更新链路。

## 总体验收标准

- 验收命令默认空：新建项目和初始项目状态不自动填入任何验收命令，空命令不会触发外部命令执行失败。
- 单条配置优先：每条需求、每条反馈都可以在输入框下方选择 Codex/Claude，提交后计划生成与后续任务执行优先使用该条配置。
- 默认继承稳定：单条需求/反馈未配置 CLI 时继承项目级默认配置，项目级配置变化不会破坏历史计划的执行一致性。
- Codex 深度生效：Codex 后端可切换思考深度，执行命令实际携带正确参数，日志可追踪实际深度。
- 后端隔离明确：Codex session 仅用于 Codex，Claude 调用不复用 Codex 会话；不同记录的 CLI 配置不会交叉污染。
- UI 交互无回归：正文、附件、粘贴图片、发送按钮、列表刷新和计划卡片展示均保持现有可用性。
- 构建验证通过：实现完成后 `npm run check`、`npm run build` 和关键 smoke/手动场景验证通过。
