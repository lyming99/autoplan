# 需求 #24 开发计划：从 GitHub 自动检查正式版本更新

- 需求编号：#24
- 需求标题：从 GitHub 自动检查正式版本更新
- 需求原文：能否实现从github自动检查正式版本更新？
- 创建时间：2026-07-01 05:45:10
- 当前应用版本：0.2.2（`package.json`）
- 发布仓库：`lyming99/autoplan`（origin: https://github.com/lyming99/autoplan.git）

## 背景与现状分析

AutoPlan 是 Electron 桌面应用，通过 GitHub Actions + electron-builder 发布 Windows/macOS/Linux 三端产物，tag 规范为 `vX.Y.Z`（稳定正式版）与 `vX.Y.Z-beta.N`（预发布版，见 README「发布 / Beta 版本说明」）。当前应用内**没有任何版本更新检查机制**：代码库全局搜索 `releases/latest`、`electron-updater`、`autoUpdater`、`checkForUpdate`、`getVersion` 均无命中，`package.json` 也不依赖 `electron-updater`。

需求要求「自动检查**正式版本**更新」，核心诉求是：

1. 应用启动后（以及按一定间隔）自动查询本仓库 GitHub 上的最新**正式版**（非 prerelease、非 draft），与本地 `app.getVersion()` 比较。
2. 当存在比本地更新的正式版时，向用户**提醒**并提供跳转到 Release 页面的入口。
3. 用户可控：可开关自动检查、可手动触发「立即检查」、可忽略当前版本（本轮不再提醒）。

工程判断与边界：

- **只检查、不自动下载安装**：AutoPlan 三端产物需 Windows 代码签名与 macOS Developer ID 签名 + Apple 公证（README「发布」），自动替换二进制会引入签名/公证与进程提权等高风险问题。本期只做「检测 + 提醒 + 跳转」，不做静默升级，符合最小风险的最优方案。
- **正式版判定**：直接使用 GitHub REST API 的 `GET /repos/lyming99/autoplan/releases/latest`，该端点官方语义即「最新的非 prerelease、非 draft Release」，天然等同于「正式版本」，无需自行过滤。同时在解析层保留 `prerelease`/`draft` 防御性过滤，避免回退到 `/releases` 列表端点时把 beta 误判为正式版。
- **本地为 beta 版本的比较**：本地可能是 `0.2.1-beta.6` 这类预发布版，比较时需按语义版本正确处理 prerelease 段（`0.2.1-beta.6 < 0.2.1`），保证「beta → 同号稳定版」也正确提示。
- **网络与限流**：GitHub 未鉴权请求限流 60 次/小时，启动后单次检查 + 默认 6 小时间隔远低于上限；请求须带 `User-Agent` 头（GitHub 强制）。使用 Electron `net` 模块（`src/main.js` 已导入 `net`），可正确复用系统代理。

## 方案设计

### 数据流

```
src/updateChecker.js（核心：解析/比较/抓取/调度）
        │  依赖 db.getSetting/setSetting 持久化 update.* 设置
        │  依赖 app.getVersion() 取本地版本
        ▼
src/main.js（实例化 checker、启动/停止调度、注册 IPC、检查后向渲染进程推送 updates:status）
        │  ipcMain.handle('updates:*')
        ▼
src/preload.js（暴露 window.autoplan.updateStatus / checkForUpdates / dismissUpdate / setAutoUpdateCheck / onUpdateStatus）
        ▼
src/renderer（useUpdateStatus 订阅 → UpdateNotice 横幅 + 设置页「关于/更新」面板）
```

### 持久化设置（`settings` 表，前缀 `update.`）

| key | 默认值 | 说明 |
| --- | --- | --- |
| `update.autoCheck` | `true` | 是否启用自动检查 |
| `update.intervalMinutes` | `360` | 自动检查间隔（分钟，默认 6 小时） |
| `update.lastCheckedAt` | `` | 上次成功检查的 ISO 时间戳 |
| `update.dismissedVersion` | `` | 用户已忽略的最新正式版本号，命中则本轮不再弹横幅 |

### UI 落点

- **全局提醒横幅** `UpdateNotice`：在项目列表页 `ProjectsPage` 与工作区 `WorkspacePage` 均渲染，有更新时置顶展示「新正式版本 vX.Y.Z 可用」，提供「前往下载」（`shell.openExternal` 打开 Release `html_url`）与「稍后提醒」（写 `update.dismissedVersion`）。
- **设置面板新增「关于/更新」**：在 `WorkspaceSettingsView` 的 `SETTINGS_NAV` 增加 `about` 面板，展示当前版本、最新正式版、上次检查时间、自动检查开关、立即检查按钮、打开 GitHub Releases 链接。
- 所有外链跳转经主进程 `shell.openExternal`，不在渲染进程直接发网络请求。

## 任务拆解

- [x] P001: 新增正式版本检查核心模块（版本解析/比较/正式版判定/抓取调度） <!-- scope: src/updateChecker.js -->
    - 验收要点：
      - 导出纯函数 `parseVersion(tag)`：兼容 `v0.2.2`、`0.2.1-beta.6`、无前缀、含 build 元数据等输入，返回结构化 `{ major, minor, patch, prerelease }`；非法输入返回 `null` 而非抛错。
      - 导出 `compareVersions(a, b)`：返回 `-1/0/1`，正确处理 prerelease 段语义（如 `0.2.1-beta.6 < 0.2.1`），保证「beta → 同号稳定版」视为有更新。
      - 导出 `parseLatestRelease(json)`：从 GitHub Release JSON 提取 `version`（去前导 `v`）、`name`、`htmlUrl`、`publishedAt`、`body` 摘要、`isPrerelease`；对 `draft`/`prerelease=true` 的 Release 视为非正式版并标记。
      - 导出 `hasStableUpdate(currentVersion, release)`：仅当 release 为非 prerelease、非 draft 且版本号严格大于本地版本时返回 `true`。
      - 导出 `createUpdateChecker({ app, net, db, repo, fetch })`：`check()` 通过 `net.request` 请求 `https://api.github.com/repos/${repo}/releases/latest`，带 `User-Agent: autoplan/<version>` 与 `Accept: application/vnd.github+json` 头；超时（如 15s）、非 2xx、JSON 解析失败均返回结构化 `{ ok:false, error }`，不抛崩、不影响 GUI。
      - `check()` 成功后用 `db.setSetting` 写入 `update.lastCheckedAt`（ISO 时间）与最新正式版号缓存；`status()` 读取缓存与设置拼装当前状态；`dismiss(version)` 写 `update.dismissedVersion`；`setAutoCheck(enabled)` 切换 `update.autoCheck` 并即时重排调度。
      - `start()/stop()`：按 `update.intervalMinutes` 周期触发；`autoCheck` 关闭时不启动定时器。
      - 抓取与解析解耦：`parseLatestRelease`/`compareVersions`/`hasStableUpdate` 不触网，便于单测。

    - 验收要点：

      - 读写统一走 `getSetting/setSetting/getSettings('update.')`，与 `mcpConfig` 既有模式一致。

- [x] P003: 主进程接入更新检查调度与 IPC <!-- scope: src/main.js -->
    - 验收要点：
      - 在 `createApp()` 中实例化 `createUpdateChecker({ app, net, db, repo: 'lyming99/autoplan' })`；`app.whenReady`/`createApp` 完成后，若 `update.autoCheck` 开启则延迟触发首次检查并启动周期调度。
      - 在 `before-quit` 中调用 `checker.stop()`，与现有 `loop.stop()`、`mcpServer.stop()` 并列，确保退出不泄漏定时器。
      - 注册 IPC：`updates:status`（返回当前状态：当前版本、最新正式版、是否有更新、上次检查时间、autoCheck、html_url）、`updates:check`（手动触发一次检查并返回结果）、`updates:dismiss`（写 `update.dismissedVersion`）、`updates:setAutoCheck`（切换并重排调度）。
      - 检查完成后通过 `mainWindow.webContents.send('updates:status', payload)` 推送，推送前复用现有 `mainWindow && !mainWindow.isDestroyed()` 存活判断，避免向已销毁窗口发送。
      - repo 来源以常量定义；当前版本取 `app.getVersion()`；手动检查与定时检查并发安全（进行中再次触发直接复用进行中的 Promise，避免重复请求）。

- [x] P004: preload 暴露更新检查 API <!-- scope: src/preload.js -->
    - 验收要点：
      - 在 `contextBridge.exposeInMainWorld('autoplan', {...})` 中新增：`updateStatus()` → `ipcRenderer.invoke('updates:status')`、`checkForUpdates()` → `invoke('updates:check')`、`dismissUpdate(input)` → `invoke('updates:dismiss', input)`、`setAutoUpdateCheck(enabled)` → `invoke('updates:setAutoCheck', { enabled })`。
      - 新增 `onUpdateStatus(handler)`：监听 `updates:status` 事件并返回取消订阅函数，风格与现有 `onLoopUpdate` 完全一致。
      - 不破坏既有 API；不暴露任何 Node/Electron 内部对象。

## 总体验收标准

- **最终验收命令**：
  - `npm run check` —— TypeScript 类型检查 + 主进程/预加载/关键脚本语法检查 + 文件长度规范。
  - `npm test` —— `node --test` 运行全部 `src/**/*.test.js`，含新增 `src/updateChecker.test.js`。
  - `npm run smoke` —— 核心流程冒烟测试，含 `updates:*` IPC stub。
  - `npm run build` —— `tsc --noEmit && vite build` 渲染进程构建。
- **验收范围**：更新检查核心模块、`settings` 持久化、主进程调度与 IPC、preload 暴露、渲染端横幅与设置面板、文档；不包含真实 GitHub 网络连通性的自动化验证（留作手动验收）。
- **通过标准**：
  1. 上述四条命令均退出码 0、无报错。
  2. 启动应用后，自动检查默认开启并能查询到 `lyming99/autoplan` 最新正式版；有更新时全局横幅可见且可跳转/可忽略。
  3. 「设置 → 关于/更新」面板可手动检查、可切换自动检查、可打开 Releases 链接；当前版本显示正确。
  4. 网络异常或 GitHub 限流时应用不崩溃、不影响既有需求/计划/循环/MCP 功能。
  5. 不新增运行时第三方依赖（仅用 Electron 内置 `net` 与 Node 内置能力）。
