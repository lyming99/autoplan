# 开发计划：修复 OpenCode 生成需求计划失败

## 背景

需求 #3 正文仅"大纲视图优化一下"7 个字。OpenCode 进程 exit 0，但：
1. 未写出 plan 文件（模型把计划写进对话 stdout，没执行落盘动作）→ planGeneration.js:134 判定失败
2. 模型反问澄清（正文太短 + prompt 引导不足，模型回复"请告诉我你希望我…"）
3. 失败后无退避，linked_plan_id 始终为 NULL → loop 每轮重试，30 分钟内失败 6 次

## 涉及文件

- `src/loop/planGeneration.js` — prompt 强化、stdout 兜底落盘、短正文上下文注入、失败计数更新
- `src/loopService.js` — 扫描 SQL 追加退避条件
- `src/database.js` — requirements/feedback 表加 generate_fail_count / last_generate_fail_at 列
- `src/loop/planGeneration.test.js` — 新增测试

- [x] P002: 实现 stdout 兜底落盘 recoverPlanFromStdout <!-- scope: src/loop/planGeneration.js -->
  - 在 `generatePlanForIntake` 函数内，`if (result.exitCode !== 0 || !fs.existsSync(planFile))` 判定之前，插入兜底逻辑：
    - 当 `result.exitCode === 0 && !fs.existsSync(planFile)` 时，调用 `recoverPlanFromStdout(planFile, result.output)` 尝试从 stdout 提取 plan 内容
    - 提取策略：优先取 stdout 中首个 `## ` 二级标题到结尾的内容（去掉前面对话式寒暄）；若找不到任何 `## `，对整段 stdout 轻量清理后整体落盘
    - 写入 planFile 后继续走正常成功分支（insertPlan / syncPlanTasks），并 appendInternalLog 记录"已从 stdout 兜底落盘"
    - 若 stdout 也无法提取出有效内容（空或仅寒暄），才走原失败分支
  - 新增模块内私有函数 `recoverPlanFromStdout(planFile, stdout)`，返回布尔值
  - 验收要点：
    - stdout 包含以 `## 任务拆解` 开头的内容时，正确切取并写入 planFile，函数返回 true
    - stdout 仅含对话式寒暄（无任何 `## ` 标题）时，函数返回 false，不写入文件
    - stdout 为空字符串时，函数返回 false
    - 兜底成功后 appendInternalLog 包含"已从 stdout 兜底落盘"记录
    - 兜底成功后正常走 insertPlan / syncPlanTasks 分支

- [x] P005: 扫描 SQL 退避条件与失败计数更新 <!-- scope: src/loopService.js,src/loop/planGeneration.js -->
  - 在 `loopService.js` 的 `runOnce` 扫描 SQL（当前 346-363 行）中，为 requirements 和 feedback 查询各追加条件：
    ```sql
    AND (generate_fail_count < 3 OR (generate_fail_count >= 3 AND last_generate_fail_at < datetime('now','-15 minutes')))
    ```

    ```sql

    ```
  - 在成功分支（insertPlan 之后）中，对对应 intake 表执行 UPDATE 清零：
    ```sql
    UPDATE ${table} SET generate_fail_count = 0, last_generate_fail_at = NULL, updated_at = ? WHERE id = ?
    ```

    - 连续失败 3 次后，第 4 次扫描时该 intake 不出现在 pendingIntakes 中

    - 失败时 fail_count 自增，last_generate_fail_at 记录为当前时间
