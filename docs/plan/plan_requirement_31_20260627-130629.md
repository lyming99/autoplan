# 需求 #31 开发计划：计划列表排序、草稿计划与状态 Tab

## 需求概述

需求 #31 要求计划列表支持调节执行顺序；计划支持草稿状态，并可在对话创建计划时勾选“草稿”来生成草稿计划；计划列表需要参考现有任务列表的交互方式，支持按“全部 / 执行中 / 已完成 / 草稿”进行 Tab 切换。该需求涉及计划数据模型、计划生成流程、IPC/API 暴露、前端计划列表交互和状态展示。

## 设计约束

- 计划执行顺序应持久化，刷新页面、重启应用后仍保持用户调整后的顺序。
- 草稿计划不得被自动循环或批量执行误触发，必须由用户显式转为可执行状态或显式执行后才进入执行流。
- 计划状态 Tab 只影响列表筛选和展示，不应隐藏任务列表已有的“全部 / 执行中 / 已完成”能力。
- 计划排序应优先复用现有 `plans`、`plan_tasks.sort_order` 和 snapshot 流程，避免引入与任务排序冲突的并行状态源。
- 对话区“草稿”勾选项应与需求/反馈的计划创建入口保持一致，避免只在某一种 intake 类型上生效。
- 完整验收节点只负责最终验收，不承载普通开发实现。

## 总体验收标准

- 计划列表必须支持用户调节执行顺序，且顺序持久化到本地数据库，刷新和重启后不丢失。
- 通过对话创建计划时必须能勾选“草稿”，勾选后生成的计划状态为 `draft`，且默认不参与自动执行。
- 计划列表必须提供“全部 / 执行中 / 已完成 / 草稿”Tab，分类数量和筛选结果与计划真实状态一致。
- 草稿计划必须能被用户显式启动，并在执行、完成、失败、停止等状态流转中保持与普通计划一致的事件和 UI 反馈。
- 最终验收命令为 `npm run typecheck`、`npm test`；若脚本不存在，则以 `npm run build` 作为最低验收命令并说明原因。
- 通过标准为所有可用验收命令成功，手工验证覆盖普通计划、草稿计划、排序调整、Tab 筛选和既有任务列表无回归。
