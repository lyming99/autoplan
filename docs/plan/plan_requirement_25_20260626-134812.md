# 需求 #25 开发计划：计划并发执行、并发建议与 scope 文件打开

## 背景

当前计划任务已经通过固定格式记录 `scope`，并在任务执行时用于判断任务影响范围。新需求要求把这份结构化信息进一步产品化：计划列表需要展示并发执行建议和入口，用户可手动确认后触发并发执行；计划任务项需要展示从 `scope` 解析出的相关文件列表；计划阅读全文时，`scope` 中的文件路径可点击打开，并允许通过配置选择系统文件夹、VSCode 或第三方编辑器等打开方式。

## 设计约束

- 并发执行必须以计划任务的 `scope` 为核心依据：互不冲突的任务可进入同一并发批次，包含 `unknown`、`validation`、空 scope 或 scope 重叠的任务不得被自动并发到同一批次。
- 并发执行建议只负责给出安全分组和原因，真正执行必须由用户在计划列表项点击并发执行按钮后触发，不得在生成建议后自动启动。
- 并发任务仍需复用现有单任务执行、停止、重试、状态更新、日志记录和进度同步能力，避免新增一套与现有执行链路分叉的任务执行逻辑。
- 计划任务相关文件列表必须来源于任务行的 `<!-- scope: ... -->`，支持多个 scope、去重、规范化路径和不存在文件的可读提示。
- 计划阅读全文中的 scope 文件链接必须限制在项目工作区内，禁止通过相对路径或绝对路径绕过工作区边界。
- 文件打开方式应为项目级或应用级配置，默认使用系统打开方式；VSCode 和第三方编辑器配置缺失、命令不存在或打开失败时，需要给出可追踪错误提示。
- UI 需要清晰区分“并发建议”“可并发批次”“需串行任务”“完整验收任务”，避免用户误把完整验收节点加入普通并发开发批次。

## 并发执行建议

- 第一批建议并发：P001、P005。P001 负责数据模型与后端建议能力，P005 负责文档说明，scope 无重叠，可并行推进。
- 第二批建议并发：P002、P003、P004。P002 负责并发执行调度，P003 负责计划列表 UI，P004 负责 Markdown 阅读器文件链接与打开配置；三者需在 P001 的数据结构和 IPC 契约明确后并发。
- P006 需在 P001-P004 完成后执行，用 smoke 覆盖并发建议、手动并发执行和文件打开链路。
- P007 为完整验收，必须最后执行，不参与开发批次并发。

## 总体验收标准

- 功能完整性：计划列表能基于任务 scope 展示并发执行建议，用户确认后才能手动并发执行安全批次，任务状态、日志和计划进度可追踪。
- 安全性：`unknown`、`validation`、空 scope、scope 重叠任务不会被建议自动并发；scope 文件点击打开必须限制在项目工作区内。
- 可配置性：文件打开方式支持系统默认/系统文件夹、VSCode 和第三方编辑器命令配置，默认配置可用，失败提示可读。
- 可读性：计划任务项和计划阅读全文均能展示 scope 相关文件，长路径、缺失文件、不可打开文件和普通 Markdown 链接互不干扰。
- 最终验收命令：执行 `npm run check` 和 `npm run smoke`；两条命令均退出码为 0，且手动 UI 验收覆盖计划列表并发入口、并发建议确认、scope 文件点击打开三类核心流程。
