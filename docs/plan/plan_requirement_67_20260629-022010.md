# 需求 #67 开发计划：脚本模块支持「选择文件」作为脚本来源

## 背景

需求原话：「脚本模块，创建脚本时，要支持选择文件作为脚本」。当前 AutoPlan 的脚本模块在创建脚本时**只能在内联代码编辑器里手写脚本内容**，无法直接引用工作区中已有的脚本文件。

现状梳理（结合代码）：

1. **脚本数据模型已具备 `path` 与 `body` 两列，但 `path` 仅用于展示、从不参与执行**。`scripts` 表（`src/database.js:182`）含 `path TEXT NOT NULL DEFAULT ''`（`database.js:186`）与 `body TEXT NOT NULL DEFAULT ''`（`database.js:188`）。渲染层的 `scriptPath`（`src/renderer/components/workspace/WorkspaceScriptsView.tsx:61`）与弹窗的 `derivedPath`（`src/renderer/components/workspace/ScriptEditorModal.tsx:124`）都把 `path` 当作「展示路径提示」，为空时回退到 `scripts/<name><ext>`；而真正执行时 `executeScriptProcess`（`src/loop/scriptHooks.js:120`）**始终把 `script.body` 写进临时文件再运行**（`writeTempScript`/`buildScriptCommand`，`scriptHooks.js:150`/`scriptHooks.js:169`），完全无视 `path`。因此即便用户填了 `path`，运行时也不会执行该文件。

2. **创建/更新链路已能透传 `path`，但渲染层表单从未暴露该字段**。`main.js:341` 的 `normalizeScriptFields` 已接收 `path`（`main.js:346`）并写入 `SCRIPT_COLUMN_LIST`/`SCRIPT_SET_ASSIGNMENTS`（`main.js:361`）；但 `ScriptDraftState`（`src/renderer/utils/workspaceForms.ts:398`）与 `createScriptDraft`（`workspaceForms.ts:432`）均无 `path`/来源相关字段，`scriptCreateInputFromDraft`（`workspaceForms.ts:485`）也不下发 `path`——也就是说前端目前没有任何入口设置 `path`，也无法区分「内联代码」与「文件来源」。

3. **没有「文件选择」IPC**。现有选择类交互只有目录选择：`projects:pickDirectory`（`src/main.js:124`，`dialog.showOpenDialog(mainWindow, { properties: ['openDirectory'] })`），preload 以 `pickDirectory` 暴露（`src/preload.js:63`）。新增「选择文件」可完全沿用这一范式（`properties: ['openFile']` + 扩展名 filters），无需新机制。

4. **既有脚本执行链路成熟可复用**。`runShell` 提供超时/日志文件/operation 取消能力；`buildScriptCommand(runtime, file)`（`scriptHooks.js:169`）已能按 runtime（node/bash/ps/cmd）拼出执行命令；`resolveWorkDir`（`scriptHooks.js:192`）已支持 `${workspace}`/`${planDir}` 占位与相对工作区解析。文件来源脚本只需把「临时文件」替换为「用户选定的真实文件」，命令拼装、超时、日志、事件、取消逻辑全部不动。

5. **快照为 `SELECT *`，新增列自动透传**。`src/loop/snapshots.js:114` 以 `SELECT * FROM scripts` 下发，新增 `source_type` 列后无需改动快照查询，渲染层即可读到。

综上，本需求的核心工作为：在数据层新增 `source_type`（`inline`/`file`）列以区分来源；在执行层按来源分流（文件来源直接运行用户选定文件、内联来源维持写临时文件）；新增 `scripts:pickFile` 文件选择 IPC；在渲染层补齐类型、draft 表单的来源态与序列化/校验，并在详情弹窗提供「内联代码 / 选择文件」切换与文件选择 UI、在列表卡片标注来源。

## 设计约束

- **新增 `source_type` 列区分来源，复用既有 `path` 列承载文件路径，不新增数据表**：以 `ensureColumn('scripts', 'source_type', "TEXT NOT NULL DEFAULT 'inline'")`（沿用 `database.js:230` 的 `ensureColumn` 迁移范式）给 `scripts` 表加列，取值 `'inline'`（执行 `body`，现状行为）/`'file'`（执行 `path` 指向的真实文件）。**文件路径复用既有 `path` 列**：`source_type='file'` 时 `path` 为用户选定文件的路径；`source_type='inline'` 时 `path` 维持原展示提示语义（通常为空）。不引入额外列，避免 schema 冗余。
- **执行分流以 `source_type` 为准，文件来源直接运行原文件、不拷贝**：`executeScriptProcess`（`scriptHooks.js:120`）读取 `script.source_type`（缺省/`'inline'` 走既有 `writeTempScript`+`buildScriptCommand`）；`'file'` 时用新增 `resolveScriptFile(script.path, { workspace, planDir })` 解析路径（镜像 `resolveWorkDir` 的 `${workspace}`/`${planDir}` 占位展开 + 相对工作区解析 + 绝对路径直通），校验文件存在后直接 `buildScriptCommand(runtime, resolvedFile)` 运行**原文件**（用户后续编辑该文件即时生效，这正是「选择文件作为脚本」的意义），**不再写临时副本**；文件不存在/路径非法时返回 `{ exitCode: -1, errorMessage: '脚本文件不存在：…' }` 并记失败事件，**不得向上抛错**（沿用 `recordRunFailure` 不冒泡原则）。
- **runtime 仍决定解释器，文件扩展名不强制校验**：文件来源脚本由 `script.runtime`（node/bash/ps/cmd）决定用哪个解释器运行该文件（与内联来源一致），不按扩展名推断、不因扩展名不匹配而报错；文件选择对话框的 filters 仅作为便捷过滤，始终保留「所有文件」兜底，避免误拦。
- **文件选择走原生 `dialog`、返回绝对路径，存储为原值但支持占位编辑**：`scripts:pickFile`（镜像 `projects:pickDirectory`，`main.js:124`）以 `dialog.showOpenDialog(mainWindow, { properties: ['openFile'], filters })` 返回用户选定文件的绝对路径字符串（取消返回 `null`）。落库存「对话框返回的原值」（通常绝对路径）；用户也可手动编辑为 `${workspace}/...` 相对形式，执行时由 `resolveScriptFile` 统一解析——两栖兼容。
- **创建与编辑共享来源切换（弹窗新建/编辑共用 `ScriptEditorModal`）**：需求虽只提「创建脚本时」，但 `ScriptEditorModal`（`ScriptEditorModal.tsx:87`）新建与编辑共用，来源态作为脚本固有属性在两态下都可读/写/切换，避免「创建时能选文件、编辑时不能改」的割裂。
- **IPC 命名与校验沿用既有风格**：新增通道 `scripts:pickFile`；`scripts:create`/`scripts:update` 的字段归一化（`normalizeScriptFields`）与列清单（`SCRIPT_COLUMN_LIST`/`SCRIPT_SET_ASSIGNMENTS`）补 `source_type`；preload 在 `window.autoplan` 上以一行式 `ipcRenderer.invoke` 暴露 `pickScriptFile`（命名与既有 `pickDirectory`/`createScript` 一致），失败/取消返回 `null` 不抛错。
- **文件长度约束（关键风险）**：`npm run check` 含 `scripts/check-file-lengths.js`，受影响文件接近上限——`src/renderer/types.ts` 977/980、`ScriptEditorModal.tsx` 586/600、`src/main.js` 789/800。本计划单独设一个任务（P002）**先行调高这三处的配置上限**（沿用该配置文件已对 `agentCli.js`/`loopService.js`/`types.ts` 等做的 per-file 覆盖范式，并补注释说明原因），其余任务实现时不触碰 `check-file-lengths.js`，避免并发改同一文件。
- **既有链路不得回退**：内联来源脚本的临时文件执行、超时/日志/取消、事件流、钩子触发（`runHookScripts`）、手动运行（`runScriptManually`）、停止（`stopScript`）、启用开关、会话恢复、MCP/项目/需求/计划/任务链路与工作区导航均保持不变；无 `source_type` 的历史脚本默认 `'inline'`，行为与现状完全一致。
- **不要把运行测试/回归/构建拆成普通开发任务**；最终检查统一放入最后的「完整验收」节点。
