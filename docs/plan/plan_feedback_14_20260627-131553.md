# 反馈 #14 开发计划：修复 macOS 发布包提示损坏

## 背景

反馈 #14 指出：macOS 下载 GitHub Action 编译出的安装包后提示“已损坏/无法打开”，怀疑发布流水线存在路径、格式或打包问题。结合当前发布配置，macOS 包由 GitHub Actions 的 `macos-latest` 执行 `npm run package:mac` 生成 `dmg` 和 `zip`，且流水线设置了 `CSC_IDENTITY_AUTO_DISCOVERY=false`，这会显式禁用自动签名发现。该问题高概率与 macOS Gatekeeper 对未签名、未公证、签名不完整或下载产物损坏的拦截有关，也需要排查产物上传/下载路径、架构产物命名和 release 附件格式是否导致用户拿到错误文件。

## 设计约束

- 本计划只处理 macOS 发布包“提示损坏/无法打开”的构建、签名、公证、产物校验和发布说明问题，不改动业务功能逻辑。
- 优先修复根因：签名、公证、entitlements、artifact 上传路径和 release 产物完整性；不要仅依赖用户执行 `xattr -dr com.apple.quarantine` 作为正式解决方案。
- 若项目暂时无法配置 Apple Developer 证书和公证密钥，必须让 CI 明确区分“未签名测试包”和“正式签名公证包”，避免用户误以为未签名包可直接安装。
- macOS x64 与 arm64 产物必须能清晰区分，避免 GitHub Release 中同名覆盖、下载错架构或 zip/dmg 混用导致误判为损坏。
- 不要把运行测试、构建、回归验证拆成普通开发任务；最终验证统一放到最后“完整验收”节点。

## 任务拆解

- [x] P001: 定位 macOS 损坏提示根因 <!-- scope: .github/workflows/release.yml,package.json,docs/release/macos.md -->
  - 验收要点：梳理 macOS 包从 `npm run package:mac`、`electron-builder` 输出、artifact 上传、release 上传到用户下载的完整链路。
  - 验收要点：确认当前 `CSC_IDENTITY_AUTO_DISCOVERY=false` 对 macOS 签名的影响，并判断“提示损坏”是否由未签名/未公证、签名破坏、下载产物不完整或架构不匹配触发。
  - 验收要点：记录需要保留的发布产物类型和命名规则，包括 `dmg`、`zip`、x64、arm64 以及 `latest-mac.yml` 的关系。
  - 验收要点：形成排查结论文档，明确后续任务应修复的配置点，不修改业务代码。

- [x] P002: 完善 macOS 签名与公证配置 <!-- scope: package.json,build/entitlements.mac.plist,build/entitlements.mac.inherit.plist,scripts/notarize-mac.js -->
  - 验收要点：`electron-builder` 的 macOS 配置包含明确的 hardened runtime、entitlements、gatekeeper assess 以及签名/公证所需字段。
  - 验收要点：新增或更新 entitlements 文件时，仅开放 Electron 应用实际需要的权限，不引入过宽授权。
  - 验收要点：公证脚本支持从 CI 环境变量读取 Apple ID、Team ID、app-specific password 或 App Store Connect API key，缺失时给出明确错误或跳过原因。
  - 验收要点：正式发布包完成签名与公证后，Gatekeeper 不应再把包识别为损坏；未配置证书时不得伪装成正式可安装包。

- [x] P003: 修正 GitHub Actions macOS 发布流程 <!-- scope: .github/workflows/release.yml -->
  - 验收要点：macOS job 明确注入签名和公证所需 secret，并只在 secret 完整时执行正式签名公证发布路径。
  - 验收要点：移除或条件化会阻断 macOS 正式签名的 `CSC_IDENTITY_AUTO_DISCOVERY=false`，避免 CI 永久生成未签名包。
  - 验收要点：CI 日志能区分签名、公证、staple、artifact 上传阶段，失败时可定位到具体阶段。
  - 验收要点：Windows/Linux 的打包行为不因 macOS 签名改动而发生非预期变化。

- [x] P004: 规范 macOS 产物命名与上传范围 <!-- scope: package.json,.github/workflows/release.yml -->
  - 验收要点：x64 与 arm64 的 `dmg`、`zip` 产物命名唯一且可读，不会在 `release/` 或 GitHub Release 中互相覆盖。
  - 验收要点：artifact 上传路径只包含预期发布产物、blockmap 和更新元数据，不上传临时目录、未完成文件或错误架构文件。
  - 验收要点：release 上传脚本能保留 macOS 产物目录结构或通过文件名准确表达平台与架构，用户可明确选择正确下载项。
  - 验收要点：`latest-mac.yml` 与实际发布产物名称保持一致，避免自动更新或下载链接指向不存在/错误文件。

- [x] P005: 增加发布产物完整性校验 <!-- scope: scripts/verify-release-artifacts.js,package.json,.github/workflows/release.yml -->
  - 验收要点：新增校验脚本能检查 macOS 产物是否存在、文件大小非零、命名包含平台和架构、数量符合预期。
  - 验收要点：在 macOS runner 上校验 `.app`、`.dmg` 或 `.zip` 的签名状态、公证票据/staple 状态和基本可挂载/可解压能力。
  - 验收要点：校验失败会让 GitHub Action 失败，避免损坏或未完成的 macOS 包进入 Release。
  - 验收要点：校验脚本只负责产物结构、签名和完整性检查，不承担最终完整验收命令的职责。

- [x] P006: 更新 macOS 发布与安装说明 <!-- scope: docs/release/macos.md,README.md -->
  - 验收要点：文档说明正式 macOS 包需要签名和公证，以及 CI 所需 GitHub Secrets 名称、用途和配置入口。
  - 验收要点：文档说明用户应根据芯片选择 arm64 或 x64 产物，并解释 `dmg` 与 `zip` 的推荐使用方式。
  - 验收要点：若提供未签名开发包，文档必须明确标记为开发/测试用途，并说明它可能触发 Gatekeeper 提示。
  - 验收要点：安装故障排查包含“损坏提示”的判断路径，但不把绕过 Gatekeeper 作为正式发布包的主要安装方式。

- [x] P007: 完整验收 <!-- scope: validation -->
  - 验收要点：执行最终验收命令 `npm run check`，并在 GitHub Actions 中通过 `workflow_dispatch` 触发一次 macOS 发布验证；范围覆盖静态检查、macOS 打包、签名、公证、产物上传和 release 附件生成。
  - 验收要点：在 macOS runner 上对生成产物执行 `codesign --verify --deep --strict --verbose=2`、`spctl --assess --type execute --verbose`、DMG 挂载检查和 zip 解压检查。
  - 验收要点：通过标准为本地 `npm run check` 退出码为 0，macOS GitHub Action job 退出码为 0，release 中 x64/arm64 的 dmg/zip 文件名唯一且大小非零。
  - 验收要点：最终手动验收需在一台未设置开发者绕过策略的 macOS 设备上下载正式 dmg，打开、拖拽安装并首次启动 AutoPlan，不再出现“已损坏/无法打开”提示。

## 总体验收标准

- 根因明确：能够说明 macOS 损坏提示来自签名/公证、产物完整性、路径上传或架构命名中的哪一类问题，并在文档中留存结论。
- 正式包可信：配置 Apple Developer 凭据后，GitHub Actions 生成的 macOS 正式包完成签名、公证和 staple，Gatekeeper 可正常放行。
- 产物不混淆：GitHub Release 中 macOS x64 与 arm64、dmg 与 zip 文件名唯一、格式正确、大小非零，用户不会因命名或路径问题下载错包。
- CI 可阻断坏包：macOS 产物缺失、未签名、公证失败、DMG 不可挂载或 ZIP 不可解压时，发布流程失败并阻止上传正式 Release。
- 文档可操作：维护者能按文档配置发布 secrets，用户能按文档选择正确架构并安装正式包。
- 最终验收命令：执行 `npm run check`，并通过 GitHub Actions `workflow_dispatch` 运行 Release workflow 的 macOS job；验收范围为 macOS 打包配置、签名公证、产物完整性、上传路径和安装启动；通过标准为命令/job 退出码均为 0，且 macOS 正式 dmg 首次打开不再提示“已损坏/无法打开”。
