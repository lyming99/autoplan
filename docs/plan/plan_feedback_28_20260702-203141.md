# 反馈 #28 开发计划：对话项边缘无法点击打开对话窗口

## 背景

反馈内容：「对话项，hover 边缘无法点击打开对话窗口」。

### 根因分析（基于实际代码）

对话列表行位于 `src/renderer/components/workspace/WorkspaceSidebar.tsx`，结构为：

```
<div className="nav-sub-item ...">          ← 整行容器，仅样式无 onClick
  <button className="nav-sub-item__pin" />  ← 置顶按钮（stopPropagation）
  <button className="nav-sub-item__main"
          onClick={() => selectConversation(conversation.id)} />  ← 唯一打开入口
  <div className="nav-sub-item__actions">   ← 更多操作菜单（stopPropagation）
</div>
```

在 `src/renderer/styles/workspace.css` 中，`.nav-sub-item` 设置了：

- `padding: 6px 8px`（上下 6px、左右 8px 内边距）
- `gap: 6px`（pin / main / actions 之间的间隙）
- `cursor: pointer` 且 `:hover` / `:focus-within` 时有 `background: var(--bg-hover)`

问题：`.nav-sub-item` 容器本身**没有绑定 onClick**，只有内部的 `.nav-sub-item__main` 按钮调用 `selectConversation` 打开对话。因此：

- 鼠标移到行的内边距边缘、按钮间隙处时，整行显示 hover 高亮且光标为 pointer（视觉与光标都暗示「可点击」）；
- 但点击这些边缘/间隙区域命中的是容器 `<div>` 本身，没有任何处理 → **无法打开对话窗口**。

### 关键约束（必须遵守）

`src/renderer/pages/WorkspacePage.test.tsx` 第 374–376 行以源码字符串断言强制约束了调用结构：

- `onTab('chat')` 在侧栏中**恰好出现 1 次**；
- `selectConversation(conversation.id)` 在侧栏中**恰好出现 1 次**，且必须挂在 `.nav-sub-item__main` 按钮上。

因此**不能**在 `.nav-sub-item` 容器或别处再新增一个 `selectConversation(conversation.id)` 调用，否则这些断言会失败。

### 修复方案（最优工程方案）

采用纯 CSS 的 **stretched-link（拉伸链接）覆盖层** 方案：

1. 为 `.nav-sub-item__main` 增加一个绝对定位、`inset: 0` 的伪元素（`::after`），其包含块为已 `position: relative` 的 `.nav-sub-item`，从而把 `__main` 按钮的命中区域**拉伸覆盖整行**（含内边距与间隙）。点击覆盖层即等同点击 `__main` 按钮 → 触发那唯一的 `selectConversation(conversation.id)`。
2. 将 `.nav-sub-item__pin` 与 `.nav-sub-item__actions` / `.nav-sub-item__menu-wrap` 设为 `position: relative` 并提升 `z-index`，使其位于覆盖层之上，保持 pin、更多操作菜单独立可点击，不被覆盖层拦截。
3. 对 `.nav-sub-item__pin` 的 `pointer-events` 在「非悬停且未置顶」时设为 `none`，避免不可见的 pin 偷走左侧行边缘的点击；悬停或已置顶时恢复 `auto`。

该方案**不改动任何 JSX/调用结构**，因此 `WorkspacePage.test.tsx` 的字符串断言无需改动即可继续通过，同时彻底解决边缘不可点击的问题。
