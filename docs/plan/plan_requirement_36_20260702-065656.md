# 需求 #36 开发计划

## 需求理解

1. **聊天窗口未打开时，左侧对话项标题高亮要淡一些**
   - 当用户切到其它 Tab（需求/反馈/验收/概览等，即 `activeTab !== 'chat'`）时，左侧侧栏里"当前选中"的对话项仍需标记为选中态，但高亮要比聊天窗口打开时更淡（一种"选中但非当前视图"的弱化高亮）。
   - 当 `activeTab === 'chat'` 时，恢复原有的完整高亮（`--brand-50` 背景 + `--brand-700` 文字）。

2. **对话首次发生后，根据内容通过 API 生成标题**
   - 新对话创建时 `title` 为空（`createConversation` 默认空串，UI 兜底显示"新对话"）。
   - 首条消息往返完成（助手回复结束）后，调用 LLM API 根据对话内容（首条用户消息为主）生成一个简短标题，写回 `conversations.title`，并实时更新侧栏显示。
   - 已有标题的对话不重复生成；生成失败不影响主对话流程。

## 方案概述

- **标题生成放在 `chatController.js`**：它已具备 `db` / `llmClient`(=`createLlmClient`) / `getConfig()`(解析后的 AI 配置) / `conversationId`，且 `chatController.test.js` 已有完整的 `llmClient` stub，最易测试。在 agent loop 的成功终止点（纯文本回复完成）触发：若标题为空/占位 → 调 LLM 生成简短标题 → 写库 → 通过 `onDone` 携带 `conversationId` 与 `title` 返回。
- **复用 `chat:done` IPC 投递**（`main.js` 转发、`useChat` 消费），不新增 IPC 通道，规避 `useChat` 流式 chunk 的早返回守卫（标题事件在流结束后到达，会被守卫拦截）。
- **高亮淡化纯前端**：`WorkspaceSidebar` 已有 `activeTab` 与 `isActive`，新增"对话视图未激活"的弱化修饰类 + CSS（浅色/深色双主题）。

- [x] P001: chatController 增加基于内容的标题生成并在成功回复后写库与回传 <!-- scope: src/chat/chatController.js -->
  - 在 `src/chat/chatController.js` 新增内部函数 `generateConversationTitle(conversationId)`：读取该对话首条 `role='user'` 消息内容（截断约 500 字），用 `getConfig()` 解析出的 AI 配置（provider/baseUrl/apiKey/model，低 temperature、不带 tools）调用注入的 `llmClient`，消费流式 `text_delta` 累加为标题文本，规范化（去除引号/换行/首尾空白、去掉句末标点、截断 ≤ 30 字）；非空则 `UPDATE conversations SET title=?, updated_at=? WHERE id=?`，并返回标题；任意异常或解析失败时静默返回 `null`，绝不抛出。
  - 新增"是否需要生成标题"的判定：当 `conversations.title` 为空或属于占位词（`新对话`/`默认对话`）时才生成，已有真实标题则直接返回 `null`（避免重复生成与覆盖用户重命名）。
  - 在 `runAgentLoop` 的纯文本回复终止分支（`toolCalls.length === 0` → `emitDone({ status: 'done' })` 处）于 `emitDone` 之前调用：`const title = await generateConversationTitle(conversationId).catch(() => null);`，并将标题连同 `conversationId` 一并传入 `emitDone`，即 `emitDone({ status: 'done', conversationId, title: title ?? undefined })`。
  - 验收要点：
    - 空标题对话在首条消息成功回复后，`conversations.title` 被更新为规范化的短标题（无引号、无换行、≤30 字），`onDone` 回调收到非空 `title` 与正确 `conversationId`。
    - 已有真实标题的对话再次发送消息时不会重新生成、不覆盖原标题。
    - 标题生成的 LLM 调用抛错或返回空内容时，主流程不受影响：`onDone` 仍以 `status:'done'` 触发，`title` 为 `undefined`，标题保持原样。
    - 中止(`aborted`)/错误(`error`)分支不触发标题生成。
    - 向后兼容：现有只断言 `done.status` / `done.error` 的调用方不受新增字段影响。
