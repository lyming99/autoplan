# Plan: 对话模块多会话、多 AI 配置与思考深度支持（需求 #28）

## 需求概述

当前对话模块仅支持每个项目一个全局配置和一个对话。需求要求：
1. **多对话**：每个项目可创建多个独立对话，各自拥有独立消息历史
2. **多 AI 配置**：可创建多个命名 AI 配置（如 "GPT-4o 正式", "DeepSeek 测试"），每个对话可选择绑定的配置
3. **OpenAI 思考深度**：支持配置 `reasoning_effort`（low/medium/high），仅对 o 系列推理模型生效
4. **DeepSeek 思考深度**：支持 DeepSeek reasoning 模型的专属思考深度配置
5. **成熟 SDK**：引入 `openai` npm 包（v4+）替代手写 fetch SSE 解析，获得更好的类型安全与推理内容流式支持
6. **思考中/回复中状态**：UI 实时区分模型正在「思考（推理）」还是「生成回复」

## 现状分析

| 维度 | 现状（需求 #26 产物） | 目标 |
|------|----------------------|------|
| 对话数量 | 每项目 1 个，chat_messages 通过 project_id 关联 | 每项目 N 个，新增 conversations 表 |
| AI 配置 | 全局唯一，扁平存储在 settings 表 `chat.*` 键 | 每项目 N 个命名配置，新建 ai_configs 表 |
| LLM 客户端 | 零依赖手写 fetch + SSE 解析（llmClient.js） | 引入 openai npm 包处理 OpenAI 兼容协议；Anthropic 保持 fetch 增强 |
| 思考深度 | 不支持 | OpenAI: reasoning_effort；DeepSeek: 对应推理参数；Anthropic: thinking budget_tokens |
| 状态展示 | 仅 "思考中…" 占位文字 | 区分推理阶段（思考中…）与生成阶段（回复中…），推理内容可折叠查看 |

## 架构设计

### 数据模型

```
ai_configs
├── id (PK)
├── project_id (FK → projects.id)
├── name (配置名称，如 "GPT-4o")
├── provider ('openai' | 'deepseek' | 'anthropic')
├── base_url
├── api_key
├── model
├── temperature
├── thinking_depth ('low' | 'medium' | 'high' | null)    ← 通用思考深度
├── thinking_budget_tokens (INTEGER, nullable)              ← Anthropic 扩展思考 token 预算
├── created_at
└── updated_at

conversations
├── id (PK)
├── project_id (FK → projects.id)
├── title (对话标题，默认取自首条用户消息)
├── ai_config_id (FK → ai_configs.id, nullable, null=使用默认配置)
├── created_at
└── updated_at

chat_messages (扩展现有表)
├── ... 现有字段 ...
└── conversation_id (FK → conversations.id)  ← 新增
```

### 向后兼容迁移

1. 现有 settings 表 `chat.*` 键 → 自动迁移为首条 `ai_configs` 记录（name="默认配置"）
2. 现有 `chat_messages` 行（conversation_id=NULL）→ 自动创建首条 `conversations` 并回填 `conversation_id`
3. 对话历史加载逻辑兼容旧数据：conversation_id 为 NULL 时回退到 project_id 全量加载

### LLM 客户端重构

- 引入 `openai` npm 包（v4+），重构 OpenAI 兼容协议流（覆盖 OpenAI / DeepSeek 及所有兼容端点）
- OpenAI 请求体增加 `reasoning_effort` 字段（仅当 thinking_depth 非空时发送）
- DeepSeek 同样使用 OpenAI 兼容路径，DeepSeek 推理模型专属参数由 thinking_depth 映射
- Anthropic 路径保持 fetch + SSE，但增加 `thinking` 扩展参数（type: "enabled", budget_tokens）
- 流式事件新增 `thinking_start` / `thinking_delta` / `thinking_end` 类型，前端据此展示思考状态
