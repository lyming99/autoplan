# 需求 #35：文件操作跨文件夹/跨项目访问开关

> 生成时间：2026-07-03
> 需求：为 `read_file` 等文件操作增加可配置的访问范围开关，默认安全（仅当前项目工作区），开启后可按白名单访问其他文件夹/项目。

---

## 需求分析与方案

### 当前现状（基于代码分析）

AutoPlan 中所有"读取/访问文件"的入口当前都硬编码为"仅允许当前项目工作区"，通过 `isInsidePath` 实现。该函数被重复实现了 **3 份**，散落在不同模块：

| 入口 | 文件 | 行为 |
| --- | --- | --- |
| Chat 工具 `read_file` | `src/chat/chatTools.js` (`handleReadFile`) | 读取工作区内文件，256KB 截断；越界返回 `FILE_PATH_OUTSIDE_WORKSPACE` |
| Chat 工具 `search_files` | `src/chat/chatTools.js` (`handleSearchFiles`) | 路径 LIKE + 内容扫描，仅扫描项目 `scan_files` |
| 打开文件（编辑器） | `src/main.js` `resolveWorkspaceFilePath`（`workspace:openFile` IPC） | 越界抛 `FILE_PATH_OUTSIDE_WORKSPACE` |
| 读取计划文件 | `src/main.js` `resolvePlanPath`（`plans:read`） | 越界抛 `PLAN_PATH_OUTSIDE_WORKSPACE` |
| 计划/需求文件写入路径 | `src/loop/workspaceFiles.js` `resolveWorkspaceChildPath` / `resolveSafeAutoPlanIntakePlanPath` | 强制锁定在 `docs/plan`，**写入安全**，**不应放宽** |

`isInsidePath` 重复存在于 `chatTools.js:359`、`main.js:1142`、`workspaceFiles.js`（逻辑等价的 `resolveWorkspaceChildPath`）。MCP 工具层（`src/mcpTools.js`）**不**提供 `read_file` 等裸文件读，仅做 DB 级的项目/计划/需求/任务管理，故不属于本次"文件读访问"范围（仅需在验收时确认未新增裸文件读）。

### 配置存储

- 全局键值 `settings` 表（SQLite），经 `AppDatabase.getSetting/setSetting/getSettings`（`src/database.js:552-566`）读写，所有值以字符串存储。
- `main.js` 另有 `readSetting(key)` 帮助函数（`main.js:1113`）读同一张表。
- `chatTools.js` 的 `getChatToolDefinitions({ db, projectId, workspacePath, intakeService })` 已注入 `db`，可直接读配置。

### 设计方案

**1. 新增共享文件访问策略模块** `src/fileAccess/policy.js`，消除 3 处重复的 `isInsidePath`，统一收敛访问判定：
- 读取配置键 `fileAccess.scope` / `fileAccess.allowCrossProject` / `fileAccess.allowedRoots`，解析为 `{ scope, allowedRoots[], effectiveRoots[], unrestricted }`。
- `isPathAllowed(targetPath, policy)` / `assertPathAllowed(targetPath, policy)`：基于 **realpath**（符号链接穿透后）做边界判定，避免软链逃逸。
- 配置默认值与键名常量、安全的 `parseAllowedRoots(json)` 解析（去重、过滤空、规范化、realpath 容错）。

**2. 配置模型**（写入全局 `settings` 表，键名带 `fileAccess.` 前缀）：
- `fileAccess.scope` ∈ `project | workspace | custom | all`，默认 `project`。
  - `project`：仅当前项目工作区（= 默认安全行为，与现状一致）。
  - `workspace`：AutoPlan 项目为单根，等同 `project`（保留枚举值以对齐需求建议 API，并兼容未来多根）。
  - `custom`：项目工作区 + `allowedRoots` 白名单。
  - `all`：不限文件系统（仍受 OS 权限约束，UI 给出显著风险警告）。
- `fileAccess.allowCrossProject` ∈ `true|false`，默认 `false`（需求明确提到的布尔开关；为 `true` 时等效于 `custom`，需配合 `allowedRoots`）。
- `fileAccess.allowedRoots`：JSON 数组字符串，绝对路径白名单，默认 `[]`。
- 有效范围解析：`all` → 不受限；否则若 `scope==custom` 或 `allowCrossProject==true` → 工作区 ∪ allowedRoots；否则 → 仅工作区。

**3. 接入所有读取类文件入口**（写入类 `docs/plan` 路径保持锁定，不放宽）：
- `chatTools.js` `read_file` / `search_files` 接入 `assertPathAllowed`。
- `main.js` `resolveWorkspaceFilePath` / `resolvePlanPath` 接入 `assertPathAllowed`。

**4. 配置持久化与 IPC**：新增 `file-access:get` / `file-access:save` IPC（`main.js`），经 `preload.js` 暴露给渲染层；底层用 `db.setSetting`。

**5. 设置 UI**：`WorkspaceSettingsView.tsx` 新增"文件访问范围"区块（范围选择 + 跨项目开关 + 白名单目录列表，复用 `projects:pickDirectory` 选目录），类型与选项数据落在 `workspaceForms.ts` / `types.ts`。

**6. 文档**：README 或 `docs/` 增加访问范围模型、默认值、安全边界与配置键说明。

**7. 测试**：新增 `policy.test.js` 并扩展 `chatTools.test.js`，覆盖默认安全、custom 放行、越界报错、realpath/软链、配置解析健壮性等场景。

## 总体验收标准

**最终验收命令**
- 单测：`npm test`（全量 `node --test` 测试套件，新增 `policy.test.js` 与扩展的 `chatTools.test.js` 必须通过）。
- 构建：`npm run build`（TypeScript + Vite + electron 构建无错误）。

**验收范围**
- 受统一策略约束的读取类入口：Chat `read_file`、`search_files`、`workspace:openFile`、`plans:read`。
- 配置与 UI：`file-access:get/save` IPC、`WorkspaceSettingsView` 访问范围区块、`workspaceForms`/`types` 类型。
- 共享策略模块 `src/fileAccess/policy.js`、文档。

**通过标准**
1. 默认（未配置 / `scope=project`）：`read_file` 等所有读取入口仅可访问当前项目工作区内部；`../` 越界、绝对路径越界、临时目录均被拒绝并返回明确错误提示（保持现有 `errorCode` 兼容或统一为 `FILE_PATH_OUTSIDE_SCOPE`）。
2. 开启跨项目访问（`scope=custom` 或 `allowCrossProject=true`）且配置 `allowedRoots` 后：可访问白名单根内文件；白名单之外的路径仍被拒绝。
3. `scope=all`：访问不受应用层限制（仍受 OS 权限约束），UI 有显著风险提示。
4. 超出允许范围的路径统一返回明确错误信息，不静默成功、不抛未处理异常。
5. 各文件操作入口（read_file、search_files、打开文件、读取计划）使用同一套访问策略与边界判定，`isInsidePath` 仅存于共享模块一处。
6. 计划/需求写入路径（`docs/plan`）保持工作区锁定，未被本次改动放宽。
7. 配置可通过设置页正确读取/保存并回填，键值以字符串存入 `settings` 表（与既有 `mcp.`/`chat.` 口径一致）。
8. `npm test` 全绿，`npm run build` 成功。
