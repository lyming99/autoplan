# 反馈 #10 开发计划：计划完成后关闭关联需求与反馈

## 背景与目标

反馈要求：计划完成后，要将反馈、需求标记为完成状态，不能继续停留在 `open`。

基于当前代码梳理，需求与反馈分别存储在 `requirements`、`feedback` 表，默认 `status='open'`；生成单条需求/反馈计划时，`src/loop/planGeneration.js` 会把对应记录的 `linked_plan_id` 回写为新 plan id。扫描待处理项时，`src/loopService.js` 会排除 `completed`、`closed`，因此如果计划完成后关联记录仍为 `open`，它们会继续表现为未完成项。计划完成目前主要发生在 `src/loop/validation.js`：空验收命令直接完成、验收命令成功完成、最后一个完整验收任务完成，但这些路径只更新 `plans.status`，没有同步更新关联需求/反馈。

目标是在所有计划成功完成路径中，按 `linked_plan_id` 将同项目关联的需求和反馈标记为 `completed`，保持幂等，不覆盖已经 `closed` 的记录，并让快照、MCP 列表和前端状态展示自然反映完成态。

## 总体验收标准

最终验收命令：

```powershell
npm run check
npm run smoke
npm run build
```

验收范围：
- 数据库层：`requirements.status`、`feedback.status`、`linked_plan_id`、`updated_at` 的完成态同步。
- 计划完成路径：空验收命令跳过校验、外部验收命令成功、最终完整验收任务完成。
- 非完成路径：验收失败、任务失败、计划中断、普通任务完成但计划未完成。
- 对外可见层：`loop.snapshot()`、需求/反馈列表、搜索状态字段、MCP `list_requirements` / `list_feedback` 的状态过滤。

通过标准：
- 绑定到已完成计划的需求和反馈不再保持 `open`，应变为 `completed`；已经 `closed` 的记录不被改写。
- `completed` 需求/反馈不会被后续扫描当作待处理 open 项重复生成计划。
- MCP 和前端快照均能看到完成态，按 `open` 查询不会混入已完成项。
- 最终验收命令全部通过，且没有新增 TypeScript、Node 语法、构建或文件长度检查错误。
