# 需求 #40 开发计划：对话项整行可点击打开对话

## 需求概述

工作区侧栏「对话」分组下展开的「对话项（会话列表项）」目前只有点击中间标题区（`.nav-sub-item__main` 按钮）才能打开对话；点击行的其它位置（左侧置顶按钮区、各项之间的间隙、左右内边距空白、右侧操作/更多菜单区）均无反应。

期望：**整行可点击**——点击对话项行的任意位置都能打开（切换到）该对话；同时置顶按钮、更多操作菜单仍按各自语义独立工作，不被整行点击逻辑误触发。

## 现状分析（基于源码）

对话项渲染于 `src/renderer/components/workspace/WorkspaceSidebar.tsx:291-390`，每个对话项结构为：

```
<div className={`nav-sub-item ${isActive ? 'active' : ''}`}>     // 行容器（无 onClick）
  <button className="nav-sub-item__pin" onClick={togglePinned}>…</button>     // 置顶按钮
  <button className="nav-sub-item__main" onClick={selectConversation}>…</button>  // 仅此处打开对话
  <div className="nav-sub-item__actions">…更多菜单…</div>                     // 操作区（无打开逻辑）
</div>
```

- 打开对话的唯一入口是 `.nav-sub-item__main` 按钮的 `onClick={() => selectConversation(conversation.id)}`（`selectConversation` 定义于 `:146-153`，内部 `void chatState?.switchConversation(conversationId)`）。
- `.nav-sub-item` 行容器（`:292-295`）本身**没有 onClick**，因此容器自身 padding（`6px 8px`）、flex `gap: 6px` 形成的间隙、置顶按钮所占区域、右侧 `.nav-sub-item__actions` 所占区域，点击都不会打开对话——这就是「点击部分位置无效」的根因。
- CSS（`src/renderer/styles/workspace.css:70-166`）：`.nav-sub-item` 已是 `cursor: pointer`、`display:flex`；`.nav-sub-item__actions` 默认 `pointer-events: none`，仅在 `:hover`/`:focus-within`/`.active` 时变为 `pointer-events:auto`。即非悬停时该区域不拦截事件，点击会自然落到行容器上——意味着只要给行容器挂上 `onClick`，这些「失效位置」即可全部生效。

## 设计方案（最优工程方案）

采用**事件委托 + 冒泡**的最小改动方案，不改变现有 DOM 层级与样式：

1. 将「打开对话」逻辑从 `.nav-sub-item__main` 按钮**上移到外层 `.nav-sub-item` 容器 div** 的 `onClick`：点击行内任意空白（左右内边距、置顶按钮与主区之间、主区与操作区之间的间隙、非悬停态下的操作区）都会冒泡/直接落到容器，从而调用 `selectConversation` 打开对话。
2. 给 `.nav-sub-item__pin` 置顶按钮、`.nav-sub-item__actions` 操作区（含「更多」触发按钮与「置顶/重命名/删除」菜单项）加 `event.stopPropagation()`，避免它们冒泡触发整行打开——点置顶只置顶、点菜单只开菜单。
3. `.nav-sub-item__main` 保留为按钮（维持 focus、`aria-current`、`title` 与键盘可达性：按钮默认 Enter/Space 激活会产生 click 并冒泡到容器打开对话），移除其冗余 `onClick`，避免与容器重复触发 `selectConversation`。
4. **重命名态**（`renamingId === conversation.id`）下行内渲染的是 `<input>`，此时容器点击不得切换对话（否则输入时误打开）——通过条件挂载 `onClick`（重命名态不挂）或在 handler 内判断 `renamingId` 提前 return 实现。

无需新增/改动 CSS（行容器已具备 `cursor:pointer`，操作区 `pointer-events` 行为恰好兼容整行点击）。
