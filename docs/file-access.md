# 文件访问范围与安全边界

> 适用版本：需求 #35（`read_file` 等文件操作跨文件夹/跨项目访问开关）

AutoPlan 中所有"读取/访问文件"的能力默认是**安全**的：仅允许访问当前项目的**工作区目录**内部。
通过设置可以按白名单放宽到其他文件夹，或完全放开（高风险）。
本文说明访问范围模型、配置项、各语义、安全边界与最小安全实践。

---

## 一、默认即安全

未做任何配置时（`scope=project`，出厂默认），以下读取入口都**只能**访问当前项目工作区内部的文件：

- 路径中的 `../` 越界、指向工作区外的绝对路径、系统临时目录等都会被**拒绝**；
- 判定基于 **realpath**（符号链接穿透后的真实路径），可防止软链逃逸——即工作区内若存在指向外部的软链，读取其目标时仍会被拦截；
- 越界访问不会静默成功，会返回明确的错误信息（统一错误码 `FILE_PATH_OUTSIDE_SCOPE`）。

---

## 二、配置项

配置存储在全局 `settings` 表（SQLite），键名统一带 `fileAccess.` 前缀，值均以字符串存储（与既有 `mcp.` / `chat.` 键口径一致）。
可在「工作区 → 设置 → 文件访问」分页可视化配置，或直接读写以下键：

| 键 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `fileAccess.scope` | 枚举 `project` \| `workspace` \| `custom` \| `all` | `project` | 访问范围模式 |
| `fileAccess.allowCrossProject` | 布尔（字符串 `true`/`false`） | `false` | 快捷开启跨项目访问；为 `true` 时等效于 `custom`，需配合白名单 |
| `fileAccess.allowedRoots` | JSON 数组字符串（绝对路径） | `[]` | 工作区之外的额外白名单根目录 |

IPC：`file-access:get` 读取（应用默认值）、`file-access:save` 保存（逐键落库；数组以 JSON 字符串存储）。

---

## 三、各 scope 语义

| scope | 允许访问范围 | 风险 | 适用场景 |
| --- | --- | --- | --- |
| `project` | 仅当前项目工作区（**默认安全**） | 低 | 绝大多数场景 |
| `workspace` | AutoPlan 项目为单根，**等同于 `project`**（保留枚举以对齐 API 并兼容未来多根） | 低 | 同上 |
| `custom` | 当前项目工作区 **∪** `allowedRoots` 白名单 | 中 | 需要引用工作区外的共享代码库/配置目录 |
| `all` | 不受应用层限制（**仍受操作系统权限约束**） | **高** | 极少数完全可信的受控环境 |

有效范围解析逻辑：

- `scope=all` → 不受限；
- 否则若 `scope=custom` **或** `allowCrossProject=true` → 工作区 ∪ 白名单；
- 否则（`project` / `workspace`）→ 仅工作区。

---

## 四、白名单用法（`custom`）

选择 `custom`（或开启 `allowCrossProject`）后，在「文件访问 → 白名单目录」中添加要放行的**绝对路径根目录**：

- 每个根通过系统目录选择器（`projects:pickDirectory`）指定；
- 白名单解析时会做 `path.resolve` 规范化、realpath 容错（不存在的根保留规范化路径）、去重、过滤空串，非法 JSON / 非数组输入降级为 `[]`；
- 白名单内的路径及其子树允许访问；**白名单之外的路径仍被拒绝**；
- 计划/需求等**写入**路径（`docs/plan`）**不**因白名单而放宽（见下文）。

---

## 五、受统一策略约束的入口

以下读取类入口共用同一套访问策略与边界判定（`src/fileAccess/policy.js`，`isInsidePath` 仅存于此一处）：

| 入口 | 位置 |
| --- | --- |
| Chat 工具 `read_file` | `src/chat/chatTools.js` |
| Chat 工具 `search_files` | `src/chat/chatTools.js` |
| 打开文件（`workspace:openFile`） | `src/main.js` |
| 读取计划（`plans:read`） | `src/main.js` |

> MCP 工具层（`src/mcpTools.js`）仅做 DB 级的项目/计划/需求/任务管理，**不**提供裸文件读取，故不属于本策略范围。

---

## 六、刻意保持锁定的写入路径

计划与需求文件的**写入**路径（`docs/plan` 下的计划/需求 Markdown）由 `src/loop/workspaceFiles.js` 的 `resolveSafeAutoPlanIntakePlanPath` / `resolveWorkspaceChildPath` 强制锁定在**工作区内**。

**本次改动刻意不放宽写入路径**：即使开启了跨项目访问或 `scope=all`，计划/需求文件仍只能写入当前项目工作区的 `docs/plan`，避免生成内容被写到工作区之外。

---

## 七、`all` 的风险与操作系统权限边界

选择 `scope=all` 时：

- 文件读取入口可访问**本机任意路径**，应用层不再做边界限制；
- 但仍受**操作系统文件权限**约束（无读取权限的文件仍会失败）；
- 设置页保存 `all` 时会回传 `warned: true` 并在界面给出显著风险提示。

`all` 仅建议在你**完全信任执行环境与执行器行为**时使用。多数情况下，用 `custom` + 最小白名单即可满足跨目录引用需求。

---

## 八、跨项目访问的最小安全实践

1. **优先保持 `project`**：默认范围已能覆盖绝大多数场景；
2. **用 `custom` 而非 `all`**：按需放行，白名单之外一律拒绝；
3. **仅配置必要的白名单根**：只添加确实需要被读取的目录，尽量收窄到具体子目录而非整个盘符/家目录；
4. **谨慎使用 `all`**：仅在受控、可信环境中临时使用，使用后及时改回；
5. **注意软链**：策略基于 realpath 判定，工作区内指向外部的符号链接会被识别并拦截，但 `all` 下软链也会被放行——保持最小范围可降低此类暴露面。
